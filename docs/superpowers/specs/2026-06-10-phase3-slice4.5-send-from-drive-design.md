# Phase 3 Slice 4.5 — Send-from-Drive (design)

**Date:** 2026-06-10
**Branch:** `phase3-drives`
**Status:** Approved
**Builds on:** Slice "Library Selection / Send / Locations" (`SendEngine`, `SendDestination`, `VolumeCopyDestination`, `AirDropDestination`, `sends.jsonl`, AirDrop verification); Slice 2.5 (`vault_presence` with `driveRelPath`, cached drive-only thumbnails); Slice 4 (drive resolution: `canonicalVaults` / `driveIsPresent` / `openVault`, `VerifiedCopy`).

---

## 1. Goal

Send (AirDrop to a phone, or copy to a volume/SD card) photos that are **drive-only** — evicted or never-local — **directly from a plugged-in canonical drive**, without rehydrating them to the Mac first.

Today the Send path resolves each selected photo to its **local** file (`LibraryService.absoluteURL(for:)`). A drive-only photo has no local file, so the send would fail. Slice 4.5 makes Send source those bytes from the connected drive instead, and gracefully handles the case where the drive isn't connected.

This is a **small slice**: the entire send/verify/journal engine is reused unchanged. The only genuinely new logic is **source resolution** (where to read each photo's bytes) plus a **pre-flight warning** in the send sheet.

### Non-goals

- No change to `SendEngine`, `SendDestination`, `VolumeCopyDestination`, `AirDropDestination`, dedup, AirDrop verification, or `sends.jsonl`.
- No temp-staging / no copying drive files to the Mac before sending (the drive file URL is handed straight to the destination — see §3).
- No Live-Photo-motion restoration (unchanged: Live Photos send as the still only).
- No new on-disk format artifact (see §6).

---

## 2. Hard invariants honored

| Invariant | How |
|---|---|
| Originals never modified/moved without explicit action | Send only **reads** the drive file; it is never moved or written. |
| Drives passive, one-way | The drive is a pure **read source** for the send. Slice 4.5 writes **nothing** to the drive (see §6). |
| Atomic, hash-verified copies | Reused verbatim: `VolumeCopyDestination` copy+verify and AirDrop fingerprint verification are unchanged. |
| Nothing hard-deletes | Send copies; it never deletes from the drive or anywhere. |

---

## 3. Source resolution (Core — the heart of the slice)

A new file `Sources/OpenPhotoCore/Send/LibraryService+SendSource.swift` adds two value types and one resolver method. The resolver is filesystem-independent (it only builds URLs and value structs — it does **not** read files), so it is fully unit-testable with mock vaults.

```swift
/// A drive-only item whose drive is not connected, so its bytes can't be read to send right now.
public struct UnreachableSendItem: Sendable, Equatable {
    public let hash: String
    public let displayName: String   // filename, for the warning list
    public let driveName: String     // which drive to connect (drive root basename)
    public init(hash: String, displayName: String, driveName: String) { … }
}

/// A selection split into what can be sent now and what can't.
public struct SendSourcePlan: Sendable {
    public var sendable: [SendItem]                 // ready: local file, or connected-drive file
    public var unreachable: [UnreachableSendItem]   // drive-only, drive not connected
    public init(sendable: [SendItem], unreachable: [UnreachableSendItem]) { … }
}

extension LibraryService {
    /// Resolve a selection into sendable `SendItem`s and the unreachable remainder.
    /// - Local item (`driveRelPath == nil`): sourced from its local vault file.
    /// - Drive-only item whose drive is in `connectedDrives`: sourced **directly** from the
    ///   drive file (`drive.absoluteURL(forRelativePath: item.driveRelPath!)`).
    /// - Drive-only item whose drive is absent: `unreachable`, tagged with `driveNames[vaultID]`.
    /// Does not touch the filesystem; the destination later reads/verifies the bytes.
    public func resolveSendSources(_ items: [TimelineItem],
                                   connectedDrives: [Vault],
                                   driveNames: [String: String]) -> SendSourcePlan
}
```

Resolution rules, per item:

1. **Local** (`item.driveRelPath == nil`): `originalURL = absoluteURL(for: item)`. If that is nil (unknown vault — shouldn't happen for a local row), drop the item (matches today's `compactMap` behavior; a present-but-vanished file is **not** dropped here — the destination reports it as failed, exactly as today).
2. **Drive-only, drive connected** (`drivesByID[item.vaultID] != nil`): `originalURL = drive.absoluteURL(forRelativePath: item.driveRelPath!)`. Direct read from the drive; no staging.
3. **Drive-only, drive absent**: append to `unreachable` with `driveName = driveNames[item.vaultID] ?? "a drive"`.

`SendItem` fields for both sendable cases come from the `TimelineItem`: `hash = item.hash`, `fingerprint = PresenceFingerprint(size: item.size, captureDateMs: item.takenAtMs, hash: item.hash)`, `displayName = (item.relPath as NSString).lastPathComponent`.

**Why direct-from-drive (no temp staging):** `VolumeCopyDestination` copies from `SendItem.originalURL` to the target and re-hashes; `AirDropDestination` hands the file URL to `NSSharingService`. Both accept any readable file URL, so the drive path works directly. Staging to a Mac temp dir would duplicate bytes, add a failure surface, and contradict the slice's purpose (avoid holding a local copy). The drive file is read-only throughout.

## 4. Pre-flight warning (UI)

`Sources/OpenPhotoApp/Send/SendSheet.swift` gains a pre-flight step **only when the plan has unreachable items**. The sheet receives the same `items: [TimelineItem]` it does today and computes the plan once via `state.sendPlan(for: items)` before sending.

Three cases:

- **Everything reachable** (`unreachable.isEmpty`): no extra step — proceed straight into the existing send/progress/result flow with `plan.sendable`. Identical to today's experience.
- **Some reachable, some not**: show a warning panel (the user's explicit ask — "warn first, but tell me which won't send and which drive to plug in"):
  - A short line: *"N photos can't be sent right now — their drives aren't connected."*
  - The unreachable photos as a thumbnail list (cached thumbnails persist for drive-only items — Slice 2.5), **grouped by `driveName`** with a small header per drive: *"On **Photos-A** — connect it to include these."* Thumbnails are rendered by mapping each `UnreachableSendItem.hash` back to the original `TimelineItem` (the sheet already holds `items`).
  - Buttons: **Send N** (the reachable count) and **Cancel**. Choosing Send runs the engine with `plan.sendable` only.
- **Everything unreachable** (`sendable.isEmpty`): the same warning panel, but **Send is disabled** (nothing to send) — the panel is purely informative, telling the user which drives to connect.

No change to where Send is offered or its enable/disable gating in the selection bar: Send remains available whenever a target device is connected and there's a selection. All per-photo reachability is handled inside the sheet, so mixed local/drive-only selections "just work."

---

## 5. Live Photos & mixed selections

- **Live Photos**: unchanged. The paired video is hidden from browse (`isLivePairedVideo = 1`), so a selection only ever contains stills; the still is sourced from the drive when drive-only. No pair lookup or pair-send is introduced (consistent with the existing "Live Photos send as the still only" rule).
- **Mixed selections** (some local, some drive-only, possibly across multiple drives): handled naturally — `resolveSendSources` classifies each item independently. Unreachable items group by their own drive, so the warning can name several drives at once.

---

## 6. Logging & format

- **`sends.jsonl`** (primary vault's `.openphoto/`): unchanged. It is hash-keyed and source-agnostic — a confirmed send is recorded identically whether the bytes came from a local file or a drive. The Locations panel keeps working with no change.
- **Primary-vault `sync-log.jsonl`**: unchanged. `SendEngine` continues to append its one `"send"` summary event to the vault passed in (`library.vaults.first`).
- **Drive `sync-log.jsonl`: deliberately NOT written.** Reading a drive's bytes to send them elsewhere is a **read-only** use of the drive; the send is already journaled durably in `sends.jsonl` (source-agnostic) and the primary vault's sync-log. Writing a "send" line to the drive would make a pure read operation mutate the drive and add a new failure surface for no information the system doesn't already have. This keeps the drive a clean read source for sends.
- **Format doc:** no change required. No new on-disk artifact is introduced, and `"send"` is already a recognized `sync-log.jsonl` event in `docs/format/vault-format-v1.md` §9. (Documentation discipline satisfied: nothing in the normative format changes.)

## 7. App glue (AppState)

- `func sendPlan(for items: [TimelineItem]) -> SendSourcePlan` — builds `connectedDrives = canonicalVaults.filter { driveIsPresent($0) }.compactMap { openVault(for: $0) }` and `driveNames` = every canonical vault's `id → (rootPath as NSString).lastPathComponent` (covers absent drives too, so unreachable items can be named), then calls `library.resolveSendSources(items, connectedDrives:driveNames:)`.
- `send(...)` is refactored to take already-resolved items: `func send(_ sendItems: [SendItem], to device: ConnectedDevice, progress:) async -> SendEngine.Result?` (engine wiring otherwise identical; still logs to / dedups against the primary vault as today). The old per-item `sendItem(for:)` helper is **removed** — `resolveSendSources` subsumes it.
- `SendSheet` is the only caller; it computes the plan, optionally shows the warning, then calls `send(plan.sendable, …)`.

## 8. Error handling / edge cases

| Case | Behavior |
|---|---|
| Drive unplugged **mid-send** | That item's copy/verify fails; it lands in the result's `failed` list (existing engine behavior). No retry, no partial-file left (atomic temp→rename). |
| Local file vanished between selection and send | Built into a `SendItem` (not existence-checked, as today); the destination reports it `failed`. |
| Selection spans several disconnected drives | All such items are `unreachable`, grouped per drive in the warning so the user knows each drive to connect. |
| Drive-only item already present on the target | Existing dedup (hash on volumes, fingerprint on phones) marks it `alreadyPresent` — unchanged. |
| No drive name resolvable | Falls back to the literal "a drive" in the warning (defensive; shouldn't occur since `driveNames` covers all canonical vaults). |

## 9. Testing

**Core — `Tests/OpenPhotoCoreTests/SendSourceResolverTests.swift`** (pure, mock vaults + temp "drive" vaults seeded via `Manifest.write`/`replaceVaultPresence`; generated mock media only — never `~/Pictures`):

1. Local-only selection → all `sendable`, each `originalURL` points at the local vault file; `unreachable` empty.
2. Drive-only item whose drive is in `connectedDrives` → `sendable`, `originalURL == drive.absoluteURL(forRelativePath: driveRelPath)`.
3. Drive-only item whose drive is **absent** → `unreachable`, `driveName` = that drive's root basename; `sendable` excludes it.
4. Mixed selection across **two** drives, one connected one not → connected drive's items + all local items in `sendable`; absent drive's items in `unreachable` tagged with the right name.
5. `SendItem` fields (hash, fingerprint size/captureDate, displayName) are populated correctly from the `TimelineItem` for both local and drive sources.

**App:** build-verified (0 compiler warnings) + manual pass — pre-flight warning shows the right grouped thumbnails and counts; Send N sends only the reachable ones; everything-unreachable disables Send; mixed local+drive selection with the drive connected sends both from the right sources.

## 10. Scope summary

Roughly three tasks:

1. **Core resolver** — `LibraryService+SendSource.swift` (`SendSourcePlan`, `UnreachableSendItem`, `resolveSendSources`) + full unit tests.
2. **AppState glue** — `sendPlan(for:)`, refactor `send(_:to:progress:)` to `[SendItem]`, remove `sendItem(for:)`; build-verified.
3. **SendSheet pre-flight** — compute plan, grouped-by-drive warning with thumbnails + Send-N/Cancel, disable Send when nothing reachable; build-verified + manual.

No format-doc change. `SendEngine` / destinations / `sends.jsonl` untouched.
