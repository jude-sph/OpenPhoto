# v0.1.27 bundle — design

**Date:** 2026-06-15
**Status:** Approved (pending spec review)

Four independent changes bundled into release 0.1.27:

1. Green "In library" badge on import tiles (replace icon + black text split).
2. Face shuffle persistence fix (shuffle silently resets).
3. Face grouping **sensitivity** control in Settings (3k unsorted faces).
4. Parallel photo imports.

None of these touch the on-disk vault format (manifest/sidecar/catalog snapshot schemas
are unchanged — the registry change rewrites the *same* `imports.jsonl` content, just
batched), so no `docs/format/` update is required for this release.

---

## 1. Green "In library" badge

**File:** `Sources/OpenPhotoApp/Devices/ImportItemCell.swift`

### Problem
A photo already present anywhere in the catalog currently surfaces as two separate things:
a black `externaldrive.fill` glyph pinned bottom-left (`badges`, lines 35-39), and — only in
the rare registry-knows-it-but-not-in-catalog case — a faint black "Already in library" text
pill (`statusBadge`, line 56). The signal is split between an icon and a banner.

### Change
Consolidate the in-catalog signal into a single green status pill reading `✓ In library`.

- The `inLibrary && !importedThisSession` case renders the green pill (checkmark + text),
  reusing the existing `statusText` pill geometry but with `Theme.green` background (instead of
  `.black.opacity(0.55)`) and white foreground. Add a leading checkmark glyph.
- Keep the rare `alreadyImported && !inLibrary` ("known to registry, since binned") case as its
  existing faint text — it is a genuinely different state and should not read as "in library".
- Remove the separate `externaldrive.fill` glyph branch (its meaning is now carried by the pill).

### Acceptance
- A photo in the catalog shows one green `✓ In library` pill, no separate drive glyph.
- "Imported ✓" (green) and "Sent from here" (blue) badges are unaffected.
- The since-binned edge case still shows its faint "Already in library" text.

## 2. Face shuffle persistence fix

**File:** `Sources/OpenPhotoApp/AppState.swift`

### Problem (root-caused)
`loadPeople()` assigns the freshly-queried bucket directly:
`self.otherFaceIDs = result.other` (line 287). The catalog query returns faces in
`quality DESC, id` order, so this wipes any user Shuffle. `loadPeople()` fires from:
- background derivation, every 200 completed faces (line 1847),
- background derivation end (line 1858),
- `setPersonCover()` (line 662).

So after a shuffle, the order snaps back the next time any of these run. The
`preservingOtherOrder(_:)` helper (lines 298-303) already solves this and is correctly used by
`refreshSuggestions`, `hideFaces`, `unhideFaces` — `loadPeople` just doesn't call it.

### Change
- `loadPeople()`: `self.otherFaceIDs = preservingOtherOrder(result.other)`. Safe on first load
  (empty prior order → returns `fresh` unchanged).
- `setPersonCover()` (line 662): replace its `loadPeople()` call with `refreshSuggestions()`,
  which already preserves order and is the cheaper path (cover change doesn't need re-clustering).

### Acceptance
- Shuffle the Other-faces bucket; assign/hide some faces; let background derivation tick — the
  shuffled order survives (surviving faces keep position, new faces append, removed faces drop).
- Changing a person's cover does not reshuffle the Other bucket.

## 3. Face grouping sensitivity control

**Files:** `Sources/OpenPhotoApp/AppState.swift`, `Sources/OpenPhotoApp/Settings/SettingsView.swift`

### Problem
The clustering knobs are hard-coded private constants (`faceClusterEps=0.5`,
`faceClusterMinPts=3`, `faceMatchThreshold=0.55`). With ~3k faces stuck in the Other bucket,
the user wants to make grouping more permissive (loosen) to pull more faces into clusters.

### Design — single sensitivity slider, applied on rescan

**State (AppState):**
- `faceSensitivity: Double` — the slider position, `0.0` (Strict) … `1.0` (Loose), default `0.5`.
  Persisted to `UserDefaults` key `"faceSensitivity"` via `didSet`, mirroring the `gridMinSize`
  pattern. This is the *pending* value.
- `appliedFaceSensitivity: Double` — the value `loadPeople()` actually uses. Also persisted (key
  `"faceSensitivityApplied"`), initialized to the same default. Committed `= faceSensitivity`
  **only** inside `rescanFaces()`.

**Mapping** (replace the three `private let` constants with computed values derived from
`appliedFaceSensitivity` `s`). The default position `s = 0.5` reproduces today's constants
exactly, so a user who never moves the slider sees no change after a rescan:

| Sensitivity | eps  | minPts | matchThreshold |
|-------------|------|--------|----------------|
| 0.0 Strict  | 0.45 | 4      | 0.50           |
| 0.5 default | 0.50 | 3      | 0.55           |
| 1.0 Loose   | 0.60 | 2      | 0.60           |

- `eps = s <= 0.5 ? 0.45 + 0.10*s : 0.50 + 0.20*(s - 0.5)` (piecewise-linear, knee at the
  default so `s=0.5 → 0.50`; Strict `0.45`, Loose `0.60`)
- `minPts = Int((4 - 2*s).rounded())`, clamped to `2...4` (`s=0→4`, `0.5→3`, `1→2`)
- `matchThreshold = 0.50 + 0.10*s` (`s=0→0.50`, `0.5→0.55`, `1→0.60`)

`loadPeople()` and `refreshSuggestions()` read these computed values (currently they snapshot
`faceClusterEps`/`faceClusterMinPts`/`faceMatchThreshold` into locals — keep that shape, just
source from the computed values).

**Apply semantics ("on next rescan, existing groupings strictly intact"):**
- Moving the slider updates `faceSensitivity` only. It does **not** call `loadPeople()`, so the
  displayed grouping does not change until the user rescans.
- `rescanFaces()` sets `appliedFaceSensitivity = faceSensitivity` first, then runs its existing
  flow (`clearDerivationJobs(FaceStage)` → `loadPeople()` → `pokeDerivation()`).
- Named people and confirmed/manual faces are never re-clustered — clustering only ever runs over
  the unassigned `auto` pool, and rescan preserves `personID` for confirmed faces (existing
  behavior, unchanged). So a sensitivity change can only re-group *unsorted* faces.

**Settings UI** (`SettingsView.library`, "People & Faces" section, above the Rescan button):
```
People & Faces
Grouping sensitivity
Strict  ◯────────●────────  Loose
Slider(value: $state.faceSensitivity, in: 0...1)
"Lower groups more cautiously (fewer wrong matches); higher pulls more loose faces into
 groups. Takes effect next time you Rescan Faces."
[ Rescan Faces… ]
```

### Caveat (documented, not fixed here)
A portion of the Other bucket is quality-gated faces stored with empty embeddings
(`FaceStage` gate: tiny/blurry/extreme-pose). They are non-clusterable at any sensitivity. The
slider shrinks the *noise* portion of Other, not that gated floor. No UI promise is made about
clearing those.

### Acceptance
- Slider persists across launches; default mid.
- Moving the slider alone does not change the People screen.
- Drag to Loose → Rescan Faces → more faces appear in suggested clusters / fewer in Other.
- A named person's faces are unchanged across the rescan.

## 4. Parallel photo imports

**Files:** `Sources/OpenPhotoCore/Import/ImportEngine.swift`,
`Sources/OpenPhotoCore/Import/ImportRegistry.swift`, tests.

### Problem
Stage 2 of `ImportEngine.run()` — fetch → hash → dedup (lines 83-109) — is a strictly serial
`for` loop and dominates import wall-clock. A second hidden serializer: `ImportRegistry.append`
takes an `NSLock` and **rewrites the entire `imports.jsonl`** on every call, so N skip/import
records cost O(N²).

### Design

**Bounded-concurrency Stage 2.** Replace the serial loop with a bounded `withTaskGroup`,
following the existing 8-wide `VolumeSource` EXIF pattern (lines 65-84). Each task performs, for
one item: `source.fetch` → `ContentHash.ofFile` → `catalog.hashPresent` and returns a per-item
outcome:

```
enum Outcome { case staged(item, url, hash); case skipped(item); case failed(item, reason) }
```

- **Width:** `min(6, ProcessInfo.processInfo.activeProcessorCount)`. Balances CPU-bound hashing
  against single-device source I/O (SD cards don't parallelize reads well; over-wide thrashes).
- **Memory:** `ContentHash.ofFile` already drains a per-1MB-chunk autorelease pool internally, so
  K concurrent hashes hold ~K×1MB, not K×filesize. No new pools needed. (`fetch` copies the whole
  file to staging first, then hashes from disk — bounded.)
- **Ordering:** collect outcomes into an index-keyed buffer so the subsequent placement loop and
  progress reporting stay deterministic regardless of completion order (same index-tuple technique
  as `VolumeSource`).
- **Progress:** a monotonic completed counter incremented as tasks finish, reported through the
  existing `@Sendable progress` callback (UI already marshals to `@MainActor`).

**Batched registry writes.** Add to `ImportRegistry`:
```
public func appendBatch(_ entries: [Entry]) throws
```
Acquires the lock once, applies the same per-entry idempotency guard (`byKey[key] == nil`), and
performs a **single** atomic `imports.jsonl` rewrite for the whole batch. Stage 2 (skips) and
Stage 6 (verified imports) collect their entries and flush once via `appendBatch` instead of
calling `append` per item. `append` stays for any remaining single-entry callers.

**Stage 3 (place/move) stays serial.** Moves are cheap metadata ops; serial avoids
directory-creation / collision-naming races for negligible time cost.

### Correctness (preserved, called out)
- **In-batch duplicate originals:** two identical *new* files in one import both pass the
  catalog dedup check (catalog isn't updated until post-import rescan) and both place — exactly as
  today — collapsing to one asset with two instances at rescan. Behavior unchanged.
- **Dedup reads** (`hashPresent`) run on the catalog's serial DB queue; correct under concurrency,
  and cheap, so they are not a parallelism win but not a hazard either. The win is overlapping
  fetch (I/O) with hashing (CPU) across items.
- **Registry idempotency** is identical under `appendBatch` (same guard).

### Testing
- `ImportRegistryTests`: `appendBatch` writes all entries in one rewrite; dedups within the batch;
  is idempotent vs. already-present keys; produces byte-identical `imports.jsonl` to N×`append`.
- `ImportEngineTests`: a multi-file import with a mix of new / already-in-catalog / failing-fetch
  items yields the same `imported`/`skipped`/`failed` partition as the serial version, with
  result ordering stable; a batch containing duplicate originals still imports correctly.

### Acceptance
- Importing a folder of K files is materially faster than serial on a multicore Mac.
- All existing import tests pass; new concurrency/batch tests pass.
- No OOM growth on a large import (per-chunk pools retained).
