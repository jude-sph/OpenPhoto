# Phase 4.5 — Re-verify "sent to device" Locations on connect (design)

**Date:** 2026-06-11
**Branch:** `phase4.5-locations-reverify` (off `main`)
**Status:** Approved
**Phase:** 4 (Intelligence) — fifth and final slice. The smallest of the five. Builds on Slice 2's **Send / Locations** machinery (`sends.jsonl` via `SendRegistry`, the `SendDestination.enumeratePresent()` read-only re-enumeration, `PresenceFingerprint.looselyMatches`, the `PresenceService` Locations panel) and the Phase-3/4 **device-connect signal** the auto-scan already hooks (`DeviceWatcher.onVolumesChanged` → `AppState.autoScanConnectedDrives`).

> The slice that makes the inspector **Locations** panel *honest going forward*. Today the panel shows a "**sent** to *Jude's iPhone*" indicator straight out of `sends.jsonl` — a **historical** record of a confirmed send, never re-checked. When the user later deletes those photos off the phone, OpenPhoto keeps claiming they're there. 4.5 closes that gap: when a device (iPhone via ImageCaptureCore / SD card / volume) **connects**, OpenPhoto **re-enumerates it read-only** and reconciles each previously-sent file against what's actually on the device, then updates the indicator — **"on this device (confirmed *date*)"** if present, or downgrades to **"sent *date* — no longer on the device"** if it's gone. Pure, fingerprint-based, per-device-connect, cheap. **Never writes to the device.**

---

## 1. Goal

Make the inspector's "sent to device" Locations indicator reflect **current reality** rather than a stale send-time snapshot — without ever mutating the device.

Three things change, all by **reusing existing machinery**:

1. **Reconcile on connect.** When a device connects, take its `sends.jsonl` entries (what we believe we put there) and its **current** read-only listing (what's actually there now), and classify each previously-sent item `present` or `gone` by the established **size + capture-second fingerprint** match.
2. **Upgrade or downgrade the indicator.** A *present* item's device location goes from `.believed` ("sent *date*") to `.confirmed` ("on this device, confirmed *date*"). A *gone* item is downgraded to a new `.stale` confidence rendered as "sent *date* — no longer on the device".
3. **Cache it, rebuildably.** The per-device reconciliation result is held in an **in-memory cache** on `AppState` and fed into `PresenceService.locations(forHash:)` so the panel updates instantly. It is **machine-derived and fully rebuildable** (re-running on the next connect re-derives it) — so there is **no catalog migration and no on-disk format change**.

Sovereignty holds cleanly: the reconciliation reads `sends.jsonl` (the durable confirmed-send record, untouched) and a **read-only** device listing, and writes the result **only** to a rebuildable in-memory presence cache. Nothing is written to the device; nothing new is persisted on disk. The send spine (`SendEngine`/`SendDestination`/`sends.jsonl`) is **reused unchanged**.

### Decisions locked in brainstorming (Jude-approved, autonomous)

- **Trigger = device/volume connect**, hooked into the **same `DeviceWatcher` signals** the auto-scan already uses (`onVolumesChanged` for volumes/SD; `ICDeviceBrowserDelegate.didAdd` for iPhones). One re-verify pass per connected device that has prior sends. This **mirrors** how Phase-3/4 hooked auto-scan / `applyPendingFolderOps` on connect.
- **Re-enumeration is the existing send-verification path, read-only.** `SendDestination.enumeratePresent()` is exactly the AirDrop send-verification re-enumeration (size+capture-date fingerprints for phones; size+capture+hash for volumes). 4.5 calls it and does **not** call `send(...)`. No bytes are written to the device.
- **Matching = the existing fingerprint match.** `PresenceFingerprint.looselyMatches` (size + capture-second; unknown capture date never matches) is the proven matcher used by both the live send-dedup and the AirDrop verification poll. The reconciler reuses it; for volumes the authoritative content `hash` (when present) is preferred, exactly as `SendEngine.isPresent` does.
- **A pure Core matcher, TDD'd.** A new `SendReverifier` value type takes the device's `sends.jsonl` entries + the current listing and returns a per-entry `present`/`gone` verdict. It is **pure** (no I/O, no ImageCaptureCore, no FileManager) and **unit-tested on synthetic data** (mock entries + mock listings → correct verdicts; empty listing, partial matches, hash-vs-fingerprint).
- **Result lives in a rebuildable in-memory cache → `PresenceService`.** No new catalog table. `PresenceService` gains an injected `reverified: [String: ReverifyVerdict]` (keyed by `destinationKey|hash`) it consults when emitting the sent-to-device location; absent an entry, behavior is **identical to today** (`.believed`). A new `Location.Confidence.stale` renders the "no longer on the device" downgrade.
- **No device writes, ever.** Read-only re-enumeration only. (The AirDrop path's `enumeratePresent()` does open the ICC session but never deletes/pushes; the volume path only reads the filesystem.)
- **Hardware-test deferred.** The live device re-enumeration can't be unit-tested (needs a real iPhone/SD card). The **pure matcher is fully unit-tested** on synthetic data; the live pass is added to `docs/manual-test-checklist.md`.

### Non-goals (out of scope)

- **Writing to the device / re-sending the gone items.** 4.5 only *reports* gone; it never re-pushes. (A "re-send the missing ones" action could layer on later; YAGNI now.)
- **Hashing iPhone files.** Phones expose no cheap content hash (ImageCaptureCore), so phone matching stays at the size+capture-second fingerprint — exactly as the send verification already does. Volumes, which do expose a hash, use it when available.
- **Persisting the reverify verdict on disk / a catalog table (v10).** The verdict is a deterministic function of `sends.jsonl` + a live listing; it is re-derived on every connect. Holding it in memory (rebuilt per connect) is correct and avoids any migration. (If a future need for cross-session persistence appears, a rebuildable `send_presence` catalog table is the documented alternative behind the same `SendReverifier` signature — **not** built now.)
- **Reconciling imports / drive-vault presence.** Those Locations lanes (`.historical` imports, `.confirmed` registered drive vaults) are already honest — drive vaults are reconciled by the existing drift scan, imports are inherently historical. 4.5 touches **only** the **sent-to-device** (`.believed`) lane.
- **A background poller / re-verify while the device stays connected.** One pass per connect is enough and cheap; no timer.
- **Changing the send flow.** `SendEngine`, `SendDestination`, `VolumeCopyDestination`, `AirDropDestination`, `sends.jsonl` schema are all **reused unchanged**.

## 2. The existing machinery this slice reuses (explored — file:line)

This feature is almost entirely composition of parts that already exist. The exploration found every piece:

### 2.1 `sends.jsonl` — the confirmed-send record

`Sources/OpenPhotoCore/Send/SendRegistry.swift`. A durable JSONL of every **confirmed** send, in the primary vault's `.openphoto/` (vault-format-v1 §13). Each `SendRegistry.Entry` (`SendRegistry.swift:6-27`) records exactly what 4.5 needs to reconcile:

- `hash` — the library asset's content hash (the Locations key).
- `destinationKey` — the device key (`cam-<serial>` for phones, `vol-<uuid>` for volumes); **same keyspace** as `SendDestination.destinationKey` and `ImportSource.sourceKey`.
- `deviceName` / `deviceKind` — friendly name + `phone`/`volume`.
- `sentAt` / `confirmedAt` — ISO-8601 ms timestamps (the date the panel shows).
- **`fpSize` / `fpCaptureDateMs`** — the **size + capture-date fingerprint** that was proven byte-stable by the round-trip spike. *This is what we re-match against the live device listing.*

Readers already in place: `entries(forHash:)` (`SendRegistry.swift:63-66`, used by `PresenceService`), `entries(forDestinationKey:)` (`SendRegistry.swift:57-60`, exactly the per-device set 4.5 reconciles), and `wasSentToDevice(destinationKey:size:captureDateMs:)` (`SendRegistry.swift:71-78`) which already demonstrates the size + capture-**second** fingerprint match (`fpCaptureDateMs / 1000 == captureDateMs / 1000`). Writer `append(_:)` is atomic. **4.5 only reads `sends.jsonl`; it never writes it.**

### 2.2 The read-only re-enumeration (= the send-verification path)

The design note "the AirDrop send-verification already re-enumerates by size+capture-date fingerprint" is `Sources/OpenPhotoApp/Send/AirDropDestination.swift`:

- `enumeratePresent() async throws -> [PresenceFingerprint]` (`AirDropDestination.swift:26-35`) opens the ICC camera session (`camera.open()`), enumerates the phone's items, and maps each to a `PresenceFingerprint(size:captureDateMs:hash: nil)`. **No hash** (phones can't be cheaply hashed). This is **read-only** — it never deletes or pushes.
- The AirDrop `send(...)` verification loop (`AirDropDestination.swift:59-70`) is the existing "is it on the device yet?" poll: it re-enumerates and matches each item with `present.contains { $0.looselyMatches(item.fingerprint) }`. **4.5 reuses this enumeration + match, just without the preceding `send`.**

Volumes: `Sources/OpenPhotoCore/Send/VolumeCopyDestination.swift` `enumeratePresent()` (`VolumeCopyDestination.swift:21-36`) walks the destination subfolder read-only and emits `PresenceFingerprint(size:captureDateMs:hash:)` **with** a content hash (`ContentHash.ofFile`). So volume reconciliation can match on the authoritative hash.

The matcher itself: `Sources/OpenPhotoCore/Send/SendDestination.swift`:
- `PresenceFingerprint` (`SendDestination.swift:9-22`) + `looselyMatches(_:)` (`SendDestination.swift:18-21`) — size equal **and** capture-second equal, with `captureDateMs != 0` guard (unknown date never matches). **This is the matcher 4.5 reuses verbatim.**
- `SendEngine.isPresent(_:in:)` (`SendEngine.swift:80-85`) — the two-layer dedup precedent: authoritative `hash` when the target exposes it (volumes), else `looselyMatches`. **The `SendReverifier` mirrors exactly this two-layer logic.**
- The `SendDestination` protocol (`SendDestination.swift:62-70`) with `enumeratePresent()` is built per connected device by `AppState.sendDestination(for:)` (`AppState.swift:1378-1386`) — **already wired** for both AirDrop (phones) and `VolumeCopyDestination` (volumes/SD).

### 2.3 `PresenceService` + the inspector Locations panel

`Sources/OpenPhotoCore/Presence/PresenceService.swift`. `locations(forHash:) -> [Location]` (`PresenceService.swift:41-81`) composes the panel from the catalog + the import/send/device registries. The **sent-to-device** block (`PresenceService.swift:64-71`) emits, for each `sends.entries(forHash: hash)`, a `Location(place: .device(...), confidence: .believed, detail: "sent " + day(confirmedAt))`. **This is the single block 4.5 modifies** — to consult the reverify cache and emit `.confirmed` (present) or `.stale` (gone) instead of always `.believed`.

`Location.Confidence` (`PresenceService.swift:12`) is currently `confirmed | believed | historical`; 4.5 **adds `.stale`**. The inspector renders the badge in `Sources/OpenPhotoApp/Inspector/InspectorView.swift` — `confidenceBadge(_:)` (`InspectorView.swift:323-333`) maps confidence → glyph/label; `locationRow(_:)` (`InspectorView.swift:286-299`) shows the `detail` line. 4.5 adds a `.stale` case (a muted "not on device" badge) and the present/gone wording flows through `Location.detail`.

`AppState.locations(for:)` (`AppState.swift:1281-1283`) → `presenceService()?.locations(forHash:)` and `presenceService()` (`AppState.swift:1273-1278`) build the `PresenceService` from the live registries — the exact construction point where the reverify cache is injected.

### 2.4 `DeviceWatcher` — the connect signal to hook

`Sources/OpenPhotoApp/Devices/DeviceWatcher.swift`. Two connect signals, both already consumed by `AppState`:

- **Volumes / SD cards:** `onVolumesChanged` (`DeviceWatcher.swift:51`), fired on `NSWorkspace.didMount/didUnmount` and at start (`DeviceWatcher.swift:93-113`). `AppState` already hooks it (`AppState.swift:1143-1155`) to call `reloadDrives()` + `await autoScanConnectedDrives()`. **The re-verify pass slots in right here**, beside `autoScanConnectedDrives()` — the exact pattern the prompt names.
- **iPhones:** `ICDeviceBrowserDelegate.deviceBrowser(_:didAdd:)` (`DeviceWatcher.swift:117-128`) appends a `.camera(...)` device. 4.5 adds (or reuses) a connect callback so a freshly-added camera triggers the re-verify too — modeled on `openedDeviceRemoved` (`DeviceWatcher.swift:37`). `connectedSendTargets()` (`AppState.swift:1372-1374`) already lists every device we can build a `SendDestination` for.

**No presence catalog table exists** — `PresenceService` derives everything in-memory each call. That is why 4.5 needs **no migration**: the reverify result is just another in-memory input to that same derivation.

## 3. The `SendReverifier` — the pure, unit-tested heart

`Sources/OpenPhotoCore/Send/SendReverifier.swift`. A pure value type. **No network, no ImageCaptureCore, no FileManager** — given the `sends.jsonl` entries for one connected device and that device's current listing (a `[PresenceFingerprint]`, the very type `enumeratePresent()` returns), it returns a verdict per sent entry. Fully unit-testable on hand-built synthetic data.

### 3.1 Shape

```swift
/// The reconciliation verdict for one previously-sent asset on a now-connected device.
public enum ReverifyVerdict: String, Sendable, Equatable {
    case present   // still on the device (re-enumeration found a fingerprint/hash match)
    case gone      // we sent it (confirmed) but it's no longer on the device
}

/// Pure reconciler: given a device's confirmed-send entries (from sends.jsonl) and that
/// device's CURRENT read-only listing, decide present/gone per entry. No I/O — the
/// unit-tested heart. The live enumeration + the connect hook are the App's job (untestable
/// without hardware); this classification is fully synthetic-testable.
public struct SendReverifier: Sendable {
    public init() {}

    /// Reconcile one device's confirmed sends against its current listing.
    /// - Parameters:
    ///   - entries:  sends.jsonl entries for this destinationKey (SendRegistry.entries(forDestinationKey:)).
    ///   - present:  the device's current contents (SendDestination.enumeratePresent()).
    /// - Returns: verdict per entry, keyed by the entry's `hash`. Two-layer match: a present
    ///   fingerprint's authoritative `hash` (volumes) is matched against the entry hash first;
    ///   else the size + capture-SECOND fingerprint (phones), reusing PresenceFingerprint.looselyMatches.
    public func reconcile(entries: [SendRegistry.Entry],
                          present: [PresenceFingerprint]) -> [String: ReverifyVerdict]
}
```

### 3.2 The matching rule (mirrors `SendEngine.isPresent`)

For each `SendRegistry.Entry`, it is `present` iff some `PresenceFingerprint` in the device listing matches it, by the **same two-layer rule the send dedup uses** (`SendEngine.swift:80-85`):

1. **Authoritative hash** — if the listing fingerprint carries a `hash` (volumes only) and it equals `entry.hash`, it's a match.
2. **Fingerprint** — else, reconstruct the entry's fingerprint as `PresenceFingerprint(size: entry.fpSize, captureDateMs: entry.fpCaptureDateMs, hash: nil)` and test `listingFP.looselyMatches(entryFP)` — size equal **and** capture-second equal, unknown-date never matches.

No match across the whole listing → `gone`. Because the rule is identical to the send-time dedup/verification, a file that *was* confirmed-sent and is *still there* always re-matches; only a genuine user deletion flips it to `gone`. The result map is keyed by `entry.hash` (one verdict per sent asset on this device); the App stores it under `destinationKey|hash` so per-device verdicts don't collide (a hash sent to two devices).

> **Why pure + synthetic-testable:** the only hardware-dependent step is *producing* the `[PresenceFingerprint]` (ICC enumeration / filesystem walk), which the App does. The *classification* is total over its two inputs — so tests hand-build `SendRegistry.Entry`s and `PresenceFingerprint`s and assert verdicts, no device, no real `sends.jsonl`, no network.

### 3.3 Representative synthetic test (illustrative)

```swift
private func sent(_ hash: String, key: String = "cam-Z",
                  size: Int64, captureMs: Int64) -> SendRegistry.Entry {
    SendRegistry.Entry(hash: hash, destinationKey: key, deviceName: "iPhone", deviceKind: "phone",
        sentAt: "2026-06-08T01:00:00.000Z", confirmedAt: "2026-06-08T01:01:00.000Z",
        fpSize: size, fpCaptureDateMs: captureMs)
}

@Test func reconcilePresentAndGoneByFingerprint() {
    let A = "sha256:" + String(repeating: "a", count: 64)   // still on phone
    let B = "sha256:" + String(repeating: "b", count: 64)   // deleted off phone
    let entries = [sent(A, size: 1000, captureMs: 1_700_000_000_000),
                   sent(B, size: 2000, captureMs: 1_700_000_500_000)]
    // The phone now lists only A's fingerprint (sub-second drift tolerated by capture-second match).
    let present = [PresenceFingerprint(size: 1000, captureDateMs: 1_700_000_000_400, hash: nil)]
    let v = SendReverifier().reconcile(entries: entries, present: present)
    #expect(v[A] == .present)
    #expect(v[B] == .gone)
}

@Test func volumeMatchesOnAuthoritativeHash() {
    let A = "sha256:" + String(repeating: "a", count: 64)
    let entries = [sent(A, key: "vol-U", size: 10, captureMs: 0)]   // capture date unknown (0)
    // A volume listing carries the real content hash → matches even though captureMs == 0.
    let present = [PresenceFingerprint(size: 10, captureDateMs: 0, hash: A)]
    #expect(SendReverifier().reconcile(entries: entries, present: present)[A] == .present)
}

@Test func emptyListingMarksEverythingGone() {
    let A = "sha256:" + String(repeating: "a", count: 64)
    let v = SendReverifier().reconcile(entries: [sent(A, size: 1, captureMs: 1)], present: [])
    #expect(v[A] == .gone)
}

@Test func noEntriesIsEmpty() {
    #expect(SendReverifier().reconcile(entries: [], present:
        [PresenceFingerprint(size: 1, captureDateMs: 1, hash: nil)]).isEmpty)
}
```

## 4. Storage — none new (rebuildable in-memory cache; **no migration, no format change**)

The reverify verdict is **machine-derived and re-derived on every connect**, so it needs no durable home.

- **In-memory cache on `AppState`:** `private var reverified: [String: ReverifyVerdict] = [:]` keyed by `"\(destinationKey)|\(hash)"` (the same composite key `SendRegistry.Entry.key` already uses, `SendRegistry.swift:26`). Populated by the connect-hook pass (§5); read by `presenceService()`.
- **`PresenceService` gains one injected parameter:** `reverified: [String: ReverifyVerdict] = [:]` (defaulted empty → **identical to today's behavior** for every existing caller/test). It is consulted **only** in the sent-to-device block.
- **No catalog table, no migration (no v10).** `Catalog.schemaVersion` stays at **9**. `PresenceService` already derives all of Locations in-memory per call; the reverify map is just one more in-memory input. (The documented-but-not-built alternative — a rebuildable `send_presence` catalog table for cross-session persistence — is explicitly **deferred**; nothing today needs the verdict to survive a quit, since it's cheap to re-derive on the next connect.)
- **No on-disk format change.** `sends.jsonl` is read unchanged; nothing is written to the device; no sidecar; `vault-format-v1.md` is untouched; `catalog-schema.md` is untouched (no new table). This is the cleanest possible sovereignty story: a pure read + an in-memory cache.

## 5. App wiring — re-verify on connect, update the panel

### 5.1 The connect-hook pass (App)

A new `AppState` method, modeled on `autoScanConnectedDrives()` (`AppState.swift:1020-1042`) and invoked from the **same** connect signals:

```swift
/// On device/volume connect, re-enumerate each connected SEND target read-only and reconcile
/// its sends.jsonl entries against what's actually there now — so the inspector Locations
/// "sent to <device>" indicator becomes "on this device (confirmed)" or "no longer on the
/// device". Read-only: never writes to the device. Reuses SendDestination.enumeratePresent().
func reverifySentToConnectedDevices() async {
    guard let sends = sendRegistry else { return }
    for device in connectedSendTargets() {                       // AppState.swift:1372
        guard let dest = sendDestination(for: device) else { continue }   // AppState.swift:1378
        let entries = sends.entries(forDestinationKey: dest.destinationKey)
        guard !entries.isEmpty else { continue }                 // nothing was ever sent here → skip
        let present = (try? await dest.enumeratePresent()) ?? [] // READ-ONLY re-enumeration
        let verdicts = SendReverifier().reconcile(entries: entries, present: present)
        for (hash, verdict) in verdicts {                        // fold into the in-memory cache
            reverified["\(dest.destinationKey)|\(hash)"] = verdict
        }
    }
    try? refreshQueries()    // re-render the inspector/panel against the updated cache
}
```

- **Volumes / SD:** added beside the existing `await self.autoScanConnectedDrives()` in the `onVolumesChanged` closure (`AppState.swift:1143-1155`) — one more `await reverifySentToConnectedDevices()`.
- **iPhones:** a `DeviceWatcher` connect callback (mirroring `openedDeviceRemoved`, `DeviceWatcher.swift:37`) fires `reverifySentToConnectedDevices()` when a `.camera` is added. (`enumeratePresent()` opens the shared ICC session via the cached `CameraSource` — `AirDropDestination.swift:26-27` — and never closes it, per its lifecycle note.)
- **At library-open:** also called once after the initial `autoScanConnectedDrives()` (`AppState.swift:1156`) so an already-plugged device is reconciled on launch.
- **Read-only guarantee:** the pass calls **only** `enumeratePresent()`, never `send(...)`. No bytes reach the device. (`SendEngine`/`SendDestination.send` are not invoked.)

### 5.2 `PresenceService` consults the cache (Core)

The sent-to-device block (`PresenceService.swift:64-71`) changes from always-`.believed` to verdict-driven:

```swift
for e in sends.entries(forHash: hash) where !seenDevices.contains(e.destinationKey) {
    seenDevices.insert(e.destinationKey)
    let name = devices.name(forKey: e.destinationKey) ?? e.deviceName
    let kindV = DeviceKind(rawValue: e.deviceKind) ?? kind(forKey: e.destinationKey)
    let day = day(e.confirmedAt)
    switch reverified["\(e.destinationKey)|\(hash)"] {
    case .present:                                               // re-enumerated and found
        out.append(Location(place: .device(key: e.destinationKey, name: name, kind: kindV),
                            confidence: .confirmed, detail: "on this device, confirmed " + day))
    case .gone:                                                  // user deleted it since
        out.append(Location(place: .device(key: e.destinationKey, name: name, kind: kindV),
                            confidence: .stale, detail: "sent " + day + " — no longer on the device"))
    case nil:                                                    // not yet re-verified (device offline) → today's behavior
        out.append(Location(place: .device(key: e.destinationKey, name: name, kind: kindV),
                            confidence: .believed, detail: "sent " + day))
    }
}
```

- A `.present` verdict surfaces the device as a **`.confirmed`** location → `isOnlyOnThisMac` (`PresenceService.swift:85-92`) already treats `.confirmed` device locations as "backed up", so this is *more* honest, not a regression.
- A `.gone` verdict surfaces **`.stale`** — which `isOnlyOnThisMac` must **not** count as a backup (the file is no longer there). So `isOnlyOnThisMac`'s predicate (`confidence == .confirmed || confidence == .believed`) **stays as-is**: `.stale` is correctly excluded, meaning a photo that was *only* "sent" and is now gone correctly reverts to only-on-this-Mac. (A small, deliberate safety win: a downgraded send no longer masks an only-copy.)
- **`nil` (device not connected this session) → unchanged `.believed`** — exactly today's panel. So the change is purely additive: no connected device, no reverify, no behavior change.

### 5.3 The new confidence in the inspector (App)

`Location.Confidence` gains `.stale` (`PresenceService.swift:12`). The inspector `confidenceBadge(_:)` (`InspectorView.swift:323-333`) adds a `.stale` case — a muted/orange "not on device" glyph (e.g. `iphone.slash` or a faint "removed" label), distinct from the blue "sent". `locationRow` (`InspectorView.swift:286-299`) already renders `detail`, so the "— no longer on the device" wording shows with no extra work; the `.confirmed` "on this device, confirmed *date*" reuses the existing green checkmark badge.

## 6. Data flow

```
device connects (DeviceWatcher.onVolumesChanged / ICDeviceBrowser didAdd)
  → AppState.reverifySentToConnectedDevices()
       for each connected send target with prior sends.jsonl entries:
          dest.enumeratePresent()                 ← READ-ONLY (ICC enumerate / fs walk); no device write
          SendReverifier().reconcile(entries, present)   ← PURE matcher (size+capture-second / hash)
          → reverified["<destKey>|<hash>"] = .present | .gone   ← in-memory cache (rebuildable)
       refreshQueries()                            ← re-render Locations panel

inspector (selected photo) → AppState.locations(for:) → PresenceService.locations(forHash:)
   sent-to-device block consults reverified[...]:
      .present → .confirmed  "on this device, confirmed <date>"   (green check)
      .gone    → .stale      "sent <date> — no longer on the device"  (muted/orange)
      nil      → .believed   "sent <date>"   (unchanged — device not connected this session)

device NOT connected (no enumeration this session) →
   reverified[...] is nil → PresenceService emits today's .believed indicator unchanged (zero behavior change)
```

## 7. Error handling / edge cases

| Case | Behavior |
|---|---|
| Device never connected this session | No reverify pass for it → `reverified[...]` is nil → `PresenceService` emits the existing `.believed` "sent *date*" — **identical to today**. |
| `enumeratePresent()` throws (ICC session fails / volume vanished mid-read) | `(try? …) ?? []` → an empty listing. **Caveat:** an empty listing would mark everything `gone`. To avoid a false downgrade on an *enumeration failure* (vs. a genuinely empty device), the pass treats a **thrown** enumeration as "couldn't check" and **skips updating the cache for that device** (leaves prior verdicts / `.believed`), only reconciling when `enumeratePresent()` **succeeds**. A successfully-enumerated empty device legitimately marks its sends `gone`. (Implemented by distinguishing the `try?`-nil throw from a real `[]`.) |
| Photo sent to two devices, present on one and gone from the other | Verdicts are keyed by `destinationKey\|hash`, so each device's row resolves independently — `.confirmed` for the one that still has it, `.stale` for the one it was deleted from. |
| Volume listing exposes a content hash | Matched on the authoritative `hash` first (mirrors `SendEngine.isPresent`); the fingerprint is the fallback. A hash match is exact and immune to capture-date drift. |
| Phone fingerprint with unknown capture date (`fpCaptureDateMs == 0`) | `looselyMatches` returns false for a 0 capture date (the existing guard) → such an entry can't fingerprint-match and (absent a hash, which phones lack) is marked `gone`. This matches the send-verification's own conservative behavior; in practice confirmed phone sends carry a real capture date. |
| The same hash also has a `.confirmed` drive-vault location or This-Mac copy | Unchanged — those lanes are emitted first (`PresenceService.swift:46-62`) and `seenDevices` already dedups; the sent-to-device reverify only ever *downgrades the device row*, never removes a genuine backup. `isOnlyOnThisMac` stays correct. |
| App quits and relaunches | The in-memory cache is empty on launch → the library-open pass (§5.1) re-derives it from the connected devices. No persistence needed; nothing stale survives. |
| User re-sends a previously-gone item | The next send re-appends to `sends.jsonl` (idempotent by `destinationKey\|hash` — already present, so no-op) and the next connect's reverify flips it back to `.present`. Self-healing. |
| Device connected but nothing was ever sent to it | `entries(forDestinationKey:)` is empty → the device is skipped (no enumeration cost). |
