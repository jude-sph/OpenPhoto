# Phase 4.3 — Faces & People (full management) (design)

**Date:** 2026-06-11
**Branch:** `phase4.3-faces-people` (off `main`)
**Status:** Approved
**Phase:** 4 (Intelligence) — third slice. Builds on Slice 4.1 (the derivation-pipeline spine: the `DerivationStage` protocol + the multi-stage `AppState` runner, `derivation_jobs`, the reachable-URL resolution) and Slice 4.2 (the `Catalog` Float16 pack/unpack idiom, the `SemanticIndex`/Accelerate precedent, the unified `Search` surface with click-to-narrow `SearchFilters`).

> The slice that turns the library into a **People** album: it detects faces in every photo, groups look-alike faces into clusters, lets you **name** a cluster into a Person, and then gives you the full management vocabulary — **merge** two people, **split** a cluster, **reassign** an individual face, with per-face **confidence** shown. Named/confirmed assignments are written to **MWG face regions in XMP sidecars** as fixed anchors that survive re-clustering — the machine never undoes a human decision. It adds the **third derivation stage** (`Faces`), entirely on-device via Apple Vision.

---

## 1. Goal

Let the user **find and organize photos by who is in them**, entirely on-device/offline. Four capabilities:

1. **Detect** — every photo runs Apple Vision face detection in the background pipeline; each detected human face becomes a `faces` row with its bounding rect, a per-face embedding, and a confidence.
2. **Cluster → name** — open the **People** view and the app groups unassigned faces into suggested clusters by embedding similarity; type a name on a cluster → it becomes a **Person** and every face in it is assigned.
3. **Manage** — **merge** two people into one; **split** selected faces off a person into a new person (or back to unassigned); **reassign** an individual face from one person to another via a menu. Each face shows its detection **confidence**.
4. **Filter** — a **person** filter folds into the 4.2 unified Search, so "photos of X" is one click (and composes with date/camera/tag/etc.).

Sovereignty holds with a careful split:

- **Machine-derived → the rebuildable catalog** (a new `faces` table; rects, embeddings, confidences, and *unnamed-cluster* groupings). Dropping it is safe: the pipeline re-detects and re-clusters.
- **Human-authored → XMP sidecars.** The only human bits are **person names** and the **confirmed assignment of a face to a person**. Those are written as **MWG `mwg-rs:Regions`** on the asset's sidecar (per face: a `Name` + a normalized `Area`). This is the durable record; the catalog's `personID`/`name` columns *mirror* it. **On-disk format change → `docs/format/vault-format-v1.md` §5 is updated in the docs task.**

Fully on-device. Vision runs face detection + the per-face feature-print on the ANE/GPU automatically — no bundled face model, no network, no hand-written Metal.

### Decisions locked in brainstorming

- **FULL People management** (not just cluster→name): detect & cluster; name a cluster → Person; **merge** two people; **split** a cluster; **reassign** an individual face; per-face **confidence** shown.
- **Vision-native, fully offline:** Apple Vision face detection (`VNDetectFaceRectanglesRequest`) + a per-face embedding via `VNGenerateImageFeaturePrint` run on the **cropped face region**. We store the feature-print's float vector. This is *decent, not best-in-class* face grouping — the feature-print is a general visual descriptor, not a face-specialized embedding — but it is built-in, offline, and good enough to suggest clusters the user confirms. **The pipeline is model-agnostic** (a `faces.source`/embedding-dim discipline like 4.2's `embeddings.model`), so a better face embedder can replace `VNGenerateImageFeaturePrint` later by re-deriving, with the human-confirmed regions untouched.
- **Auto-run** as a background derivation stage (a third stage alongside OCR/Embed). **Human faces only** — no pets/animals.
- **Named/confirmed faces → MWG face regions in XMP sidecars** (human-authored → portable sidecars; fixed anchors that survive re-clustering). **Unnamed clusters stay catalog-only** — machine guesses never leave the catalog (this is already promised in `vault-format-v1` §5).
- **Person filter folds into the 4.2 unified Search** (a new `person` field on `SearchFilters` + a `faces` join in `Catalog.structuredFilter`; a person control in the `SearchView` filter bar).
- Builds on the 4.1/4.2 spine: the `DerivationStage` protocol + the multi-stage runner (registry currently `[OCRDerivationStage, EmbedStage]` → add `FaceDerivationStage`), `Catalog` (GRDB), the reachable-URL resolution (`goodCopyURL`). **Catalog is now at v7** (4.2's `embeddings`); the faces/people migration is **v8**.

### Non-goals (later slices / phases / out of scope)

- **Pets/animals** — human faces only in 4.3.
- **A bundled face-specialized model** — `VNGenerateImageFeaturePrint` is the v1 embedder; a better model is a future re-derivation (the schema supports it; not built here).
- **Cross-library person identity / training a recognizer** — clustering is unsupervised + human-confirmed; there's no per-person classifier that auto-labels new faces. (Re-clustering pulls newly-detected faces toward confirmed anchors heuristically — §4.3 — but no learned model is trained.)
- **Place/geo work** → 4.4 (reverse-geocode + Map). `SearchFilters` already reserves `place` conceptually; 4.3 only adds `person`.
- **Approximate-NN over face vectors** — clustering is over the *unassigned* set, computed on demand; brute-force is fine at this scale (§4.4). A vector index is YAGNI.

## 2. Face detection + embedding — the `FaceStage`

### 2.1 `FaceStage` (Core, headless — mirrors `OCRStage`)

`Sources/OpenPhotoCore/Derivation/FaceStage.swift` (`import Vision`). Headless + synchronous; the runner calls it off the main actor inside `Task.detached(.utility)`, exactly like OCR/Embed.

```swift
public struct DetectedFace: Sendable {
    public let rect: CGRect      // Vision normalized boundingBox (origin bottom-left, 0…1)
    public let embedding: [Float]
    public let confidence: Float // VNObservation.confidence (0…1)
}

public enum FaceStage {
    public static let id = "faces"
    /// Detect human faces in the image at `url`; for each, crop the boundingBox region and run
    /// `VNGenerateImageFeaturePrint` on the crop → its float vector. Returns one DetectedFace per
    /// face (possibly empty for a photo with no faces), or nil only if the image can't be read.
    public static func detect(in url: URL) -> [DetectedFace]?
}
```

Pipeline per photo:

1. `VNImageRequestHandler(url:)` → `VNDetectFaceRectanglesRequest`. Each `VNFaceObservation` gives `boundingBox` (normalized) + `confidence`.
2. For each observation, crop the source image to the (slightly padded) bounding box → a `CGImage`.
3. Run `VNGenerateImageFeaturePrint` on the crop (a *new* `VNImageRequestHandler` over the crop) → `VNFeaturePrintObservation`; read `.data` → `[Float]` (the feature-print is `Float32`; `elementCount` gives the dim).
4. Return a `DetectedFace` per face. **`nil` only when the source image itself can't be decoded** (so the runner records a *failure*, retry-capped); an image that decodes but has **no faces** returns `[]` → success ("analyzed, no faces").

`source: "auto"` for everything the stage produces; `confidence` from the `VNFaceObservation`. The crop+feature-print is intentionally simple and model-agnostic — swapping `VNGenerateImageFeaturePrint` for a face-specialized embedder later is a localized change here + a re-derivation.

### 2.2 `FaceDerivationStage` (the `DerivationStage` conformance)

A `struct FaceDerivationStage: DerivationStage` (same file) wires `FaceStage` into the runner:

```swift
public struct FaceDerivationStage: DerivationStage {
    public let id = "faces"
    public let eligibleKind = "photo"
    public init() {}
    public func run(hash: String, url: URL, catalog: Catalog) async -> Bool {
        guard let faces = FaceStage.detect(in: url) else { return false }  // unreadable → failure
        try? catalog.replaceFaces(forHash: hash, with: faces.map {
            FaceRow(hash: hash, rect: $0.rect, embedding: $0.embedding,
                    confidence: $0.confidence, source: "auto")
        })
        return true   // [] is still success — "analyzed, no faces"
    }
}
```

- `eligibleKind = "photo"`; `isAvailable = true` (Vision is built-in — no model file, so unlike `EmbedStage` it never skips for a missing resource).
- **Resumable + retry-capped** for free via `derivation_jobs` (a `"done"` row for `(hash, "faces")` means detected). `Catalog.eligibleKind(forStage:)` already returns `"photo"` for the `default` case, so `"faces"` is photo-eligible with no change — but we add an explicit `case "faces": return "photo"` for clarity.
- `replaceFaces(forHash:with:)` is delete-then-insert of the **auto** rows for that hash (idempotent re-derivation); it must **not** clobber human-confirmed regions — see §3.2.

### 2.3 Why store the embedding in the catalog

The face vectors are **machine-derived** and rebuildable, so they live in the catalog only (the `faces.embedding` BLOB), packed as **Float16** reusing 4.2's `packFloat16`/`unpackFloat16` idiom. They never go to sidecars. Only the human decision (this face is *Alice*) is durable, and that travels as an MWG region (§3.2).

## 3. Storage — catalog migration **v8** + sidecar regions

### 3.1 Catalog tables (migration v8)

Machine-derived/rebuildable **except** `personID`/`name`, which **mirror** human decisions recorded in sidecars (the catalog can always be rebuilt by re-detecting + re-reading the sidecar regions).

```
faces(id        INTEGER PRIMARY KEY AUTOINCREMENT,
      hash      TEXT NOT NULL,        -- references assets.hash (the photo the face is in)
      rectX     REAL NOT NULL,        -- Vision normalized boundingBox.origin.x  (0…1)
      rectY     REAL NOT NULL,        -- Vision normalized boundingBox.origin.y  (0…1, bottom-left)
      rectW     REAL NOT NULL,        -- normalized width
      rectH     REAL NOT NULL,        -- normalized height
      embedding BLOB NOT NULL,        -- dim × Float16 LE feature-print vector (machine-derived)
      dim       INTEGER NOT NULL,     -- embedding dimensionality (VNGenerateImageFeaturePrint dim)
      personID  INTEGER,              -- NULL = detected-but-unassigned; else → people.id
      confidence REAL NOT NULL,       -- VNFaceObservation.confidence (0…1)
      source    TEXT NOT NULL)        -- "auto" (machine) | "confirmed" (human-assigned to a person)

people(id          INTEGER PRIMARY KEY AUTOINCREMENT,
       name        TEXT NOT NULL,
       createdAtMs INTEGER NOT NULL)
```

- Indexed on `faces.hash` (face lookup per photo) and `faces.personID` (a person's faces + the Search join).
- No FK on `faces.hash → assets.hash` — same rationale as `instances` (rebuildable; a re-detect repairs orphans).
- `source` transitions `"auto" → "confirmed"` when a human assigns the face to a person; a `"confirmed"` row is a **fixed anchor** — re-detection (§3.2) never deletes or re-rects it.
- `embedding`/`dim` mirror 4.2's `embeddings`: Float16 on disk, fp32 in memory for clustering.

### 3.2 The auto/confirmed split — re-derivation safety

`replaceFaces(forHash:with:)` (the stage's idempotent write) deletes only the **`source = 'auto'`** rows for that hash and inserts the freshly-detected ones. **Confirmed rows are never touched by the machine.** This guarantees the invariant *the machine never undoes a human decision*: re-running detection on a photo with a confirmed "Alice" region re-detects the auto faces but leaves Alice's confirmed row (and its MWG region) intact.

When a confirmed face's auto-detected twin re-appears, de-dup is by **rect overlap** (IoU above a small threshold against existing confirmed rows for that hash) — a re-detected auto face that overlaps a confirmed region is dropped, so we don't end up with two rows for the same physical face.

### 3.3 CRUD (Core, TDD)

All on `Catalog` (extension `Catalog+Faces.swift`). `FaceRow` is a small value type (`hash`, `rect: CGRect`, `embedding: [Float]`, optional `id`, `personID`, `confidence`, `source`).

```swift
// — Detection write (stage) —
func replaceFaces(forHash hash: String, with rows: [FaceRow]) throws   // delete auto rows + insert; keep confirmed
func insertFaces(_ rows: [FaceRow]) throws -> [Int64]                  // low-level insert, returns new ids
func faces(forHash hash: String) throws -> [FaceRow]                   // all faces in one photo (for region overlay)

// — Clustering inputs —
func unassignedFacesWithEmbeddings() throws -> [(id: Int64, vector: [Float])]  // personID IS NULL, source 'auto'

// — People CRUD —
func createPerson(name: String) throws -> Int64
func people() throws -> [PersonRow]                                    // id, name, faceCount, representativeFaceID
func deletePerson(_ id: Int64) throws                                  // faces revert to unassigned (personID=NULL, source='auto')

// — Assignment / management —
func assignFaces(_ ids: [Int64], to personID: Int64) throws           // set personID + source='confirmed'
func reassignFace(_ id: Int64, to personID: Int64?) throws            // nil → unassign (personID=NULL, source='auto')
func mergePerson(_ src: Int64, into dst: Int64) throws                // move src's faces to dst, delete src
func faces(forPerson personID: Int64) throws -> [FaceRow]             // a person's faces (→ their photos)
```

Notes on the human/machine boundary:

- `assignFaces`/`reassignFace(to: non-nil)` flip `source` to `"confirmed"` — these are the only writers that create anchors. The **App** pairs each such catalog write with a sidecar MWG-region write (§3.4) so the durable record and the mirror stay in sync. `reassignFace(to: nil)` and `deletePerson` revert to `source='auto'` and, in the App, **remove** the face's region from the sidecar.
- `mergePerson(src, into: dst)` re-points every `src` face to `dst` (kept `confirmed`) and deletes the now-empty `src` person row. Sidecar regions for those faces are rewritten with `dst`'s name.
- `people()` returns a representative face id (e.g. the highest-confidence confirmed face) + a count, for the People-view cards.
- `personID`/`name` are the **only human-authored columns**; everything else (rects/embedding/dim/confidence/`source='auto'`) is machine-derived. The catalog is fully rebuildable: re-detect (machine) + re-read the sidecar regions (human) reconstructs every row.

### 3.4 MWG sidecar regions (Core, TDD) — the format change

Extend the existing XMP sidecar read/write (`Sources/OpenPhotoCore/Sidecar/XMP.swift` + `SidecarData.swift`; written/read via `SidecarStore`) to round-trip **`mwg-rs:Regions`**. Per named/confirmed face: a region with a `Name` and an `Area`:

```xml
<mwg-rs:Regions rdf:parseType="Resource">
  <mwg-rs:RegionList>
    <rdf:Bag>
      <rdf:li rdf:parseType="Resource">
        <mwg-rs:Name>Alice</mwg-rs:Name>
        <mwg-rs:Type>Face</mwg-rs:Type>
        <mwg-rs:Area stArea:x="0.42" stArea:y="0.31" stArea:w="0.18" stArea:h="0.24"
                     stArea:unit="normalized"/>
      </rdf:li>
    </rdf:Bag>
  </mwg-rs:RegionList>
</mwg-rs:Regions>
```

- Namespaces: `mwg-rs="http://www.metadataworkinggroup.com/schemas/regions/"`, `stArea="http://ns.adobe.com/xmp/sType/Area#"`. Added alongside the existing `xmp`/`dc` namespaces on the `rdf:Description`.
- **`Area` is MWG-normalized** (`x`/`y` are the region **center**, `w`/`h` the size; `unit="normalized"`, all 0…1). `Type="Face"`.
- **Coordinate convention:** the catalog stores Vision's normalized boundingBox (origin **bottom-left**, x/y = lower-left corner). The MWG `Area` uses the **center** with a **top-left** origin. The conversion (`vision rect ↔ mwg center/size`, flipping y) lives in one place (`FaceRegion`), is symmetric, and is round-trip tested. This keeps sidecars portable (MWG is what Lightroom/Apple Photos/digiKam read) while the catalog keeps Vision's native frame.
- `SidecarData` gains `var faces: [FaceRegion]` (`FaceRegion = (name: String, rect: CGRect)` in *MWG-center-normalized* form, or a small struct). `XMP.serialize` emits the `Regions` block when non-empty; `XMP.parse` reads it back. Existing rating/favorite/caption/tags handling is unchanged and the round-trip stays lossless for them.
- **On ingest the regions are read back as fixed anchors:** when a photo (re)enters the catalog or its sidecar is read, its confirmed face regions are reconstituted as `source='confirmed'` `faces` rows (matched to the owning person by `Name`, creating the `people` row if absent). This is what makes a confirmed assignment survive a catalog rebuild and travel on a drive.

**This is an on-disk format change → `vault-format-v1.md` §5 (Sidecars) MUST be updated** to specify the concrete MWG region shape (it currently only names "MWG Regions … `Type="Face"` with `Name`" in the table + the §5 note). The docs task adds the normative subsection.

### 3.5 Snapshot / portability

`faces`/`people` ride along in the per-drive `catalog-snapshot/catalog.sqlite` (VACUUM INTO) like every other catalog table. `faces` is a **droppable, rebuildable cache** (re-detect + re-read sidecars). The **authoritative** record of a confirmed person assignment is the **sidecar MWG region**, not the catalog — so even a reader that ignores the snapshot recovers people from sidecars. `Catalog.schemaVersion` bumps **7 → 8**.

## 4. Clustering — the pure, thoroughly-tested heart

### 4.1 `FaceClusterer` (Core, PURE — no Vision dependency)

`Sources/OpenPhotoCore/Faces/FaceClusterer.swift`. Fully unit-testable on synthetic vectors (no Vision, no Core ML, no real faces):

```swift
public enum FaceClusterer {
    /// Greedy agglomerative grouping of face vectors by cosine distance.
    /// `threshold` is the maximum cosine distance (1 − cosine similarity) for two faces to be in the
    /// same cluster. Returns clusters of face ids, each cluster ordered by input order, clusters
    /// ordered largest-first. Singletons are returned as 1-element clusters.
    public static func cluster(_ items: [(id: Int64, vector: [Float])],
                               threshold: Double) -> [[Int64]]
}
```

**Distance: cosine (1 − cosine similarity).** Justification: `VNGenerateImageFeaturePrint` vectors are not L2-normalized and their **magnitudes vary**, so raw L2 (Euclidean) distance conflates "different content" with "different brightness/scale". Cosine compares **direction** only, which is the stable signal for visual similarity here; it matches the dot-product machinery already in 4.2 and gives a threshold (~0.3–0.5 distance) that's interpretable and tunable. We L2-normalize each vector once up front, then cosine distance = `1 − dot`.

**Algorithm (greedy single-link agglomerative):** normalize all vectors; walk items in input order; for each item, find the existing cluster whose nearest member is within `threshold` and join it (single-link), else start a new cluster. O(n·k) for n faces, k clusters — fine for the unassigned set (typically thousands, not the whole library, since confirmed faces are excluded). Deterministic ⇒ directly testable.

> **Why not full hierarchical / DBSCAN?** Greedy single-link with a fixed threshold is deterministic, dependency-free, and good enough to *suggest* clusters a human confirms — the human is the arbiter, so we optimize for predictability and testability over clustering sophistication. A better algorithm is a localized future change behind the same signature.

### 4.2 Representative test (illustrative)

```swift
@Test func clustersThreeClearGroups() {
    // Three well-separated directions in 3-D; threshold splits them cleanly.
    let items: [(id: Int64, vector: [Float])] = [
        (1, [1, 0, 0]), (2, [0.99, 0.01, 0]),     // group A (≈ +x)
        (3, [0, 1, 0]), (4, [0.02, 0.98, 0]),     // group B (≈ +y)
        (5, [0, 0, 1]),                           // group C (≈ +z), singleton
    ]
    let clusters = FaceClusterer.cluster(items, threshold: 0.2)
    let sets = Set(clusters.map { Set($0) })
    #expect(sets == Set([Set([1, 2]), Set([3, 4]), Set([5])]))
}

@Test func mergesAtLooseThresholdSplitsAtTight() {
    let items: [(id: Int64, vector: [Float])] = [(1, [1, 0]), (2, [0.7, 0.7]), (3, [0, 1])]
    #expect(FaceClusterer.cluster(items, threshold: 0.05).count == 3)  // tight → 3 singletons
    #expect(FaceClusterer.cluster(items, threshold: 0.9).count == 1)   // loose → all one cluster
}
```

### 4.3 Clustering against confirmed anchors (App-side service)

The **suggested clusters** shown in the People view are computed from `unassignedFacesWithEmbeddings()` only (confirmed faces are already assigned). A lightweight nicety the service does (not the pure clusterer): after the unassigned set is clustered, each suggested cluster can be *attracted to* an existing person if its members are within `threshold` of that person's confirmed faces' mean vector — surfacing "is this also Alice?" suggestions. This is heuristic and **never auto-assigns**; the human confirms. (Optional polish; the committed path is suggest-unassigned-clusters → name them.)

### 4.4 Why brute-force is fine

Clustering is over the **unassigned `auto`** set, on demand (when the People view opens), off the main actor. Even a large library has at most a few faces per photo, and confirmed faces drop out — so the working set is bounded and shrinks as the user names people. No ANN/vector index (YAGNI; noted, not built).

## 5. Clustering service + the People view (App)

### 5.1 `PeopleService` (App, off-main)

`AppState` gains a small clustering service path (mirrors `runSearch`'s structure): when the People view opens (or after a faces-stage drain), it computes suggested clusters off the main actor and publishes them:

```swift
// AppState
var people: [PersonRow] = []                 // named people cards
var suggestedClusters: [FaceCluster] = []    // unnamed clusters (id lists + a representative face)
var facesLoading = false
func loadPeople()                            // people() + cluster(unassignedFacesWithEmbeddings(), threshold:)
```

`FaceCluster` is `(faceIDs: [Int64], representativeFaceID: Int64, count: Int)`. The threshold is a tuned constant (with room for an adjustable control later — the master spec mentions "an adjustable threshold"; v1 ships a sensible default, the slider is optional polish). Computed in a `Task.detached(.userInitiated)`, published on the main actor.

### 5.2 Face thumbnails

A face cell shows the **cropped face region** over the photo's thumbnail. We reuse `ThumbnailImage`/`ThumbnailStore` to load the asset thumbnail, then crop to the face's `rect` (the catalog Vision rect → view rect). No new thumbnail cache — the crop is a view-time transform over the existing per-asset thumbnail. Confidence is shown as a small badge/overlay on the cell.

### 5.3 `PeopleView` (App, build-verified + manual)

- **`SidebarItem.people`** (Library group, between `folders` and `search` or after `search` — placed in the `CaseIterable` enum with a `label "People"` + `symbol "person.2"` / `"person.crop.circle"`), picked up by the existing `RootView.detail` switch (`case .people: PeopleView(state:)`) and the `ForEach(SidebarItem.allCases)` sidebar.
- **Layout:** a grid of **named-person cards** (representative face crop + name + count), then a **"Suggested"** section of unnamed clusters (each a representative face crop + face count + a **name text field** inline). Tapping a person (or a cluster) opens **that person's faces/photos grid** (reuse the `MediaTile` grid from `SearchView`/timeline, over `faces(forPerson:)` → `items(forHashes:)`, or a face-cell grid for management).
- **Name a cluster:** the inline text field → `createPerson(name:)` + `assignFaces(clusterIDs, to:)` + **write MWG regions** to each affected asset's sidecar. The cluster moves out of "Suggested" into a named card.
- **Merge:** select two people (or a person-detail "Merge into…" menu) → `mergePerson(_:into:)` + rewrite the moved faces' sidecar regions with the surviving name.
- **Split:** in a person's face grid, select faces → "Split to new person" (`createPerson` + `reassignFace(to:newID)` each, rewrite regions) or "Remove from person" (`reassignFace(to: nil)` + delete those regions from the sidecars).
- **Reassign a face:** a per-face context menu "Move to…" listing people (+ "New person…" / "Not a person") → `reassignFace(_:to:)` + sidecar region update.
- **Per-face confidence:** shown on each face cell (a small percent/bar), from `faces.confidence`.

Every catalog write that changes a *confirmed* assignment is paired in the App with the corresponding sidecar MWG-region write/delete, so the durable record stays authoritative. These sidecar writes go through `SidecarStore` (atomic temp→fsync→rename) and reuse the existing per-asset relPath resolution.

## 7. Pipeline integration — the third stage

The 4.2 runner already drains a registry off-main with combined progress, reachable-URL resolution, and retry-caps. 4.3 adds **one element** to the registry:

```swift
private let derivationStages: [any DerivationStage] =
    [OCRDerivationStage(), EmbedStage(), FaceDerivationStage()]
```

- `FaceDerivationStage.isAvailable == true` (Vision is built-in), so it always runs — no graceful-absence path like the embed model.
- The combined "Analyzing N/M…" progress line now sums OCR + Embed + Faces; unchanged code (it already sums `derivationProgress(stage:)` over available stages).
- Resumable/retry-capped via `derivation_jobs` (`stage = "faces"`); a drive-only asset whose drive is unplugged is skipped and retried on connect — identical to the other stages.
- After a faces-stage drain, the People view's suggested clusters are stale → mark a `facesDirty` flag so `loadPeople()` recomputes (mirrors `semanticIndexDirty`).

## 8. Data flow

```
library open / scan / ingest → derivation runner drains [OCR, Embed, Faces]
   Faces: detect human faces (Vision) → crop each → VNGenerateImageFeaturePrint → faces rows
          (rect + Float16 embedding + confidence, personID NULL, source "auto"), off-main, resumable
   → facesDirty = true

open People view →
   people()                          → named cards (representative crop + count)
   cluster(unassignedFacesWithEmbeddings(), threshold) → "Suggested" unnamed clusters

name a cluster "Alice" →
   createPerson("Alice") → assignFaces(ids, to: alice)   [catalog: personID set, source "confirmed"]
   + SidecarStore.write MWG Regions(Name="Alice", Area=face rect)   [durable, portable]

merge / split / reassign → the analogous catalog op + sidecar region rewrite/delete

Search: filters.person = alice → structuredFilter joins faces ON personID = alice → MediaTile grid

ingest / catalog rebuild → re-detect (machine) + read sidecar MWG regions back as confirmed anchors
   → confirmed faces + people reconstituted; the machine's re-detect never disturbs them
```

## 9. Error handling / edge cases

| Case | Behavior |
|---|---|
| Photo unreadable / corrupt | `FaceStage.detect` returns nil → `markDerivationFailed` (+attempts), capped at 3 → never loops. Same as OCR/Embed. |
| Photo decodes, no faces | `detect` returns `[]` → success ("analyzed, no faces"); a `"done"` job is written so it isn't re-tried. |
| Asset unreachable (drive-only, drive off) | Skipped this pass, **not** failed; retried when reachable (runner re-poked on connect). |
| Re-detection over a confirmed face | `replaceFaces` deletes only `source='auto'` rows; confirmed rows + their MWG regions are untouched. Re-detected auto twin overlapping a confirmed rect (IoU) is dropped. |
| `deletePerson` | Its faces revert to `personID=NULL, source='auto'`; the App deletes those faces' regions from the sidecars. The faces re-enter the unassigned/clusterable pool. |
| `reassignFace(to: nil)` | Same revert for one face + remove its sidecar region. |
| `mergePerson(src, into: dst)` | `src` faces → `dst` (confirmed), `src` person deleted; sidecar regions rewritten with `dst.name`. Idempotent on the catalog side. |
| Sidecar write fails (read-only/missing folder) | The catalog op still succeeds (mirror), but the durable record didn't land → surfaced as a best-effort failure; on next sidecar read the mirror is reconciled. (Sidecar writes are atomic; a partial write can't corrupt the existing sidecar.) |
| Two people share a `Name` across photos | The same `Name` maps to one `people` row on ingest (matched by name); regions with that name reconstitute under it. |
| Feature-print dim varies across OS versions | `faces.dim` is stored per-row; the clusterer filters to a consistent dim (drops mismatched rows from a group, like `SemanticIndex` does), so a mixed set never crashes the math. |
