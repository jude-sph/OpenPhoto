# Phase 4.1 — Derivation pipeline foundation (+ OCR) (design)

**Date:** 2026-06-10
**Branch:** `phase4.1-derivation-pipeline` (off `main`)
**Status:** Approved
**Phase:** 4 (Intelligence) — first slice. Builds on Phase 1 (`Scanner`, `MetadataExtractor`, `ThumbnailStore`, `Catalog`/GRDB) and the existing reachable-file resolution (`AppState.goodCopyURL`).

> The spine every later Phase-4 stage plugs into: a resumable, per-asset background **derivation pipeline**. This slice builds the spine and proves it end-to-end with the first ML stage — **OCR** (on-device Vision text recognition → FTS5). Faces, semantic embeddings, and reverse-geocoding are later slices that each add one `DerivationStage` + its table.

---

## 1. Goal

When an asset is in the library, derive machine-only intelligence from it **automatically, in the background**, as **resumable per-asset jobs** — so derived data is computed once, **outlives eviction** (kept by content hash, survives the local file being released), and a re-scan never re-does completed work. Phase 1 already derives metadata + thumbnails *synchronously at scan*; this slice adds the *eager background* pipeline for the heavier stages, starting with OCR.

Sovereignty holds: all derived data is **machine-derived → the rebuildable catalog only** (no sidecars, no on-disk format change). Fully on-device/offline. UX: automatic, low-priority, a small sidebar progress line, no settings (chosen in brainstorming).

### Non-goals
- Only one stage here (**OCR**). Faces / semantic embeddings / reverse-geocode are later slices — each is just a new `DerivationStage` + table on this spine.
- No search *UI* yet (the unified Search box is slice 4.2); this slice only stores OCR text + a Core `searchOCR` query that 4.2 will consume.
- No settings/controls, no pause button, no power/idle gating (YAGNI for v1).

---

## 2. Schema (catalog migration v5 — rebuildable, machine-derived)

- **`derivation_jobs(hash TEXT, stage TEXT, status TEXT, attempts INTEGER, updatedAtMs INTEGER, PRIMARY KEY(hash, stage))`** — per-asset per-stage completion. `status ∈ {"done","failed"}`; **the absence of a `done` row = pending**. This drives resumability, the retry cap, and progress counts. A `stage` is a short string (`"ocr"`, later `"faces"`/`"embed"`/`"geocode"`).
- **`ocr`** — an **FTS5** virtual table `(hash UNINDEXED, text)` for full-text search over recognized text. (`hash` is `UNINDEXED` so it's stored/retrievable but not tokenized; `text` is the searchable column.)

Both are rebuildable caches: dropping them on a future migration is safe (the pipeline re-derives). No `instances`/`assets`/manifest change.

---

## 3. Core components (TDD)

**Catalog queries (`Catalog+Derivation.swift`):**
- `pendingDerivation(stage:limit:) -> [String]` — asset hashes of an **eligible kind for the stage** (OCR → `kind = 'photo'`; videos are skipped for OCR — a later stage could OCR a keyframe) that have **no `done` `derivation_jobs` row** for `stage` **and** are not over the attempt cap (a `failed` row with `attempts >= maxAttempts` is excluded). Newest-first (`assets.takenAtMs DESC`) so recent photos analyze first. Each stage declares its eligible kinds.
- `markDerived(hash:stage:)` — upsert a `done` row.
- `markDerivationFailed(hash:stage:)` — upsert/increment a `failed` row's `attempts`.
- `derivationProgress(stage:) -> (done: Int, total: Int)` — `total` = count of eligible-kind assets for the stage (photos, for OCR), `done` = `done` rows for the stage. Drives the progress line.
- `upsertOCR(hash:text:)` — replace the `ocr` row for `hash` (delete-then-insert; FTS5 has no upsert).
- `searchOCR(_ query: String) -> [String]` — hashes whose `ocr.text` matches the FTS5 query (sanitized to a prefix/phrase query). Consumed by slice 4.2; here it just needs to return correct hashes.

**`OCRStage` (`Derivation/OCRStage.swift`, `import Vision`):**
- `static func recognizeText(in url: URL) async -> String?` — a `VNImageRequestHandler(url:)` + `VNRecognizeTextRequest` (`.accurate`, language-correction on), joining the top candidate of each observation. Returns `nil` on an unreadable image (→ the runner records a failure). Headless — no UI import — so it's testable and a future CLI can run it.

A tiny `DerivationStage` notion keeps later stages uniform: a stage is `(id: String, run: (hash, url) async -> Bool)` where `run` stores its output + returns success. For v1 the runner calls `OCRStage` directly; the abstraction is introduced properly when the second stage lands.

---

## 4. App orchestration

**`AppState.derivationRunner`** — a background `Task` (`.utility` priority) owned by `AppState`, started at **library-open** and **re-poked after each scan/ingest** (the points that add assets). Loop:
1. `let batch = catalog.pendingDerivation(stage: "ocr", limit: 16)`. If empty → the runner idles (suspends until re-poked).
2. For each hash: resolve a **reachable file URL** (a local instance, else a connected durable drive that holds the hash — the existing `goodCopyURL`-style resolution, generalized to "any reachable copy"). If unreachable → skip (don't mark failed; retried when reachable).
3. `OCRStage.recognizeText(in: url)` off-main (already async/Vision). On a result → `catalog.upsertOCR(hash, text)` + `catalog.markDerived(hash, "ocr")`. On `nil` → `catalog.markDerivationFailed(hash, "ocr")`.
4. Update the published `derivationProgress: (done, total)?`. Yield between batches so the UI stays smooth. When `pendingDerivation` is empty → set `derivationProgress = nil` (idle).

**Reachability:** the runner only processes assets whose bytes are reachable now (local or a connected drive). Drive-only assets on a disconnected drive are simply skipped and picked up on a later pass (re-poked on connect). Derived OCR text is keyed by **hash**, so once derived it persists regardless of where the bytes live or whether they're later evicted.

**Sidebar progress:** a small line at the bottom of `SidebarView` (next to the existing `scanProgress` area) — `"Analyzing N / M…"` bound to `derivationProgress`, shown only while non-nil.

---

## 5. Data flow

library open → `scanAll` catalogs assets (metadata, Phase 1) → runner poked → drains OCR jobs in the background (`.utility`, batched) → `ocr` text + a `done` job row written per asset → progress line counts up → idles when none pending. Relaunch resumes from `derivation_jobs` (pending = assets without a `done` row). A new import/scan re-pokes the runner; already-`done` assets are skipped.

---

## 6. Error handling / edge cases

| Case | Behavior |
|---|---|
| Unreadable/corrupt image (Vision fails) | `markDerivationFailed` (+attempts); capped at `maxAttempts` (e.g. 3) → never retried again, never loops. |
| Asset unreachable (drive-only, drive off) | Skipped this pass, **not** marked failed; retried when its drive connects (runner re-poked on connect). |
| App quit mid-run | The `Task` is cancelled; completed jobs are persisted; the rest are still "pending" → resumed next launch. |
| Re-scan / re-import | Re-pokes the runner; `done` assets are skipped (idempotent). |
| Huge library backfill | Low-priority, batched, yields — UI stays responsive; runs until idle (minutes-to-overnight per the design's estimate). |
| A photo with no text | OCR returns empty string → stored as empty, marked `done` (not failed) — it's correctly "analyzed, no text". |

## 7. Testing

**Core (TDD, temp catalog + generated fixtures, never `~/Pictures`):**
1. `derivation_jobs` lifecycle — `pendingDerivation` returns assets with no `done` row; after `markDerived` they drop out; `markDerivationFailed` increments attempts and excludes once over the cap; `derivationProgress` reports `(done, total)`.
2. `upsertOCR` + `searchOCR` — insert OCR text for two hashes, an FTS5 `MATCH` returns the right hash(es); `upsertOCR` replaces (no duplicate rows); a query matching nothing returns `[]`.
3. `OCRStage` (lighter) — render known text (e.g. "OPENPHOTO") into an image with Core Graphics, assert `recognizeText` returns a string containing it (Vision can vary, so this is the one soft assertion; the storage/jobs logic above carries the hard ones).

**App:** build-verified (0 warnings) + manual — open a library, watch the sidebar "Analyzing N / M…" count up then disappear; relaunch resumes without redoing; (with a future search UI absent, verify via a temporary debug or by inspecting the catalog that `ocr` rows exist).

3. **App** — `AppState.derivationRunner` (background `.utility` Task: poke at library-open + after scans/ingest; batched pending→resolve→OCR→store→progress; reachable-URL resolution) + `derivationProgress` state + the sidebar progress line. Build-verified + manual; rebuild bundle.
4. **Docs** — `docs/format/catalog-schema.md` (the new `derivation_jobs` + `ocr` tables, marked rebuildable); master spec §10 (Phase 4 slice 4.1 done, the agreed 5-slice decomposition + order recorded). No `vault-format-v1` change (catalog-only).

No on-disk vault format change. `Scanner`/`MetadataExtractor`/`ThumbnailStore`/`SyncEngine`/manifests untouched; this adds a parallel background derivation lane keyed by hash.
