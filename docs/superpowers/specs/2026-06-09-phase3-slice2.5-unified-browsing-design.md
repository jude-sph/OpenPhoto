# Phase 3 Slice 2.5 — Unified / Drive-Only Browsing: Design Spec

**Status:** Approved (2026-06-09). Detailed design for Phase 3 **Slice 2.5**, inserted before the destructive slices. Part of the roadmap in `docs/superpowers/specs/2026-06-09-phase3-drives-design.md` (revised order: 1 ✅ · 2 ✅ · **2.5** · 3 deletion-propagation · 4 evict/rehydrate · 5 clone/migrate).

**One-line goal:** make assets that live *only* on a canonical drive appear in the timeline and folders as thumbnail-only entries — so adopting a drive file makes it visible, and (later) evicted photos stay visible.

---

## 1. Why this slice exists

OpenPhoto's browse (timeline, folder tree) is built from the catalog's **local `instances`** — an asset with no file on the Mac can't appear. That's why **Adopt** (Slice 2) records a drive file but it never shows in the library. The same gap will bite **eviction** (Slice 4): the moment you delete a local copy, the photo must stay visible or it vanishes.

This slice is the **non-destructive foundation** that makes drive-resident photos first-class library citizens: browsable (thumbnail + metadata) even with the drive in a drawer, opened full-res only when the drive is connected. It is the "browse everything, even what's in a drawer" half of the unified-library vision.

## 2. Invariants (unchanged)

- Non-destructive: this slice only **reads** drives and **adds** rows/thumbnails to the rebuildable catalog. Nothing deletes, moves, or overwrites an original.
- The catalog is rebuildable; the drive's `manifest.jsonl` + sidecars remain the portable truth.
- **View-only** for drive-only assets: they can be browsed and opened (when the drive is connected) but **not edited** (tags/rating/caption) until they're local again. Editing a drive-resident asset means writing a sidecar onto a passive drive + queueing while unplugged — that's eviction/rehydrate-era work, explicitly out of scope here.

## 3. The model

An asset is shown in browse from its **best available location**:
- a **local instance** exists → show it as today (full-res available); or
- no local instance, but the asset is present on a canonical drive (in `vault_presence`) and has an `AssetRecord` → show it **thumbnail-only**, badged "on drive."

Because the drive-only branch requires *no local instance*, the two branches are mutually exclusive → **each asset shows once** (local wins). No dedup of the existing per-instance query is needed.

A drive-only asset needs three things to be browsable, all cached locally so it survives the drive being unplugged:
1. an **`AssetRecord`** (metadata: `takenAtMs` for sorting, kind, dimensions, …);
2. a **thumbnail** in the local thumbnail cache (keyed by hash);
3. a **presence row** carrying its path + size (so it can be placed in a folder and read off the drive when connected).

## 4. Catalog changes

### 4.1 `vault_presence` gains path/size (migration `v3`)
`vault_presence` is today `(vaultID, hash)`. It becomes the catalog's cached mirror of a drive's contents:

```
vault_presence(vaultID TEXT, hash TEXT, relPath TEXT, dirPath TEXT, size INTEGER,
               driveRelPath TEXT, PRIMARY KEY(vaultID, hash))   -- index on hash
```

- `driveRelPath` — the asset's path **on the drive** (e.g. `Pictures/rome2022/IMG.heic`); used to read full-res off the drive.
- `relPath` / `dirPath` — the **Mac-aligned** path used for folder grouping/display: `driveRelPath` with the leading source-vault basename stripped when it matches a configured Mac vault (`Pictures/rome2022/IMG.heic` → `rome2022/IMG.heic`, dir `rome2022`). If no basename matches (e.g. a stray `Extra/…` dir on the drive), `relPath == driveRelPath`.
- `size` — bytes (from the manifest).

`vault_presence` is rebuildable, so migration `v3` **drops and recreates** it with the new columns; it repopulates on the next scan/sync.

### 4.2 Population — `replaceVaultPresence` takes entries
Replace `replaceVaultPresence(vaultID:hashes:[String])` with `replaceVaultPresence(vaultID:entries:[VaultPresenceEntry])` where `VaultPresenceEntry = (hash, relPath, dirPath, size, driveRelPath)`. The two existing callers already read the drive manifest, so they build entries cheaply (applying the basename-strip via a helper that's handed the source-vault basenames):
- Slice 1 sync (`AppState.refreshCanonicalPresence`) — from the drive manifest after a sync.
- Slice 2 drift (`AppState.driftScan` / `autoScanConnectedDrives`) — from the present-and-verified manifest entries.

### 4.3 Browse queries gain a drive-only branch
`timelineItems()`, `items(inDir:)`, `folderCounts()` become a `UNION` of the existing local-instance query and a drive-only query:

```sql
-- drive-only branch (one row per asset with no local instance)
SELECT a.<cols>, vp.vaultID, vp.relPath, vp.dirPath, vp.size, vp.driveRelPath
FROM assets a
JOIN vault_presence vp ON vp.hash = a.hash
WHERE a.isLivePairedVideo = 0
  AND NOT EXISTS (SELECT 1 FROM instances i WHERE i.hash = a.hash)
  AND vp.rowid = (SELECT MIN(rowid) FROM vault_presence v2 WHERE v2.hash = a.hash)  -- one drive if on several
```

`folderCounts`/`items(inDir:)` group the drive-only branch by `vp.dirPath` (Mac-aligned), so a drive-only photo lands in the *same* folder node as its local siblings.

### 4.4 `TimelineItem` gains `driveRelPath: String?`
`nil` for local items; non-`nil` for drive-only ones. It's how the rest of the app knows an item is drive-only (thumbnail-only) and how it locates the file on the drive. The local branch selects `NULL AS driveRelPath`; the drive-only branch selects `vp.driveRelPath`.

## 5. Ingest-on-adopt

`DriftReconciler.adopt` (Slice 2) currently hashes the file + writes the manifest line + presence. A new **`CatalogIngest`** step makes the adopted asset browsable:
1. extract metadata (`MetadataExtractor.extract`) → build an `AssetRecord`;
2. generate + cache a thumbnail from the **drive file** (`ThumbnailStore`);
3. `catalog.upsert(assets:)`;
4. write the `vault_presence` row (hash, mapped paths, size, driveRelPath).

This runs for single Adopt and Adopt-all. (For future **eviction**, the asset already has an `AssetRecord` + thumbnail from when it was local, so eviction needs only the presence row + instance removal — no new ingest. The browse-query change in §4.3 serves both.)

**Live Photos (best-effort):** if both halves of a drive-only Live Photo are adopted, pair them via the existing `LivePhotoPairer`; otherwise they show as standalone photo + video. Full pairing of partially-adopted pairs is out of scope.

## 6. App / UI

- **Cells:** a drive-only cell (item.driveRelPath != nil) shows a small "on drive" glyph and is rendered thumbnail-only. Thumbnails come from the local cache (so they show unplugged).
- **`absoluteURL(for:)`:** for a drive-only item, resolve `driveVault(forVaultID).rootURL + item.driveRelPath`; for local items, unchanged. Returns `nil` (and the viewer shows "Plug in *<drive>* to view full-res") when the drive isn't connected.
- **Viewer:** opening a drive-only asset shows the thumbnail and, if the drive is connected, the full-res; if not, the plug-in prompt. No zoom/edit affordances beyond viewing.
- **Editing guard:** tag/rating/caption controls are disabled for drive-only items (view-only, §2).
- **Inspector:** the existing Locations panel already shows the drive as a confirmed location; the "Not on the canonical drive" flag (just added) naturally reads correctly for these.

## 7. Where the code lives

- **Modify** `Sources/OpenPhotoCore/Catalog/Catalog.swift` — migration `v3`; `replaceVaultPresence(vaultID:entries:)`; `VaultPresenceEntry`.
- **Modify** `Sources/OpenPhotoCore/Catalog/Queries.swift` — `UNION` drive-only branch in the three browse queries; a `drivePresence(forHash:)` lookup if needed.
- **Modify** `Sources/OpenPhotoCore/Catalog/Records.swift` (or wherever `TimelineItem` is defined) — add `driveRelPath: String?`.
- **Create** `Sources/OpenPhotoCore/Sync/CatalogIngest.swift` — ingest one drive file (metadata + thumbnail + asset + presence); a `driveToMacRelPath(_:sourceBasenames:)` mapping helper.
- **Modify** `Sources/OpenPhotoCore/Sync/DriftReconciler.swift` — `adopt` calls `CatalogIngest`; presence population carries paths.
- **Modify** `Sources/OpenPhotoApp/AppState.swift` — pass source basenames into presence/ingest; `absoluteURL(for:)` drive-only handling; thread driveRelPath.
- **Modify** the cell views + viewer + inspector edit guards for thumbnail-only items.

## 8. Testing (TDD, Swift Testing, generated fixtures only)

Core:
- migration `v3` recreates `vault_presence`; `replaceVaultPresence(entries:)` round-trips path/size.
- `driveToMacRelPath` strips a matching source basename (`Pictures/rome2022/x` → `rome2022/x`) and leaves a non-matching prefix intact.
- `timelineItems()` includes a drive-only asset (present + AssetRecord, no instance); an asset with **both** a local instance and drive presence appears **once** (local, `driveRelPath == nil`); a drive-only asset on **two** drives appears once.
- `items(inDir:)`/`folderCounts()` place a drive-only asset in its Mac-aligned folder.
- `CatalogIngest` (adopt): adopting a stray drive JPEG creates the AssetRecord + a cached thumbnail + presence row → it then appears in `timelineItems()`. Browsing still works after the drive path is made unreachable (thumbnail served from cache).

App: build-verified + manual (drive-only badge, full-res-needs-drive prompt, edit controls disabled).

## 9. Out of scope (later slices / deferred)

Editing drive-only assets; the "Sources" filter; auto-cataloguing *every* stray drive file (only adopted ones ingest; the browse change covers evicted ones for free); full Live-Photo pairing of partially-adopted pairs; rehydrate (Slice 4); deletion/eviction.

## 10. Milestones

- **M1 — Catalog**: migration `v3`, `VaultPresenceEntry` + `replaceVaultPresence(entries:)`, `driveToMacRelPath` helper, update Slice 1/2 callers.
- **M2 — Browse**: `TimelineItem.driveRelPath`; drive-only `UNION` in the three queries; dedup correctness.
- **M3 — Ingest-on-adopt**: `CatalogIngest` (metadata + thumbnail + asset + presence); wire into `adopt`/`adoptAll`.
- **M4 — App**: `absoluteURL(for:)` drive-only; thumbnail-only cell badge; viewer full-res-needs-drive prompt; edit guards.
