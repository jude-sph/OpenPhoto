# Phase 4.2 — Semantic image embeddings + unified Search (design)

**Date:** 2026-06-11
**Branch:** `phase4.2-semantic-search` (off `main`, after Phase 4.1 merged at `6bb7074`)
**Status:** Approved
**Phase:** 4 (Intelligence) — second slice. Builds on Slice 4.1 (the derivation-pipeline spine: `derivation_jobs`, the `AppState` background runner, the `ocr` FTS5 table + `searchOCR`) and Phase 1 (`Catalog`/GRDB, the `TimelineItem` timeline union, `MediaTile`/`ThumbnailImage`).

> The slice that makes the library *searchable*: one free-text box + click-to-narrow filters, backed by three lanes — **semantic** (on-device MobileCLIP image/text), **text-in-image** (OCR FTS, already built in 4.1), and **your own labels** (tag/caption). It adds the second derivation stage (image embeddings), so this is also where the generic `DerivationStage` abstraction — deferred in 4.1 — lands.

---

## 1. Goal

Let the user **type what they're looking for and find it**, entirely on-device/offline. Three things combine in one box:

1. **Semantic** — type `sunset`, `red car`, `birthday cake` → photos that *look like* that, via a CLIP-class image/text model (MobileCLIP) whose image embeddings are derived per asset and whose text encoder embeds the query at search time. Cosine similarity in a shared space.
2. **Text-in-image** — type a word visible on a sign/receipt/whiteboard → matches via the existing `ocr` FTS5 table (`searchOCR`, built in 4.1).
3. **Your labels** — matches against your tags (`tagsJSON`) and caption.

On top, the user **narrows** with click-controls (no typed query language in v1): date range, camera, rating, favorite, video-only, and tag. Also folds in the [Phase 4] backlog item *"tag search in timeline & folder views."*

Sovereignty holds: embeddings are **machine-derived → the rebuildable catalog only** (a new `embeddings` table; no sidecars, no vault-format change). Fully on-device. The model runs via Core ML on the ANE/GPU automatically — no hand-written Metal.

### Decisions locked in brainstorming
- **Inputs:** free-text box (drives semantic + text lanes) + structured filters as **UI controls** (chips/menus/toggles). **No** `key:value` / natural-language parsing in v1 (the LLM rewriter is Phase 5; the filter model is built so token-parsing can layer on later without rework).
- **Model:** **MobileCLIP** (S2 or the v2 variant with a clean Core ML export — the exact variant is the spike's call). Bundled in the app; ~40–150 MB. The pipeline is **model-agnostic** (an `embeddings.model` column), so a later swap just re-derives.
- **Ranking:** a single ordered list — **exact textual matches** (your tags/caption/visible text) first, then **semantic** by descending similarity; ties by recency. Empty box → the structured filter set, newest-first.
- **Placement:** Search is its own **sidebar entry**, reusing the existing `MediaTile` results grid.
- **Tag search** reuses the unified-search tag filter (not a separate per-view engine); the browse surfaces get an entry point into it.

### Non-goals (later slices / phases)
- **Person & place filters** → 4.3 (Faces/People) and 4.4 (geocode/Map). The `SearchFilters` struct reserves space for them but v1 does not implement them.
- **Typed query language / NL parsing** → Phase 5 (LLM rewriter), slots in front without changing indexes.
- **ANN / vector index** — brute-force cosine is <100 ms at this scale (see §4); a real index is YAGNI for v1.
- **Image-to-image "find similar" / near-dup** — that's `VNGenerateImageFeaturePrint` / perceptual-hash territory (Phase 5); this slice is text→image + filters.

## 2. The embedding model + on-device integration

### 2.1 Spike (first task — lean, de-risking)
The one genuinely non-trivial piece is the **text encoder's tokenizer** and the no-Xcode conversion/compile path. A focused spike (throwaway code under `docs/spikes/` + a tiny executable target, mirroring the existing `ICCSpike` precedent) must prove, before anything is built on top:

1. **Variant selection** — pick the MobileCLIP variant trading size vs retrieval quality (aim small); record file, embedding `dim`, and image-embed latency on Apple Silicon.
2. **Conversion** — the one-time dev-side `coremltools` conversion (Python, dev-time only — *never* at runtime) that produces the bundled `.mlpackage`(s) for the image and text encoders. How the model file reaches the build (committed via Git LFS vs a documented setup-script download+convert) is decided here; the **app only ever consumes** the bundled artifact.
3. **No-Xcode runtime path** — `MLModel.compileModel(at:)` compiles the bundled `.mlpackage` to a cached `.mlmodelc` on first launch; `MLModelConfiguration.computeUnits = .all` (ANE/GPU). Prove image embed + **text embed** work from Swift in the SPM-built bundle.
4. **Tokenizer** — port the CLIP-class BPE tokenizer (vocab + merges) to Swift (lean on HF `swift-transformers` / existing Swift CLIP ports). Prove a query string → token ids → text embedding.
5. **Retrieval sanity** — on a handful of **generated** test images (distinct, unambiguous content; never `~/Pictures`), the right text query ranks the right image highest.

Output: a short spike doc recording the variant, dims, latency, the tokenizer approach, and the conversion/bundling decision — feeding the real `EmbedStage`.

### 2.2 `EmbedStage` (Core, headless — mirrors `OCRStage`)
`Sources/OpenPhotoCore/Derivation/EmbedStage.swift` (`import CoreML`):
- `func embedImage(at url: URL) -> [Float]?` — load → Core ML image encoder → **L2-normalized** vector; `nil` only on a thrown/failed inference (so the runner records a failure, retry-capped — exactly like OCR).
- `func embedText(_ s: String) -> [Float]?` — tokenize → Core ML text encoder → L2-normalized vector (used at *query* time, not by the background runner).
- Holds the lazily-compiled `MLModel`s (compile-once, cache the `.mlmodelc`). Headless — no UI import — so a future `openphoto` CLI can run it.
- `modelID: String` — the bundled model identifier (e.g. `"mobileclip_s2"`), written into each `embeddings.model`.

### 2.3 Storage — catalog migration **v6** (machine-derived / rebuildable)
A new table:

```
embeddings(hash TEXT PRIMARY KEY,
           model TEXT NOT NULL,      -- e.g. "mobileclip_s2"; lets a model swap invalidate
           dim INTEGER NOT NULL,     -- variant-dependent (~512)
           vector BLOB NOT NULL)     -- dim × Float16, little-endian, L2-normalized
```

Vectors are **L2-normalized** and stored as **Float16** (halves disk + snapshot footprint; ~1 KB per 512-dim). Because they're normalized, **cosine == dot product** — the query reduces to a matrix·vector dot.

Rebuildable cache: dropping it is safe (the pipeline re-derives). Rides along in the existing per-drive `catalog-snapshot/catalog.sqlite` (VACUUM INTO) like every other catalog table. **No `instances`/`assets`/manifest change.** `Catalog.schemaVersion` bumps 5 → 6.

**Model-swap reconcile (forward-compat, trivial in v1's single-model world):** at library open, `reconcileEmbeddingModel(current:)` deletes `embeddings` rows whose `model != current` **and** their `embed` `derivation_jobs` rows, so the pipeline re-derives them. One model in v1 → a no-op, but the mechanism exists.

## 3. Pipeline generalization — the `DerivationStage` abstraction

4.1 deferred this ("introduced properly when the second stage lands"). Now it lands.

```swift
protocol DerivationStage: Sendable {
    var id: String { get }            // "ocr", "embed" — the derivation_jobs `stage`
    var eligibleKind: String { get }  // "photo" for both v1 stages
    /// Compute + STORE the derived output for this asset; return success.
    func run(hash: String, url: URL, catalog: Catalog) async -> Bool
}
```

- `OCRStage` conforms: `run` calls `recognizeText` → `upsertOCR` + `true`; `nil` → `false`. (Empty string is still success — "analyzed, no text".)
- `EmbedStage` conforms: `run` calls `embedImage` → `upsertEmbedding(hash, model, dim, vector)` + `true`; `nil` → `false`.
- `Catalog.eligibleKind(forStage:)` gains `"embed" → "photo"`.

**Runner (`AppState`):** the 4.1 runner stops hard-coding OCR. It iterates a **stage registry** `[OCRStage, EmbedStage]`; for each stage it drains its whole pending set (`pendingDerivation(stage:)`) over the same reachable-URL resolution, off-main, marking `done`/`failed` per asset — each stage resumes independently via its own `derivation_jobs` rows. The sidebar "Analyzing N/M…" line generalizes to the **combined** remaining work across stages (sum of per-stage `(done,total)`), still hiding when idle.

## 4. Search query engine (Core, TDD)

### 4.1 `SemanticIndex` (Core)
Holds the embeddings in memory for the cosine scan:
- `hashes: [String]`, a contiguous `[Float]` matrix (`count × dim`, row-major, fp32, L2-normalized), `dim: Int`.
- `func query(_ q: [Float], topN: Int) -> [(hash: String, score: Float)]` — dot-product `q` against every row (normalized ⇒ cosine) via Accelerate (`cblas_sgemv`/`vDSP`), partial-sort the top-N. <100 ms at 100k×512.
- Built from the `embeddings` table; **invalidated** when new embeddings land (after an embed drain) and rebuilt lazily before the next search. Memory: ~200 MB at 100k×512 fp32 — acceptable on target hardware; quantization/ANN is a future option if libraries get huge (noted, not built).
- **No hard cosine threshold in v1** — CLIP cosine magnitudes are unreliable across the modality gap, so we take **top-N** (default ~300) and let the structured-filter intersection + the result ordering handle volume. A learned/percentile cutoff is a future refinement.

### 4.2 The lanes (Core methods on `Catalog`)
- `structuredFilter(_ f: SearchFilters) -> [String]` — SQL over the timeline union honoring: `dateRange` (`takenAtMs`), `camera` (`cameraModel`), `minRating` (`rating >=`), `favoritesOnly`, `videoOnly` (`kind='video'`), `tags` (**AND** semantics — asset must carry every selected tag — via `json_each(tagsJSON)`). Returns the candidate hash set (newest-first when used directly).
- `textMatches(_ query: String) -> [String]` — `searchOCR(query)` ∪ caption/tag substring matches (SQL `LIKE`/`json_each`), de-duped, OCR-rank-ordered first.
- `distinctCameras() -> [String]`, `distinctTags() -> [String]` — populate the filter controls.
- `items(forHashes: [String], preservingOrder: Bool) -> [TimelineItem]` — materialize the final ranked hashes into renderable `TimelineItem`s (the timeline union, `WHERE hash IN (…)`), preserving the given order.

### 4.3 `SearchRanker` (Core, pure — the unit-tested heart)
`combine(structured: [String], text: [String], semantic: [(String,Float)], hasText: Bool) -> [String]`:
- `hasText == false` → return `structured` (already newest-first).
- else: `candidate = (Set(text) ∪ Set(semantic.hashes)) ∩ Set(structured)`; ordered = the `text` hashes within `candidate` (in text order) followed by the `semantic` hashes within `candidate` not already included (in score-descending order). Deterministic ⇒ directly testable without any ML.

### 4.4 Orchestration (`AppState.runSearch()`)
Off-main where it's heavy: `EmbedStage.embedText(query)` (Core ML) → `SemanticIndex.query`; `Catalog.textMatches` + `Catalog.structuredFilter`; `SearchRanker.combine`; `Catalog.items(forHashes:preservingOrder:)` → publish `searchResults: [TimelineItem]`. Debounced on the box; runs filter-only when the box is empty but filters are set.

## 5. Search UI (App, build-verified + manual)

- **`SidebarItem.search`** (Library group) → a new `case .search: SearchView(state:)` arm in `RootView.detail`; the sidebar picks it up via `CaseIterable` + a symbol/label.
- **`SearchView`:** the free-text box (debounced → `runSearch`), a compact **filter bar** — date range, camera menu (`distinctCameras`), rating control, favorite toggle, video-only toggle, and tag chips (`FlowLayoutLite`, from `distinctTags`) — bound to flat `AppState` properties (`searchQuery`, `searchFilters`), matching the existing `videoOnly`/`foldersRecursive` scalar-filter pattern. Results render in the **same `LazyVGrid` + `MediaTile`** as the timeline (selection, inspector, viewer all work for free), as a **flat relevance-ordered grid** with a result-count header (relevance order isn't time-grouped). Empty states: *"Type to search…"* (nothing entered) vs *"No matches"* (query/filters, no results).
- **Tag fold-in (browse surfaces):** the inspector's existing tag chips become **tappable** → navigate to Search with that tag filter active. Minimal, reuses the one filter engine, and satisfies *"tag search in both browse surfaces."* (A timeline/folders toolbar tag-picker is an optional extra if cheap; the inspector tap is the committed path.)

## 6. Data flow

```
library open → reconcileEmbeddingModel(current) → derivation runner drains [OCR, Embed]
   (image embeddings written to `embeddings`, keyed by hash, off-main, resumable, retry-capped)
   → SemanticIndex marked dirty

user types "dog" + clicks filters →
   embedText("dog")  ──┐
   searchOCR/caption/tag ┤→ SearchRanker.combine(structured ∩ (text ∪ semantic)) → items() → grid
   structuredFilter(filters) ┘
```

Relaunch resumes embeddings from `derivation_jobs` (pending = no `done` embed row). A new import/scan re-pokes the runner; done assets skip. Embeddings persist by **hash**, so they outlive eviction and travel on the drive snapshot.

## 7. Error handling / edge cases

| Case | Behavior |
|---|---|
| Image fails to embed (corrupt/unreadable) | `markDerivationFailed` (+attempts), capped at 3 → never loops. Same as OCR. |
| Asset unreachable (drive-only, drive off) | Skipped this pass, **not** failed; retried when reachable (runner re-poked on connect). |
| Query while embeddings still backfilling | Text + filter lanes work immediately; semantic lane covers only assets embedded so far; coverage grows as the progress line counts up. |
| Model swap (future) | `reconcileEmbeddingModel` drops old-model rows + their embed jobs → re-derived. |
| Empty box, no filters | "Type to search…" empty state (never the whole library). |
| Box empty, filters set | Filter-only mode: structured set, newest-first. |
| FTS/`LIKE` punctuation | Reuse 4.1's `searchOCR` sanitizer; caption/tag `LIKE` parameterized (no injection). |

## 8. Testing

**Core (TDD, temp catalog + generated fixtures, never `~/Pictures`):**
1. `embeddings` CRUD — `upsertEmbedding` replaces (no dup), round-trips Float16 within tolerance; `reconcileEmbeddingModel` deletes only non-matching-`model` rows + their embed jobs; `pendingDerivation(stage:"embed")` reflects them.
2. `SemanticIndex` — hand-built normalized vectors with a known nearest; `query` returns them in the right order, respects `topN`.
3. `SearchRanker.combine` — exact-text-before-semantic ordering; intersection with `structured`; empty-box passthrough; de-dup across lanes.
4. `Catalog` lanes — each structured filter (date/camera/rating/favorite/video/tag-AND) in isolation + combined; `json_each` tag filter; `items(forHashes:preservingOrder:)` preserves order; `distinctCameras`/`distinctTags`.
5. `EmbedStage` (lighter, soft) — a generated image → a finite, L2-normalized vector of the expected `dim`. (Real retrieval quality is the **spike's** job, not a unit test.)

**App:** build-verified (0 warnings) + manual — type `dog`, watch filters narrow; relaunch resumes embedding backfill without redoing; tap an inspector tag → lands in filtered Search.
