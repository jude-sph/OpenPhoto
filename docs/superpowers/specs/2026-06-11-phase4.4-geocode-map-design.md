# Phase 4.4 — Reverse-geocoding + Map (design)

**Date:** 2026-06-11
**Branch:** `phase4.4-geocode-map` (off `main`)
**Status:** Approved
**Phase:** 4 (Intelligence) — fourth slice. Builds on Slice 4.1 (the derivation-pipeline spine: the `DerivationStage` protocol + the multi-stage `AppState` runner, `derivation_jobs`, the reachable-URL resolution), Slice 4.2 (the gitignored-model discipline — a `tools/fetch-*.sh` script into a gitignored dir, graceful absence via `isAvailable`, `make-app.sh` injection), and Slice 4.3 (the `Catalog`-migration cadence, `SearchFilters`/`Catalog.structuredFilter` click-to-narrow Search, the `SidebarItem` Library group + `RootView.detail` switch).

> The slice that gives every geotagged photo a **place name** — "Taipei, Taiwan" — and a **Map** to browse the library spatially. It adds the **fourth derivation stage** (`Geocode`), which reverse-geocodes a photo's GPS to a city/region/country entirely **offline** against a **bundled GeoNames dataset** (no network, ever). Place names show in the **inspector**, fold into the unified **Search** as a place filter (alongside the 4.3 person filter), and anchor an Apple-Photos-style **MapKit** surface under the Library sidebar group: pannable/zoomable, **clustered photo thumbnails** at their locations, tapping a cluster zooms in / opens a photo grid.

---

## 1. Goal

Let the user **find and browse photos by where they were taken**, entirely on-device/offline. Three capabilities:

1. **Reverse-geocode** — every photo *with GPS* runs an offline city/region/country lookup in the background pipeline; the result becomes a `geocode` row (`city`, `region`, `country`, `countryCode`) keyed by the asset hash. Photos without GPS are simply skipped.
2. **See the place** — the **inspector** shows the resolved place name ("Taipei, Taiwan") for the selected photo; a **place** filter folds into the 4.2/4.3 unified Search ("photos in Japan" / "photos in Taipei" is one click, and composes with date/camera/person/etc.).
3. **Map** — a new **Map** entry in the Library sidebar opens an Apple-Photos-style MapKit surface: geotagged photos appear as **clustered annotations carrying a representative thumbnail**; panning/zooming re-clusters; tapping a cluster zooms to split it; a region/cluster opens a **photo grid** of those photos. Photos without GPS never appear.

Sovereignty holds cleanly: reverse-geocoding is **100% machine-derived** (a place name is a deterministic function of the photo's GPS + the dataset), so it lives in the **rebuildable catalog only** — there is **no on-disk format change** and **no sidecar write**. Dropping the `geocode` table is always safe: the pipeline re-derives every place name from the assets' stored lat/lon. The only new on-disk artifact is the **gitignored, dev-fetched GeoNames dataset**, which is bundled into the built `.app` (never into the user's library).

### Decisions locked in brainstorming

- **Offline city/region-level reverse-geocoding** from a **bundled GeoNames dataset** (~tens of MB). **NO network at runtime.** `CLGeocoder` is rejected — it is online and rate-limited. The bundled data gives "Taipei, Taiwan"-level place names (city + admin1 region + country).
- **Dataset handling mirrors the 4.2 MobileCLIP model exactly:** the dataset is **gitignored**, fetched by `tools/fetch-geonames.sh` into a gitignored dir (`.models/Resources/`, the same dir the model uses), the **build tolerates its absence** (geocode degrades gracefully → no place names if the dataset is missing, never crashes; `main` always builds with **no hard SwiftPM resource dependency**), and **`make-app.sh` injects it** into the built `.app`'s `Contents/Resources/`.
- **Reverse-geocode runs as a new background derivation stage** (`"geocode"`), like OCR/Embed/Faces — resumable, retry-capped, off-main. Eligibility: photos **with GPS only** (lat/lon non-null in `assets`).
- **Place names show in the inspector** AND **a place filter folds into the unified Search** (alongside the existing person filter).
- **Map = MapKit**, a new `SidebarItem.map` under the **Library** group, **Apple-Photos-style**: pannable/zoomable, **clustered photo thumbnails** at their locations, tapping a cluster zooms in (splits it), region/cluster → a `MediaTile` photo grid.
- Builds on the 4.1–4.3 spine: `DerivationStage` + the multi-stage runner (registry currently `[OCRDerivationStage, EmbedStage, FaceDerivationStage]` → add `GeocodeStage`), `Catalog` (GRDB), `SearchFilters`/`Catalog.structuredFilter`, `SearchView`. **Catalog is now at v8** (4.3's `faces`/`people`); the geocode migration is **v9**.

### Non-goals (later slices / phases / out of scope)

- **Street-level / POI geocoding** — city + admin1 region + country is the resolution; no street address, no points-of-interest, no neighborhoods. (A finer GeoNames extract is a future swap behind the same `ReverseGeocoder` signature.)
- **Online geocoding / `CLGeocoder`** — explicitly rejected; everything is offline against the bundled dataset.
- **Forward geocoding / a place search box that types a place name into the dataset** — Search's place filter is a *picker* over the places that actually appear in the library (`distinctPlaces()`), not a free-text geocoder. (Free-text place search could layer on later; YAGNI now.)
- **Writing place names to sidecars** — place is purely machine-derived and rebuildable; it stays catalog-only. **No on-disk format change, no sidecar write.** (Unlike 4.3's confirmed faces, there is no human decision to preserve.)
- **Trips / Memories / clustering photos into named "places I've been"** — out of scope; the Map and the place filter are the v1 spatial surfaces.
- **Map editing / dropping a manual pin to set a photo's location** — the Map is read-only browsing; it never edits an asset's GPS.

## 2. The dataset — bundled GeoNames

### 2.1 What ships

A city-level GeoNames extract, ~tens of MB, fetched at dev-setup into the gitignored model dir and injected into the `.app` by `make-app.sh`. Three tab-separated public files from `download.geonames.org/export/dump/`:

| File | Role | Columns used |
|---|---|---|
| `cities15000.txt` | The city list — every populated place ≥ 15 000 people (~25 k rows, ~6 MB). | `name` (col 2), `latitude` (5), `longitude` (6), `country code` (9), `admin1 code` (11). |
| `admin1CodesASCII.txt` | admin1 code → region name. | key `"<countryCode>.<admin1Code>"` → region name (col 2). |
| `countryInfo.txt` | country code → country name (and the ISO name). | ISO country code → country name (the `#`-comment header is skipped). |

> **`cities15000.txt` is the chosen extract** (smallest of the city tiers that still covers every notable city — the Map/inspector want recognizable place names, not hamlets). `cities5000.txt` (~12 MB, ~55 k rows) is the documented finer alternative behind the *same* loader/signature; the fetch script defaults to `cities15000` and accepts the tier as an argument. The resolution target is "nearest notable city", so 15000 is the right default.

A place resolves to **city + region + country**: the nearest city row gives the city name + its `country code` + its `admin1 code`; `admin1CodesASCII.txt` maps `"<cc>.<admin1>"` → the region name; `countryInfo.txt` maps `<cc>` → the country name. So `(lat, lon)` near Taipei → `Place(city: "Taipei", region: "Taipei", country: "Taiwan", countryCode: "TW")`.

### 2.2 License + attribution

GeoNames data is licensed **CC-BY 4.0** — free to redistribute (so bundling it in the `.app` is fine), **attribution required**. The attribution string ("Place data © GeoNames, CC BY 4.0 — https://www.geonames.org") is surfaced in an **About/credits** string in the app (a `Credits`/about line; v1 may place it in the About panel or a credits constant), and the requirement is documented in `tools/fetch-geonames.sh`'s header and in the docs task. The fetch script prints the license notice on completion.

### 2.3 Gitignore + graceful absence + injection (mirrors 4.2)

- **Gitignored:** the dataset lands in `.models/Resources/geonames/` — already covered by the existing `.models/` rule in `.gitignore` (the same dir MobileCLIP uses). Nothing is committed.
- **`tools/fetch-geonames.sh`** downloads the three files into `.models/Resources/geonames/` (curl, `--retry`, public URLs). Run once at dev setup. Mirrors `tools/fetch-mobileclip.sh`'s shape and the gitignored-dest convention.
- **Graceful absence:** the build has **no SwiftPM resource dependency** on the dataset (`main` builds on a fresh checkout with no `.models/`). When the dataset is absent, the `ReverseGeocoder` loader returns `nil` → `GeocodeStage.isAvailable == false` → the runner **skips** the geocode stage entirely (jobs stay pending, nothing marked failed), exactly like `EmbedStage` when the model is missing. No place names appear; nothing crashes.
- **`make-app.sh` injection:** the existing model-injection block in `scripts/make-app.sh` (which copies from `.models/Resources/` into `$APP/Contents/Resources/`) is extended to also copy `geonames/` (best-effort, skipped silently if absent, with a `note:` like the model path). At runtime the `ReverseGeocoder` loads the dataset from `Bundle.main.resourceURL/geonames/` (the same `Bundle.main.resourceURL` default `EmbedStage` uses).

## 3. The `ReverseGeocoder` — the pure, thoroughly-tested heart

`Sources/OpenPhotoCore/Geocode/ReverseGeocoder.swift`. A pure value type **over the loaded city data** — no network, no `CLGeocoder`, no Core Location lookups. Fully unit-testable on a tiny **synthetic in-memory** city list (no real dataset needed for tests).

### 3.1 Shape

```swift
public struct Place: Sendable, Equatable {
    public let city: String
    public let region: String        // admin1 name (e.g. "Taipei", "California"); may be ""
    public let country: String       // e.g. "Taiwan", "United States"
    public let countryCode: String   // ISO-2 (e.g. "TW", "US")
    public init(city: String, region: String, country: String, countryCode: String)
}

public struct ReverseGeocoder: Sendable {
    /// A single city entry used to build the lookup. The loader produces these from the GeoNames
    /// files; tests hand-build a handful directly (no real dataset).
    public struct City: Sendable {
        public let name: String
        public let lat: Double
        public let lon: Double
        public let countryCode: String
        public let region: String     // resolved admin1 name
        public let country: String    // resolved country name
        public init(name: String, lat: Double, lon: Double,
                    countryCode: String, region: String, country: String)
    }

    /// Build the spatial index from an in-memory city list. Empty list → a geocoder that always
    /// returns nil (the graceful-absence case). PURE — no I/O.
    public init(cities: [City])

    /// Nearest city to (lat, lon) within `maxKilometers` (default ~250 km) → its Place, else nil
    /// (open ocean / absurdly remote point / empty dataset). Great-circle (haversine) distance.
    public func place(lat: Double, lon: Double, maxKilometers: Double = 250) -> Place?

    /// True iff the geocoder holds any city (i.e. the dataset loaded). Drives stage availability.
    public var isLoaded: Bool { get }
}
```

### 3.2 Spatial lookup — grid bucket

The index is a simple **lat/lon grid bucket**: cities are placed into 1°×1° cells keyed by `(floor(lat), floor(lon))`. `place(lat:lon:)` scans the query cell plus its 8 neighbors (a 3×3 block of cells, widened by one ring if the block is empty near sparse regions), computing **haversine** distance to each candidate and returning the nearest within `maxKilometers`. This is O(cities-per-cell) per query — fast, dependency-free, deterministic, and trivially correct to reason about. (A k-d tree is the documented alternative behind the same `place(_:)` signature; the grid bucket is plenty at ~25 k cities and far simpler to test. No external spatial-index dependency.)

> Longitude-cell width shrinks toward the poles, but at city latitudes the 1° grid with a one-ring neighbor scan comfortably covers the `maxKilometers` radius; the neighbor-ring widening guards the sparse high-latitude / antimeridian edge cases. The math (haversine, antimeridian wrap) lives in one place and is unit-tested.

### 3.3 Loading from the GeoNames files (App/Core boundary)

A separate **loader** (`GeoNamesLoader`, same folder) parses the three tab-separated files into `[ReverseGeocoder.City]` and constructs the `ReverseGeocoder`:

```swift
public enum GeoNamesLoader {
    /// Parse cities15000.txt + admin1CodesASCII.txt + countryInfo.txt under `directory`
    /// (…/geonames/) → a loaded ReverseGeocoder. Returns an EMPTY-but-valid geocoder (isLoaded ==
    /// false) if the directory or the required files are absent — never throws for absence.
    public static func load(directory: URL?) -> ReverseGeocoder
}
```

- Parsing is **pure-ish I/O at the edge**: read each file, split on tabs, build the admin1 + country lookup dictionaries first, then map each `cities15000` row to a `City` (resolving its region via `"<cc>.<admin1>"` and its country via `<cc>`). Rows that fail to parse (short columns, non-numeric lat/lon) are skipped, never fatal.
- **Graceful absence is total:** `directory == nil`, a missing dir, or missing files → `ReverseGeocoder(cities: [])` (an empty geocoder whose `place(...)` always returns `nil`, `isLoaded == false`). The loader **never throws on absence** and never touches the network.
- The loader is constructed once (lazily, like `EmbedStage`'s model load) and reused for the whole library; parsing ~25 k rows is milliseconds.

### 3.4 Representative test (illustrative — synthetic, no real dataset)

```swift
@Test func nearestCityWins() {

        ReverseGeocoder.City(name: "Taipei", lat: 25.03, lon: 121.57,
                             countryCode: "TW", region: "Taipei", country: "Taiwan"),
        ReverseGeocoder.City(name: "Tokyo", lat: 35.68, lon: 139.69,
                             countryCode: "JP", region: "Tokyo", country: "Japan"),
```

## 4. Storage — catalog migration **v9** (machine-derived, rebuildable)

### 4.1 The `geocode` table (migration v9)

Purely machine-derived/rebuildable — a deterministic function of `assets.latitude`/`assets.longitude` + the dataset. **No human columns, no sidecar mirror.**

```
geocode(hash        TEXT PRIMARY KEY,   -- references assets.hash (the geotagged photo)
        city        TEXT,               -- nearest-city name (may be "" if unresolved-but-derived)
        region      TEXT,               -- admin1 region name (may be "")
        country     TEXT,               -- country name
        countryCode TEXT)               -- ISO-2 code (drives the Search country filter + grouping)
```

- Keyed by `hash` (PK) — one place per asset; an upsert idempotently replaces it on re-derivation.
- No FK on `geocode.hash → assets.hash` (same rationale as `instances`/`faces`: rebuildable; a re-derive repairs orphans).
- Indexed on `countryCode` and `city` for the Search place picker (`distinctPlaces()`).
- A photo whose GPS resolves to **open ocean / no nearby city** gets **no row** (the stage marks the job `done` without writing — it was analyzed, there's just no place). So presence of a row == "has a resolved place".

### 4.2 CRUD (Core, TDD) — `Catalog+Geocode.swift`

```swift
public struct GeocodeRow: Sendable, Equatable {
    public var hash: String
    public var city: String
    public var region: String
    public var country: String
    public var countryCode: String
    public init(hash: String, city: String, region: String, country: String, countryCode: String)
}

extension Catalog {
    /// Idempotent upsert of a resolved place for an asset.
    public func upsertGeocode(_ row: GeocodeRow) throws

    /// The resolved place for one asset (for the inspector), or nil if not yet geocoded / no place.
    public func geocode(forHash hash: String) throws -> GeocodeRow?

    /// Distinct places present in the library (for the Search place picker), grouped by country
    /// then city, each with the asset count. Ordered country, then city.
    public func distinctPlaces() throws -> [PlaceFacet]
}

public struct PlaceFacet: Sendable, Equatable, Hashable {
    public let countryCode: String
    public let country: String
    public let city: String          // "" → a country-level facet ("all of Japan")
    public let count: Int
}
```

`distinctPlaces()` returns both **country-level** facets (one per `countryCode`, `city = ""`, summed count) and **city-level** facets (per distinct `(countryCode, city)`), so the Search picker can offer "Japan" *and* "Tokyo, Japan". Implemented with two grouped `SELECT`s (or one `GROUP BY` plus a country roll-up) over `geocode` joined to nothing (it's already hash-keyed).

### 4.3 GPS eligibility — the derivation-jobs mechanism

The geocode stage must only process **photos with GPS**. `eligibleKind(forStage: "geocode")` returns `"photo"` as before, but the pending query needs a **lat/lon-non-null** restriction so non-GPS photos never enter the geocode pending set (and don't inflate progress). Two options, both documented; the plan implements the **stage-specific pending filter**:

- `Catalog.pendingDerivation(stage:)` gains a small per-stage `WHERE` augmentation: for `stage == "geocode"`, add `AND a.latitude IS NOT NULL AND a.longitude IS NOT NULL`. Symmetrically, `derivationProgress(stage:)`'s `total` counts only geotagged photos for the geocode stage, so the "Analyzing N/M" line is honest.
- This keeps the geocode stage's eligible set = geotagged photos; a photo with no GPS is never pending and never counted. (The existing OCR/Embed/Faces stages are unaffected — the augmentation is gated on `stage == "geocode"`.)

### 4.4 Snapshot / portability

`geocode` rides along in the per-drive `catalog-snapshot/catalog.sqlite` (VACUUM INTO) like every other catalog table. It is a **droppable, rebuildable cache** — re-derive from the assets' lat/lon + the dataset. An external snapshot reader **MAY** use it as a hint for "where was this photo" but MUST treat it as absent in older snapshots and MUST NOT rely on it (the assets' lat/lon is the authoritative GPS; the place name is a convenience). `Catalog.schemaVersion` bumps **8 → 9**.

## 5. The `GeocodeStage` — the fourth derivation stage

`Sources/OpenPhotoCore/Derivation/GeocodeStage.swift`. Mirrors `EmbedStage`'s structure (a class holding the lazily-loaded resource + graceful absence) but is **simpler — it needs no image bytes**.

### 5.1 Key wrinkle: geocode keys off the catalog's GPS, not the image file

OCR/Embed/Faces all read the asset's **bytes** (the runner resolves a reachable URL and passes it in). Geocode needs **only lat/lon**, which already live in `assets`. So `GeocodeStage.run(hash:url:catalog:)`:

1. **Ignores `url`** for its computation — it reads `catalog`'s stored lat/lon for `hash` (a tiny `Catalog.coordinate(forHash:)` read), not the image. (The runner still resolves a URL because the pending set is GPS-gated to assets that *have* coordinates; but a geocode job does not actually need the file to be reachable. The plan notes this and keeps geocode keyed off catalog GPS — a drive-only asset whose drive is unplugged can *still* be geocoded because its lat/lon is in the catalog. To get that benefit, the runner's per-item reachable-URL skip is geocode-exempt — see §7.)
2. `ReverseGeocoder.place(lat:lon:)` → a `Place?`.
3. If a place resolves, `catalog.upsertGeocode(GeocodeRow(hash:, city:, region:, country:, countryCode:))`. If `nil` (ocean/remote), write nothing.
4. Return `true` either way (analyzed; success). The job is marked `done` and never re-tried.

```swift
public final class GeocodeStage: @unchecked Sendable {
    public let id = "geocode"
    public let eligibleKind = "photo"
    private let geocoder: ReverseGeocoder      // lazily loaded once from the bundled dataset
    /// - Parameter datasetDirectory: …/geonames/. When nil, defaults to Bundle.main.resourceURL.
    public init(datasetDirectory: URL? = nil)
}

extension GeocodeStage: DerivationStage {
    /// True iff the GeoNames dataset loaded (isLoaded). Absent dataset → stage skipped (jobs stay
    /// pending), exactly like EmbedStage with a missing model.
    public var isAvailable: Bool { geocoder.isLoaded }
    public func run(hash: String, url: URL, catalog: Catalog) async -> Bool {
        guard let (lat, lon) = try? catalog.coordinate(forHash: hash) else { return true }  // no GPS → nothing to do
        if let place = geocoder.place(lat: lat, lon: lon) {
            try? catalog.upsertGeocode(GeocodeRow(hash: hash, city: place.city, region: place.region,
                                                  country: place.country, countryCode: place.countryCode))
        }
        return true   // analyzed (place written or no-nearby-place) → success, never re-tried
    }
}
```

- `isAvailable = geocoder.isLoaded` (lazily loaded from `datasetDirectory ?? Bundle.main.resourceURL` + `/geonames/`). Absent dataset → `false` → the runner skips the whole stage (jobs pending, nothing failed) — the established graceful-absence path.
- `run` returns `false` only if it genuinely can't proceed (it essentially never does — no GPS is *success* "nothing to geocode", a missing nearby city is *success* "no place"). There's no retry loop because there's no transient failure mode (the dataset is either loaded for the whole run or the stage is skipped via `isAvailable`).
- `coordinate(forHash:) -> (Double, Double)?` is a tiny new catalog read (lat/lon, nil if either is null).

## 6. Pipeline integration — the fourth stage

The 4.1 runner already drains a registry off-main with combined progress, reachable-URL resolution, and retry-caps. 4.4 adds **one element**:

```swift
private let derivationStages: [any DerivationStage] =
    [OCRDerivationStage(), EmbedStage(), FaceDerivationStage(), GeocodeStage()]
```

- `GeocodeStage.isAvailable == geocoder.isLoaded`, so on a machine without the dataset the stage is skipped (jobs pending) and `combinedProgress()` excludes it — no inflated total, identical to the embed-model-absent behavior.
- The combined "Analyzing N/M…" line now sums OCR + Embed + Faces + Geocode over **available** stages; the geocode total counts only **geotagged** photos (§4.3) so it's honest.
- **Reachable-URL exemption (the one runner change):** the geocode stage doesn't need the file — only the catalog's lat/lon. The runner's per-item `goodCopyURL` skip is gated so a `geocode` job is **not** skipped when the file is unreachable (a drive-only asset whose drive is unplugged still gets a place name from its stored GPS). Concretely: the runner passes a sentinel/placeholder URL (or skips the reachability check) when `stage.id == "geocode"`. This is the only modification to `drainDerivation`; OCR/Embed/Faces behavior is unchanged.
- After a geocode drain, mark a `geocodeDirty` flag (mirrors `semanticIndexDirty`/`facesDirty`) so the Map/place-picker can refresh.

## 7. App — inspector place + Search place filter + Map

### 7.1 Inspector place (App)

`InspectorView` already has a location section (it shows a mini-map marker for a geotagged asset and a `locations(for:)` list). 4.4 adds the **resolved place name** there: for the selected item, `catalog.geocode(forHash:)` → render "Taipei, Taiwan" (city + country; include region when it differs from the city) as a row in the details/location section. Nil → omit the row (no GPS, or not yet geocoded). Tapping the place name **deep-links into Search** filtered by that place (mirrors the existing inspector-tag/person fold-in: set `searchFilters.place`/`country` + `selection = .search`).

### 7.2 Place filter in unified Search (App + Core)

**Core:**
- `SearchFilters` gains `public var place: PlaceFilter?` where `PlaceFilter` is a small value identifying either a country (`countryCode`) or a city (`countryCode` + `city`). Defaulted `nil`, included in `isEmpty` + `init`.
- `Catalog.structuredFilter(_:)` adds, when `place != nil`, a constraint that the asset's hash appears in `geocode` matching the facet:
  ```sql
  AND hash IN (SELECT hash FROM geocode WHERE countryCode = ?)              -- country facet
  AND hash IN (SELECT hash FROM geocode WHERE countryCode = ? AND city = ?) -- city facet
  ```
  parameterized, joined into the existing timeline-union `WHERE` builder alongside camera/rating/tags/person. Composes with every other filter.

**App:** the `SearchView` filter bar gains a **place menu** (from `Catalog.distinctPlaces()`, loaded in `.task` like `cameras`/`allTags`/`allPeople`): "Anywhere" + a grouped list (countries, then cities within each) → sets `searchFilters.place` and re-runs. A `filterChip` shows the selected place name, active when `place != nil`.

### 7.3 The Map (App) — `SidebarItem.map` + `MapView`

- **`SidebarItem.map`** — a new case in the `SidebarItem` enum (Library group; placed after `.folders`/`.people`, before `.search` — e.g. `case timeline, folders, people, map, search, drives, bin`), with `label "Map"` + `symbol "map"`. It's picked up automatically by the `ForEach(SidebarItem.allCases)` sidebar and the `RootView.detail` switch (`case .map: MapView(state:)`).
- **`MapView`** (`Sources/OpenPhotoApp/Map/MapView.swift`) — SwiftUI **MapKit `Map`**, Apple-Photos-style:
  - **Data:** load every geotagged asset's `(hash, lat, lon, takenAtMs)` via a small `Catalog.geotaggedAssets()` read (assets with non-null lat/lon — independent of whether they're geocoded; the Map plots GPS, the geocode names are a bonus). Photos without GPS simply aren't in the set and never appear.
  - **Clustering:** **manual grid-cluster at the current zoom** — bucket the visible assets into a grid sized to the current `MKCoordinateRegion`/zoom (so two pins that are far apart on screen stay separate while a dense city collapses to one cluster), each cluster annotated with a **representative thumbnail** (the cluster's newest asset, loaded via `ThumbnailImage`/`ThumbnailStore`) + a count badge. (MapKit's built-in `clusteringIdentifier` is the documented alternative; the manual grid-cluster gives the precise Apple-Photos "thumbnail bubble" look and reuses our thumbnail cache directly. The plan picks the manual grid-cluster.)
  - **Interaction (Apple-Photos-style):** tapping a cluster **zooms the map to that cluster's bounding region** (which re-runs the grid-cluster at the finer zoom → the cluster splits). A cluster (or the current region) can **open a photo grid** of its assets — reuse the `MediaTile` grid (over `items(forHashes:)`) in a sheet/detail, so "the photos here" is one tap. A single-photo annotation opens the viewer.
  - **Re-cluster on pan/zoom:** the cluster set is recomputed (off-main, debounced) whenever the visible `MKCoordinateRegion` changes, so panning/zooming continuously re-buckets — the Apple-Photos feel.
  - Empty state when no assets have GPS ("No photos with location").

## 8. Data flow

```
library open / scan / ingest → derivation runner drains [OCR, Embed, Faces, Geocode]
   Geocode (photos WITH GPS only): read catalog lat/lon for hash → ReverseGeocoder.place(lat,lon)
      → upsertGeocode(city, region, country, countryCode)  [or no row for ocean/remote]
      off-main, resumable; needs NO image bytes (keys off catalog GPS) → geocodeDirty = true

inspector (selected photo) →
   geocode(forHash:) → "Taipei, Taiwan"   (tap → Search filtered by that place)

Search: filters.place = {country: "JP"} → structuredFilter joins geocode → MediaTile grid
   (composes with date / camera / person / tag …)

Map (SidebarItem.map) →
   geotaggedAssets() → grid-cluster at current zoom → thumbnail-bubble annotations + counts
   pan/zoom → re-cluster; tap a cluster → zoom (splits) / open the photos-here grid

dataset absent (fresh checkout, no .models/) →
   GeocodeStage.isAvailable == false → stage skipped, jobs pending, no place names, no crash
   (Map still plots GPS pins; inspector/Search place simply have nothing to show)
```

## 9. Error handling / edge cases

| Case | Behavior |
|---|---|
| Photo has no GPS | Never enters the geocode pending set (GPS-gated query); not counted in progress. If somehow `run`, `coordinate(forHash:)` is nil → returns `true` (nothing to do), job `done`. |
| GPS resolves to open ocean / no city within `maxKilometers` | `place(...)` returns nil → **no `geocode` row written**; job `done` (analyzed, no place). Inspector/Search/Map place simply absent for it (the Map still plots its GPS pin). |
| Dataset absent (no `.models/geonames/`) | `geocoder.isLoaded == false` → `GeocodeStage.isAvailable == false` → stage skipped, jobs **pending** (not failed), nothing crashes; resumes once the dataset ships. Mirrors EmbedStage's model-absent path. |
| Dataset present but a file malformed / a row unparseable | The loader skips bad rows (short columns, non-numeric lat/lon); a totally-missing required file → empty geocoder (graceful). Never fatal. |
| Drive-only asset, drive unplugged | Geocode keys off catalog GPS, not the file → it **can** still be geocoded (the runner's reachable-URL skip is geocode-exempt, §6). OCR/Embed/Faces still skip-and-retry as before. |
| Re-derivation (re-poke) over an already-geocoded photo | `upsertGeocode` idempotently replaces the row; the `done` job means it isn't re-run unless the job is cleared. Dropping the table + jobs → full re-derive from lat/lon. |
| Antimeridian / high-latitude query | Haversine handles the wrap; the grid's neighbor-ring widening covers the sparse/poles/±180° edge cases. Unit-tested. |
| Place name vs region duplication (city == region, e.g. "Taipei"/"Taipei") | The inspector/UI shows "city, country" and includes the region only when it adds information (region ≠ city). The `geocode` row stores all four fields regardless. |
| Map with zero geotagged photos | Empty state ("No photos with location"); no annotations. |
| Map at extreme zoom-out (whole world) | Grid-cluster collapses dense areas to a few thumbnail bubbles with large counts; performant (clustering is over `(hash, lat, lon)` tuples, off-main, debounced — thumbnails only loaded for the handful of visible cluster representatives). |
