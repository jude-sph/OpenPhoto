## Scale model, hot paths & index coverage

## 1. Scale model

OpenPhoto's cost is driven by a small set of countable quantities. Mapped to this codebase:

| Param | Maps to | Where it lives |
|---|---|---|
| **N** | Total catalogued assets (distinct `assets.hash`, photos + videos). The dominant term for every browse/scan/derivation pass. | `assets` table |
| **M** | Local **instances** = (asset × local vault) rows. M ≥ N because the same hash can appear in two local vaults (intentional v1 dup); `instances` is keyed `(vaultID, relPath)`. | `instances` table |
| **F** | Distinct folders / `dirPath` values on the Mac. Drives `folderTree`, `folderCounts`, folder filters. | `instances.dirPath`, `vault_presence.dirPath` |
| **V** | Number of registered vaults (1 local + connected/known drives). Multiplies presence rows and sync sources. | `vaults` table |
| **D** | **Drive-only** presence rows = assets on a drive with no local instance. Each adds a `vault_presence` row and a row to the timeline UNION's drive branch. | `vault_presence` table |
| **E** | Embeddings present (CLIP image vectors, ≤ N photos). Sizes the in-memory `SemanticIndex` matrix (E × dim fp32) and the brute-force search `vDSP_mmul`. | `embeddings` table |
| **P** | Pending derivation jobs across stages (ocr/embed/faces/geocode/phash) — the unstarted/failed-retryable work the drain must traverse. Worst case ≈ 5·N at first index. | `derivation_jobs` + `pendingDerivation` |
| **T** | OCR text corpus (recognized-text rows in the FTS5 `ocr` table, ≤ N photos). Sizes full-text search. | `ocr` virtual table |
| **K** | Detected faces (≈ a few × N for people-heavy libraries); of those, **U** = unassigned auto faces feeds the clusterer. | `faces` table |
| **B** | Geotagged assets (assets with non-null lat/lon, ≤ N). Drives the Map surface + grid clustering. | `assets.latitude/longitude`, `geocode` |

**Three reference sizes** (N = catalogued assets; others scale roughly with N):

- **10k** — single user, one local vault, maybe one backup drive. N≈10k, M≈N, F≈hundreds, E/T/B a fraction of N, K≈10–30k. Everything fits comfortably in memory; full-library in-Swift passes are fine.
- **100k** — the realistic "friend's real import" target (the live acceptance test). N≈100k, F≈1–5k, K≈100–300k, E≈80k. In-memory full-library materializations (timeline UNION → `[TimelineItem]`, `SemanticIndex` matrix, face clustering) start to cost real seconds/RAM. This is the size the memory-safety work (autorelease pools, autorelease OOM) was provoked by.
- **1M** — stress ceiling / multi-drive archive. N≈1M, D potentially large (cold-storage drives), E≈1M × dim fp32 (hundreds of MB resident), K in the millions, F≈10k+. Several hot paths here are O(N) full scans or O(N²)/O(N·k) Swift loops that will not hold; this is where the deferred tile-grid renderer and the un-indexed timeline UNION become the bottlenecks.

**Headline gaps for the 100k–1M range:**
1. The **timeline UNION** (`timelineSQL`) is the spine of browse/folder/search, yet every wrapper re-materializes it and re-sorts by `takenAtMs` — the index can't propagate through `SELECT * FROM (… UNION ALL …)`, so all three of `refreshQueries`'s calls pay an O(N+M+D) scan+sort.
2. The **drive-only branch** relies on a **correlated `MIN(rowid)` subquery** for per-asset dedupe (appears in `driveSelect`, `folderCounts`); cost grows with D.
3. **Search predicates** `rating`/`favorite`/`kind`/`cameraModel` and all **tag** queries (`json_each`) have **no index** — Search is a full union scan, and `people()` plus `folderCounts` lean on **correlated subqueries**.

# Part I — Scalability & complexity

## How OpenPhoto scales (10k → 100k → 1M)

_This narrative was synthesized in the main loop from the complexity findings below, after the workflow's dedicated synthesis agent was cut off by the spend limit. It is grounded in those findings but, like them, has not been through the adversarial verification pass._

OpenPhoto is fast and correct at its tested size and degrades along a small number of predictable axes as the library grows. The pattern is consistent: **full-library work that is materialized in RAM, run on the main actor, or done with a whole-file-rewrite/`O(n²)` algorithm.** None of these bite at 10k; most bite hard at 100k; several are disqualifying at 1M.

**At ~10k (today's tested size).** Everything fits comfortably. The whole-library timeline UNION (`Catalog/Queries.swift` `timelineSQL`), the `SemanticIndex` fp32 matrix, the `O(n²)` matchers, and the whole-file manifest/bin rewrites are all small enough that no one notices. The only user-visible scaling problem at this size is the **dense-grid tile hitch** (Part III) and, intermittently, the per-search Core ML recompile (`CX` "per-query `EmbedStage()`"), which is independent of N and so is felt even here.

**At ~100k (the "friend's real import" acceptance target).** This is where the design's central assumption — that full-library passes are cheap — stops holding, and it shows up first on the main actor. `refreshQueries()` is fired from ~27 sites and on every mutation synchronously rebuilds **all** derived browse state: it materializes the entire timeline UNION (`O(N log N)`, no pagination, and the wrapping `SELECT * FROM (union)` prevents the `takenAtMs` index or a `LIMIT` from short-circuiting it) into **two** resident copies (`sections` + `flatItems`), then walks the **entire vault directory tree** with a `FileManager` enumerator for `folderTree()`. So every star, tag edit, delete, sync tick, and derivation tick produces a multi-hundred-ms beachball — worse on a slow or external volume because the FS walk is in the hot path. Around it: folder navigation recursively `GLOB`-scans the whole library (`dirPath GLOB '<dir>/*'` can't use a B-tree index, and the wrapping subquery hides even the `dirPath = ?` equality from its index); structured filters full-scan with a `json_each` parse per row; the first-ever scan is single-threaded (wall-clock = the sum of every file's hash + ImageIO latency) and **any FSEvent re-runs the full scan**, not just the changed paths; `SemanticIndex` pins ~205 MB of fp32 and **rebuilds after every embed batch**; and the bulk operations are quadratic — drift-repair, folder move, and delete each rewrite the **whole** manifest (or bin log) **per file** (`O(R·M)`), so "repair 2,000 files" or "move 500 photos" rewrites a 100k-line manifest thousands of times, and deleting 1,000 photos into a 50k-entry bin is `O(K·B)`. The matchers compound it: face clustering is `O(U²·d)` over tens of thousands of unnamed faces, the duplicate grouper is `O(n²)` per folder (catastrophic if a Takeout/Photos import dumped everything into one directory), and Live-photo pairing is `O(P²)` on an iPhone library where most items are Live.

**At ~1M (archive / stress ceiling).** The `O(N)` materializations, `O(n²)` matchers, and whole-file rewrites do not hold. The timeline UNION sort of a million rows on the main actor is a multi-second freeze on every refresh; `SemanticIndex` wants ~2 GB of fp32 resident for the session; the single-threaded, non-resumable scan takes hours and can't checkpoint; and the quadratic bulk paths turn routine maintenance into minutes of disk churn. This size is simply not reachable without the structural changes below.

**The 5–6 changes that move the ceiling the most**, in rough leverage order:

1. **Get heavy reads off the main actor and stop full-rebuilding on every event.** Make the `Catalog`/`LibraryService` boundary `async`/an actor so `timelineItems`/`folderTree`/`reconcileFinderTags` can't run inline on `@MainActor`, and replace `refreshQueries()`'s fire-from-27-sites full rebuild with event-scoped invalidation that recomputes only what changed, off-main, publishing results back. This single change removes the per-mutation beachball — the most visible 100k regression.
2. **Window the browse.** Keyset-paginate on `takenAtMs` and stop materializing the whole UNION twice; either denormalize a browse table or reshape the query so the `takenAtMs` index + `LIMIT` can short-circuit. RAM and refresh cost stop scaling with N.
3. **Index for the predicates that exist, and kill the GLOB recursion.** Add the missing `dirPath` index and the `assets` columns the structured filter uses; replace recursive folder browse with a `dirPath`-prefix range (`>=`/`<` bounds) or a folder closure table.
4. **Replace whole-file rewrites with batch mutation.** Give `manifest.jsonl`, the bin log, and the import registry an owner with read-once/mutate-set/write-once APIs. This converts the quadratic bulk repair/move/delete paths to linear.
5. **Bound and incrementalize the scan.** Parallelize hashing + metadata extraction with bounded concurrency, make FSEvents incremental to the delivered `eventPaths` instead of a full re-walk, and make the pass resumable/streamed so memory stays flat.
6. **Replace the brute-force matchers.** A Hamming/BK-tree index for pHash dedup, an ANN or blocked-clustering approach for faces (with a face-identity embedding, not the generic scene feature-print), and a base-name dictionary for Live pairing instead of `firstIndex`.

## Detailed complexity findings

### Critical

#### CX01 — timelineItems / timelineSQL union materialized as a wrapping subquery for every browse query

- **Severity:** Critical  |  **Confidence:** Confirmed  |  **Subsystem:** Catalog, queries & search  |  **Effort:** L
- **Location:** `Sources/OpenPhotoCore/Catalog/Queries.swift:30`, `Sources/OpenPhotoCore/Catalog/Queries.swift:33-40`, `Sources/OpenPhotoCore/Catalog/Queries.swift:35`
- **Time:** O(N log N) — builds the full local∪drive union of all N rows then sorts by takenAtMs; the wrapping `SELECT * FROM (union)` forces full materialization, so no index/LIMIT can short-circuit it
- **Space:** O(N) resident — every catalogued asset returned as a full ~20-column TimelineItem struct (hash/relPath/dirPath strings dominate); nothing paginated
- **Trigger:** refreshQueries() → timelineSections() on app open and after EVERY mutating action (delete, restore, evict, import, sync, drive connect/disconnect, undo, tag-sync, metadata edit). Also PeekSource and Finder-tag full reconcile.
- **Cliff:** 100k: each refresh is a few-hundred-ms main-actor stall + tens of MB transient allocation, felt as UI hitch after every edit. 1M: multi-second beachball on open and after every action; hundreds of MB allocated per refresh, GC/ARC churn.
- **Improvement:** Stop returning all N rows: paginate the timeline (keyset pagination on takenAtMs, fetch the visible window + lookahead, target O(P log N) with P≈screen). Do NOT wrap the union in `SELECT * FROM (...)` — push WHERE/ORDER/LIMIT into each UNION branch so SQLite uses idx on takenAtMs/dirPath. Drop driveRelPath/relPath from the hot path and hydrate lazily. Cache the section structure and apply incremental diffs instead of full rebuild on every refreshQueries.

#### CX02 — timelineItems() — whole-library browse fetch (UNION ALL of local instances + drive-only presence, ORDER BY takenAtMs DESC, no LIMIT)

- **Severity:** Critical  |  **Confidence:** Confirmed  |  **Subsystem:** LibraryService orchestration & selection  |  **Effort:** L
- **Location:** `Sources/OpenPhotoCore/Catalog/Queries.swift:33`, `Sources/OpenPhotoCore/Catalog/Queries.swift:30`, `Sources/OpenPhotoCore/LibraryService.swift:88`, `Sources/OpenPhotoApp/AppState.swift:1533`
- **Time:** O(N log N) — the outer SELECT * FROM (union) ORDER BY takenAtMs cannot use assets.takenAtMs index through the UNION subquery, so SQLite filesorts all N rows; dominant term is the N-row sort + N-row row decode into structs
- **Space:** O(N) — every catalogued asset decoded into a TimelineItem struct array held resident in AppState.sections + AppState.flatItems simultaneously (two copies of ~N items)
- **Trigger:** refreshQueries(), called after every delete/restore/evict/move/metadata edit/drive connect/disconnect/import and on every grouping toggle (TimelineView.swift:180); runs synchronously on the @MainActor
- **Cliff:** 100k assets: a multi-hundred-ms hang on the main thread after every mutation and every grouping change (visible stutter); 1M: multi-second beachball on each refresh plus hundreds of MB resident for the duplicated struct arrays
- **Improvement:** Paginate/window the browse query (LIMIT/OFFSET or keyset on takenAtMs which IS indexed) so only the visible P + buffer rows are fetched — target O(P log P) fetch, O(P) resident; or run timelineItems off the main actor and diff into the grid. Avoid holding both sections and flatItems; derive flat indices lazily.

#### CX03 — CatalogIngest.ingestDriveFile — per-file knownHashes() + full presence delete/reinsert, looped per adopted file

- **Severity:** Critical  |  **Confidence:** Confirmed  |  **Subsystem:** Sync & send engines  |  **Effort:** M
- **Location:** `Sources/OpenPhotoCore/Sync/CatalogIngest.swift:21 catalog.knownHashes() per file`, `Sources/OpenPhotoCore/Sync/CatalogIngest.swift:33 vaultPresenceRows(forVault) per file`, `Sources/OpenPhotoCore/Sync/CatalogIngest.swift:35 replaceVaultPresence (full DELETE+reinsert) per file`, `Sources/OpenPhotoApp/AppState.swift:1265 ingestAdopted loop over relPaths`, `Sources/OpenPhotoCore/Catalog/Catalog.swift:264 replaceVaultPresence`, `Sources/OpenPhotoCore/Catalog/Catalog.swift:481 knownHashes (SELECT all hashes into Set)`
- **Time:** O(A·(N + P)) — A = adopted/ingested files. Per file: knownHashes() loads ALL N catalog hashes into a Set, vaultPresenceRows loads the drive's full presence set P, and replaceVaultPresence DELETEs all P presence rows for the vault then reinserts P+1 rows. - **Space:** O(N + P) resident per file (full hash Set + full presence array rebuilt each iteration).
- **Trigger:** 'Adopt all' on a drive with many unknown files (DriftReviewSheet 'Adopt all'), or any flow that ingests drive-resident files one at a time. A scales with unknown-file count on a freshly-attached/foreign drive.
- **Cliff:** Adopting 10k unknown files into a 100k catalog: 10k × (100k-hash Set load + full 10k-row presence delete/reinsert) ≈ 1e9 row touches + 10k SQLite write transactions. Many-minute beachball; the presence rewrite alone is O(A·P) = quadratic in the adopted set.

Target O(N + A): one hash-set load, one presence write.

#### CX06 — SemanticIndex — full embedding matrix resident in RAM + per-query full sort

- **Severity:** High  |  **Confidence:** Confirmed  |  **Subsystem:** Catalog, queries & search  |  **Effort:** M
- **Location:** `Sources/OpenPhotoCore/Search/SemanticIndex.swift:6-39`, `Sources/OpenPhotoCore/Search/SemanticIndex.swift:20-22`, `Sources/OpenPhotoCore/Search/SemanticIndex.swift:36`, `Sources/OpenPhotoApp/AppState.swift:503`
- **Time:** Build O(E·dim) (unpack Float16→Float32, one contiguous append). Query O(E·dim) for the vDSP_mmul plus O(E log E) to fully sort ALL scores just to take topN — the sort, not the matmul, dominates ranking and is wasteful for topN≪E.
- **Space:** O(E·dim) resident fp32 = E·512·4 bytes. 100k → ~205 MB; 1M → ~2.05 GB held live for the whole session (the on-disk blobs are Float16/half that, but RAM copy is fp32). Plus a transient O(E) score array per query.
- **Trigger:** Built lazily in runSearch() whenever semanticIndexDirty (set true after every embed-drain and after import growth) → first text search after any new embeddings rebuilds the whole matrix. query() runs on every text search (300ms debounce).
- **Cliff:** 100k: ~200MB persistent footprint is tolerable but the full rebuild after each embed batch stalls the first search by hundreds of ms. 1M: ~2GB resident → memory pressure / OOM risk on 8–16GB Macs (note the prior 41GB indexing OOM in MEMORY.md), and rebuild becomes multi-second.
- **Improvement:** Keep the matrix as Float16 and use a half-precision dot (or quantize) to halve RAM. Replace the full sort with a bounded top-N selection (partial sort / heap, target O(E + topN log topN)). Make rebuild incremental (append new rows instead of reloading all E). For 1M, move to an ANN index (HNSW/IVF) or memory-mapped vectors to cap resident RAM and get sublinear query.

#### CX07 — FaceClusterer.cluster — greedy single-link agglomerative face clustering

- **Severity:** High  |  **Confidence:** Confirmed  |  **Subsystem:** Cull & faces (clustering/dedup)  |  **Effort:** L
- **Location:** `Sources/OpenPhotoCore/Faces/FaceClusterer.swift:43-60`, `Sources/OpenPhotoCore/Faces/FaceClusterer.swift:47-49`, `Sources/OpenPhotoCore/Faces/FaceClusterer.swift:83-93`, `Sources/OpenPhotoApp/AppState.swift:217`
- **Time:** O(n²·d) worst-case, dominant term the nested member scan × per-pair dot product. n = unassigned auto faces (personID IS NULL, source='auto'), d = embedding dim (~512-768 floats per VNGenerateImageFeaturePrint vector). Outer loop over n faces (line 43); inner loops over every existing cluster and, via members.contains (line 47), over every member already placed → in the degenerate case where most faces stay separate (singletons) this is Σ comparisons ≈ n²/2, each calling cosineDistance which is an O(d) dot product (line 88-89). So ~ (n²/2)·d float-multiply-adds.
- **Space:** O(n·d) resident. normalized[] holds every face's full unit vector (line 34-38): n × d Float = for 50k unassigned faces at d=512 that is ~100MB of Float arrays held for the whole pass, plus clusters[] index arrays O(n).
- **Trigger:** AppState.loadPeople() → Task.detached → FaceClusterer.cluster (AppState.swift:217). Fires when the People view appears with empty state (PeopleView.swift:66-67) and is re-called on EVERY face mutation: nameCluster, assign/unassign/merge/rename and after each derivation drain set facesDirty=true then loadPeople() (AppState.swift:309,326,345,365,398,1513). So a full O(n²·d) recluster runs after every single person-naming action.
- **Cliff:** Power-user 100k library where face detection yields tens of thousands of UNNAMED faces (a fresh library before any naming is the worst case — everyone is unassigned). At ~20k-50k unassigned faces the detached pass takes many seconds to a minute of CPU; because it re-runs after every nameCluster, the People view feels like it hangs/respins a spinner (facesLoading) after each name, and a 100k cold library can spin for minutes. Symptom: long facesLoading spinner, stale suggestions, high CPU; not a main-thread beachball (it is detached) but a perceived hang of the People feature.
- **Improvement:** Two fixes. (1) Correctness+structure: single-link chaining (join cluster if ANY member within threshold, line 47) transitively merges A~B~C even when A and C are different people — replace with centroid/medoid linkage or a proper density method (HDBSCAN) so a face joins only if near the cluster CENTROID, dropping per-join cost from O(cluster size·d) to O(d). (2) Scaling: build the index over face embeddings once and do approximate nearest-neighbour (a vptree / HNSW / LSH over the 512-d unit vectors) → target O(n·log n·d) build and O(n·d) query instead of O(n²·d); cap the unassigned working set and recluster incrementally (only the newly-detected faces) instead of from scratch on every facesDirty.

#### CX08 — DuplicateGrouper.group — per-folder union-find over pairwise pHash Hamming

- **Severity:** High  |  **Confidence:** Confirmed  |  **Subsystem:** Cull & faces (clustering/dedup)  |  **Effort:** M
- **Location:** `Sources/OpenPhotoCore/Cull/DuplicateGrouper.swift:18-23`, `Sources/OpenPhotoCore/Cull/DuplicateGrouper.swift:19-20`, `Sources/OpenPhotoApp/AppState.swift:260`, `Sources/OpenPhotoApp/AppState.swift:265`
- **Time:** O(Σ_f n_f²) where n_f = photos in folder f (Σ n_f ≈ N). Within each folder the double loop (line 18-19) compares every pair, each comparison an O(1) 64-bit XOR popcount (PerceptualHash.hamming). Best case (photos spread over F folders of even size) ≈ N²/F; WORST case (one giant folder, e.g. a flat dump of everything in one directory, common after a Google Takeout / Apple Photos import) collapses to a single O(N²) bucket. Union-find itself is near-linear; the pairwise scan dominates.
- **Space:** O(N) — byFolder dictionary holds every (hash,value) row plus parent[] arrays per folder. Modest, but the input rows[] is materialised for the whole library.
- **Trigger:** AppState.loadCullGroups() (.duplicates → hammingThreshold 2; .similar → 6) at AppState.swift:260/265. - **Cliff:** A power-user (100k) or anyone whose import landed many thousands of photos into ONE folder. Symptom: long spinner / unresponsive Cleanup mode switching at large flat folders.
- **Improvement:** Replace the O(n_f²) brute-force Hamming scan with a BK-tree (metric tree over Hamming distance) or multi-index-hashing per folder: insert n_f hashes, query each for neighbours within the small threshold (2 or 6) → target O(n_f·log n_f) typical instead of O(n_f²). # Part II — Design & architecture

## Target architecture & refactor roadmap

_Synthesized in the main loop from the design findings below, after the workflow's synthesis agent was cut off by the spend limit. Grounded in those findings; not adversarially verified._

The design findings converge on one root tension: **OpenPhoto's Core is clean and well-separated, but the App layer has accreted a single `@MainActor` god object (`AppState`, ~1,740 lines, seven cohesive subsystems, one invalidation surface) that both holds observable state and performs synchronous full-library I/O.** Almost every high-severity design finding is a symptom of that, or of the data-model decisions that make full-library work the default. The target below is the end-state; the roadmap orders it for leverage × low risk.

### End state

- **Decompose `AppState` into focused `@Observable` services** behind a thin coordinator, each owning its own published slice so a derivation tick no longer invalidates the whole UI: `LibraryStore` (browse/query state), `SearchService`, `DerivationRunner`, `DrivesCoordinator`, `PeopleService`, `CullService`, `UndoService`. This directly resolves the "single @Observable god object" and "drives/role orchestration sprawl" findings.
- **Introduce one drives *reconcile primitive*.** The ~10 role/sync/drift methods today hand-repeat the same `presence → badge → drift → queries` sequence; replace them with a single `reconcile(drive:)` that the actions call, so the sequence exists once.
- **Move the derivation runner into Core.** The stage registry, the idempotency/poke guard, and the drain loop are pure orchestration policy currently embedded in `@MainActor AppState`; they belong in Core where they're testable and off-main by construction.
- **Make the `Catalog`/IO boundary an actor (or async repository).** `LibraryService` is a plain `Sendable` class today, so `timelineItems`/`folderTree`/`reconcileFinderTags` execute their `O(N)` bodies inline on whatever actor calls them — usually `@MainActor`. An `actor` (or async API returning to the caller) makes off-main execution the only option and removes the structural main-thread hazard.
- **Reshape the browse data model.** Replace "fetch the whole non-materialized UNION and sort in RAM with no pagination" with a windowed/keyset-paginated query, and add the indexes the real predicates need (`dirPath`; the `assets` filter columns). Replace `GLOB`-recursion folder browse with a `dirPath`-prefix range or a folder closure table. Add **orphan pruning** (today `assets`/embeddings/face rows for vanished content are never removed because `replaceInstances` only reconciles `instances`).
- **Give the on-disk logs an owner with batch mutation.** `manifest.jsonl`, the bin log, and `imports.jsonl` are whole-file values mutated per-row today (`O(M)` each, `O(M²)` in loops). A `ManifestStore`/`BinStore`/`RegistryStore` with `read-once → mutate-set → write-once` (and a crash-durable atomic write — `F_FULLFSYNC` + dir fsync, per `CODE_AUDIT` H11) makes bulk operations linear and durable.
- **Lift the move/reorg use case out of SwiftUI.** Multi-drive move/format logic lives in `AppState+FolderReorg` (the View layer) today; it belongs in a Core use-case object the App just invokes, closing the Core↔App boundary leak.
- **Tighten the import abstraction.** `ImportSource` underspecifies real capabilities, so the engine and UI downcast to concrete source types. Model capabilities as typed protocols (e.g. `DeletableSource`, `MetadataSidecarSource`) so the engine stops switching on concrete types.
- **Own a single long-lived Core ML `EmbedStage`** (model + tokenizer compiled once per process), shared by the registry and the query path, instead of rebuilding it per search.

### Roadmap (prioritized)

1. **Contained, high leverage — do first.** (a) One owned `EmbedStage` shared by registry + search (kills the per-query recompile). (b) Move `refreshQueries`' heavy reads off-main and coalesce them to event-scoped invalidation. These two remove the worst 100k symptoms (search recompile + per-mutation beachball) with small blast radius and no schema change.
2. **The data-scaling backbone — medium.** Make `Catalog`/`LibraryService` async/actor; window/paginate the browse; add the missing indexes and replace `GLOB` folder recursion. This is what lets the library cross 100k smoothly.
3. **Mechanical but broad — medium.** Convert `manifest`/bin/registry to batch mutation + crash-durable writes. Removes the quadratic bulk repair/move/delete cost across Sync, reorg, and delete.
4. **Structural — larger.** Decompose `AppState` into the services above, move `DerivationRunner` to Core, and add the drives reconcile primitive. Unlocks testability and stops cross-subsystem invalidation; best done after (1)–(2) have de-risked the data path.
5. **Pipeline — targeted.** Bounded-parallel + incremental (FSEvents-driven) + resumable scan; orphan pruning.
6. **Algorithmic — as the roadmap demands 10k+ libraries.** Indexed/ANN matchers (Hamming/BK-tree for pHash, ANN/blocked clustering for faces) and the face-identity-embedding fix.

#### DZ01 — refreshQueries() is a synchronous main-actor full-rebuild fired from ~27 sites — main-thread hazard that scales with library size

- **Severity:** Critical  |  **Confidence:** Confirmed  |  **Area:** concurrency-architecture  |  **Subsystem:** AppState architecture  |  **Effort:** L
- **Location:** `Sources/OpenPhotoApp/AppState.swift:1531 (refreshQueries)`, `Sources/OpenPhotoCore/LibraryService.swift:88 (timelineSections)`, `Sources/OpenPhotoCore/LibraryService.swift:178 (folderTree)`, `Sources/OpenPhotoCore/LibraryService.swift:150 (directoriesUnder FS walk)`, `Sources/OpenPhotoApp/AppState.swift:1061 (refreshPendingDeletions)`, `Sources/OpenPhotoCore/Catalog/Queries.swift:30 (timelineSQL UNION)`
- **Problem:** refreshQueries() runs entirely on the main actor and synchronously rebuilds ALL derived browse state on every call: catalog.timelineItems() executes the full local∪drive-only UNION and materializes every row, groupedSections() is O(N), folderTree() runs catalog.folderCounts() PLUS a recursive FileManager.enumerator walk of the entire primary-vault filesystem tree (directoriesUnder), then binItems() and refreshPendingDeletions() (a loop over every connected drive issuing per-drive vaultPresenceRows/instanceHashes queries). It is invoked from ~19 in-class call sites and 8 view sites (videoOnly didSet, every move/delete/evict/rehydrate/rename/sync/drift/adopt/send/reverify path, plus onChange handlers in views). At the documented 10k+ import smoke this is a multi-hundred-ms main-thread stall per state change; it directly causes the known dense-grid Space-switch hitch and blocks the deferred tile-grid optimization. Because it is `throws` and synchronous it cannot move heavy work off-main without a signature change.
- **Proposed change:** Split into an async producer + a main-actor publisher. Introduce a `LibraryQueryService` (or method on a new query coordinator) that computes (sections, flatItems, folderTree, binEntries, pendingDeletions) inside Task.detached(.userInitiated) returning a single immutable `BrowseSnapshot` value; AppState (or a focused `@Observable BrowseModel`) assigns the snapshot on the main actor. Add coalescing (a generation token / debounce) so the ~27 triggers collapse to one rebuild per event loop, and make folderTree’s filesystem walk incremental or cached (invalidated only by folder-reorg ops, not by every delete/rename). Keep a synchronous fast path only for the create-folder case that legitimately needs no DB reload.
- **Risk:** Medium-high blast radius: every call site and the 8 view onChange handlers must move from `try? refreshQueries()` to an async/await or fire-and-forget request; making the result async introduces a window where views show stale counts, so the snapshot must be applied atomically. Mitigated by the fact that all call sites already tolerate `try?` best-effort semantics.

#### DZ02 — Browse is one non-materialized UNION view, fetched whole and sorted in RAM with no pagination

- **Severity:** Critical  |  **Confidence:** Confirmed  |  **Area:** data-model  |  **Subsystem:** Catalog, queries & search  |  **Effort:** L
- **Location:** `Sources/OpenPhotoCore/Catalog/Queries.swift:8-40`, `Sources/OpenPhotoCore/Catalog/Queries.swift:33-40`, `Sources/OpenPhotoCore/LibraryService.swift:88-95`, `Sources/OpenPhotoApp/AppState.swift:131`
- **Problem:** Every browse, folder view, search-result hydration, dedupe scan, peek, and dedupe-key build runs `SELECT * FROM (<localSelect> UNION ALL <driveSelect>) ORDER BY takenAtMs DESC` with no LIMIT/OFFSET. The UNION is an inline derived table re-parsed and re-planned at each of ~12 call sites, and its top-level `ORDER BY takenAtMs DESC` cannot use the `idx assets.takenAtMs` index because the column emerges from a derived UNION — so SQLite materializes the full union into a temp b-tree and sorts it. `timelineItems` then returns ALL TimelineItem structs to the main actor (AppState.swift:131), where `groupedSections` walks the entire array. At 100k assets this is ~100k structs + a full temp-sort per browse; at 1M it is a multi-hundred-MB allocation and a guaranteed UI hitch on every Space switch / filter change. There is no pagination anywhere in the read path. This is the documented scaling ceiling and directly blocks the 'friend's real 10k import' acceptance test growing into the intended large-library and server-consumer roadmap.
- **Proposed change:** Promote the union to a first-class browseable projection rather than re-deriving it inline. Two complementary moves: (1) Make `timelineSQL` a real SQL VIEW (or a materialized `browse_rows` table refreshed on scan-reconcile) with a covering index on `(takenAtMs DESC, kind, isLivePairedVideo)` so the timeline order is index-served, not temp-sorted. (2) Add keyset pagination to the public API: `timelineItems(before: takenAtMs?, limit: Int)` returning a page, and have LibraryService/AppState page the grid (the grid already needs windowing for the Phase-5 tile renderer). Keep the in-RAM `[TimelineItem]` only for small result sets (search hits), not for whole-library browse.
- **Risk:** Medium-High. A materialized table adds a refresh path that must stay consistent with instances/vault_presence (mitigated because the catalog is declared rebuildable). A VIEW is lower-risk but doesn't fix the temp-sort unless paired with the covering index. Pagination touches AppState's grid model and FolderGridView — a moderate blast radius but mechanical. No on-disk format change (catalog is rebuildable), so no docs/format migration cost.

#### DZ03 — Single @Observable god object: seven cohesive subsystems share one invalidation surface and one 1740-line file

- **Severity:** High  |  **Confidence:** Confirmed  |  **Area:** architecture  |  **Subsystem:** AppState architecture  |  **Effort:** L
- **Location:** `Sources/OpenPhotoApp/AppState.swift:51 (class AppState)`, `Sources/OpenPhotoApp/AppState.swift:164-528 (search state+runSearch)`, `Sources/OpenPhotoApp/AppState.swift:192-401 (people state+management)`, `Sources/OpenPhotoApp/AppState.swift:231-293 (cull state)`, `Sources/OpenPhotoApp/AppState.swift:662-1306 (drives/roles/drift ~640 lines)`, `Sources/OpenPhotoApp/AppState.swift:1453-1529 (derivation runner)`
- **Problem:** AppState owns the state and orchestration for library/query, search, people, cull, drives/roles, derivation, send, peek, import, settings toggles, and undo — all as flat stored properties on one @Observable class bound into 34 views as `@Bindable var state` with 73 distinct member accesses. Consequences: (a) publishing granularity is all-or-nothing — mutating searchResults invalidates the Timeline, Folders, People and Drives views because they all observe the same object; (b) the file is unmaintainable and every feature widens the same surface; (c) there are no testable seams — testing search ranking or cull grouping requires instantiating the whole @MainActor god object with a live LibraryService; (d) the documented roadmap (people, cull, search refinement) keeps growing this class. The subsystems are already cleanly separable: search state is self-contained (searchQuery/searchFilters/searchResults/semanticIndex/runSearch), people likewise (people/suggestedClusters/loadPeople + the 6 management methods), cull likewise, derivation likewise.
- **Proposed change:** Decompose into focused `@Observable @MainActor` models owned by AppState (which becomes a thin coordinator holding navigation/selection + the shared library handle): SearchModel, PeopleModel, CullModel, DrivesModel (see next finding), DerivationRunner, and a BrowseModel (sections/flatItems/folderTree/binEntries). Each owns its state and its orchestration methods; each is injected into its view subtree via @Environment so a search mutation only invalidates Search views. Cross-model coordination (e.g. a move that must refresh browse + presence) goes through a small set of explicit calls or an event, not shared mutable fields. Move the nonisolated sidecar helpers into a Core `PersonSidecarWriter` (see error/concurrency finding).
- **Risk:** High blast radius but mechanical: 34 views change their binding from `state.x` to `searchModel.x`/`peopleModel.x` etc., and OpenPhotoApp.swift must construct + inject the sub-models. Low correctness risk because behavior is unchanged; can be done subsystem-by-subsystem (extract SearchModel first as a vertical slice, prove the pattern, then People/Cull) so it is incrementally shippable.

#### DZ04 — Drives/role orchestration sprawl: ~10 methods hand-repeat the same presence→badge→drift→queries refresh sequence with no reconcile primitive

- **Severity:** High  |  **Confidence:** Confirmed  |  **Area:** duplication  |  **Subsystem:** AppState architecture  |  **Effort:** M
- **Location:** `Sources/OpenPhotoApp/AppState.swift:1109 (driftScan)`, `Sources/OpenPhotoApp/AppState.swift:1128 (verifyAllConnected)`, `Sources/OpenPhotoApp/AppState.swift:1188 (verifyIntegrity)`, `Sources/OpenPhotoApp/AppState.swift:1209 (autoScanConnectedDrives)`, `Sources/OpenPhotoApp/AppState.swift:763-808 (recoverCanonical/cloneToBackup)`, `Sources/OpenPhotoApp/AppState.swift:972-988 (refreshCanonicalPresence/reloadCanonicalPresence)`
- **Problem:** The drives subsystem is ~640 lines inside AppState and the role/sync/drift methods repeat a near-identical fixed sequence: DriftReconciler.scan/verify → annotateRecoverability(presenceService) → catalog.replaceVaultPresence(presenceEntries(...)) → reloadCanonicalPresence() → driveDrift[id]=report → refreshPendingDeletions(). This appears verbatim across driftScan, verifyAllConnected, verifyIntegrity, autoScanConnectedDrives, and is partially repeated in cloneToBackup/recoverCanonical/promoteToCanonical (each ends with reloadDrives(); reloadCanonicalPresence(); try? refreshQueries()). The duplication means a change to the reconcile contract (e.g. presence semantics, the snapshot-write step) must be made in ~6 places and is easy to get subtly inconsistent — a correctness risk for the canonical/backup presence sets that drive sovereignty-critical badges and deletion eligibility. It also couples AppState directly to DriftReconciler/Manifest/SyncEngine/CatalogSnapshot internals.
- **Proposed change:** Extract a `DriveCoordinator` (Core or App-side @MainActor) owning durableVaults/canonicalPresence/driveDrift/drivePendingDeletions/ejectedDrives/driveKinds and exposing intent methods (scan/verify/repair/clone/recover/promote/forget). Factor the repeated tail into one private primitive, e.g. `applyDriftResult(_ report, for: drive)` that does annotate→replaceVaultPresence→reloadPresence→store→refreshPendingDeletions, and one `reconcileAfterRoleChange()` for the reloadDrives+presence+queries tail. AppState delegates Drives-view calls to this coordinator. Move the Manifest/SyncEngine touch-points behind a Core method so the App layer stops reaching into Core sync internals.
- **Risk:** Medium: the sequence ordering is load-bearing (driftScan re-derives presence BEFORE refreshQueries reads it — see evict at line 1608), so the extracted primitive must preserve ordering exactly. Pure mechanical extraction with no behavior change; well-covered by the existing drive tests if any, otherwise needs targeted tests around the reconcile primitive.

#### DZ05 — `assets` (the hot filter table) has only a takenAtMs index; structured filters full-scan

- **Severity:** High  |  **Confidence:** Confirmed  |  **Area:** data-model  |  **Subsystem:** Catalog, queries & search  |  **Effort:** S
- **Location:** `Sources/OpenPhotoCore/Catalog/Catalog.swift:39-56`, `Sources/OpenPhotoCore/Search/Catalog+Search.swift:10-107`, `Sources/OpenPhotoCore/Catalog/Catalog+Derivation.swift:23-38`
- **Problem:** The `assets` table is indexed only on `takenAtMs` (plus the `hash` PK). But `structuredFilter` predicates on `kind`, `favorite`, `rating`, `cameraModel`, `livePairHash` and every query filters `isLivePairedVideo = 0`; `distinctCameras` scans `cameraModel`; `pendingDerivation`/`derivationProgress` filter `kind = ? AND isLivePairedVideo = 0` (+ `latitude/longitude` for geocode); `geotaggedAssets` filters `latitude IS NOT NULL`. None of these columns are indexed, so each is a full scan of `assets` — and inside the timeline union, which is itself a temp-materialized full scan. As the library grows, every facet click, every derivation-runner poll (`pendingDerivation` runs repeatedly with a LEFT JOIN on derivation_jobs), and every map open re-scans the whole table. The index coverage simply does not match the documented query surface.
- **Proposed change:** Add targeted secondary indexes matched to the actual predicates: a partial index `CREATE INDEX idx_assets_browse ON assets(takenAtMs DESC) WHERE isLivePairedVideo = 0` (serves the dominant browse + hides paired videos for free), `idx_assets_kind(kind) WHERE isLivePairedVideo = 0`, `idx_assets_camera(cameraModel)`, and a partial geo index `idx_assets_geo(takenAtMs) WHERE latitude IS NOT NULL`. For favorites/rating, a partial `idx_assets_fav(takenAtMs) WHERE favorite = 1`. Add these in a single new migration (v13) — pure index DDL on a rebuildable table, no schema/format change.
- **Risk:** Low. Index-only additive migration; bump schemaVersion to 13 and add a changelog line in docs/format/catalog-schema.md per the documentation-discipline rule. Slightly larger catalog file and marginally slower writes during bulk scan ingest (acceptable; ingest is batched in one transaction).

#### DZ06 — Folder browse/counts rely on GLOB recursion a B-tree cannot serve

- **Severity:** High  |  **Confidence:** Confirmed  |  **Area:** data-model  |  **Subsystem:** Catalog, queries & search  |  **Effort:** M
- **Location:** `Sources/OpenPhotoCore/Catalog/Queries.swift:61-136`, `Sources/OpenPhotoCore/Search/Catalog+Search.swift:41-56`, `Sources/OpenPhotoCore/Catalog/Catalog.swift:385-411`
- **Problem:** Recursive folder browse (`items(inDir:recursive:)`), the structured-filter folder facet, and `rewriteVaultPresencePaths` all match subtrees via `dirPath GLOB '<dir>/*'`. SQLite cannot use a b-tree index for a GLOB whose pattern doesn't reduce to a literal prefix range, so each recursive folder open is a full scan of the union (instances + vault_presence). `folderCounts` additionally GROUP BYs `dirPath` over both branches plus a correlated `MIN(rowid)` dedupe per drive-only row — full scan of vault_presence on every Folders-view render. The `dirPath` index that exists is only usable for the exact-match (non-recursive) case. This caps the Folders view, which is a primary navigation surface, and the rewrite-on-move path scales with total presence rows on every folder drag.
- **Proposed change:** Two options. Cheap: exploit that GLOB '<dir>/*' is a prefix range — rewrite as `dirPath >= '<dir>/' AND dirPath < '<dir>/￿'` (or the byte-successor of '/') so the existing `dirPath` b-tree serves it, keeping the Swift-side hasPrefix guard. Structural: add a `folder_depth`/materialized closure or store a normalized `pathSegments` and index a `(dirPrefix)` column, letting recursive counts be a single GROUP BY over an indexed range. Precompute `folderCounts` incrementally on scan-reconcile into a `folder_counts(dirPath, n)` cache table rather than recomputing the GROUP BY + correlated dedupe on every render.
- **Risk:** Medium. The range-rewrite is low-risk and local but must handle the collation/successor-character edge correctly (GLOB is case-sensitive, default collation is BINARY — they happen to align, but this needs a test). A folder_counts cache adds a refresh path. No format change.

#### DZ07 — SemanticIndex holds all embeddings in one fp32 RAM matrix with a full per-query sort

- **Severity:** High  |  **Confidence:** Confirmed  |  **Area:** concurrency-architecture  |  **Subsystem:** Catalog, queries & search  |  **Effort:** L
- **Location:** `Sources/OpenPhotoCore/Search/SemanticIndex.swift:6-39`, `Sources/OpenPhotoCore/Catalog/Catalog+Embeddings.swift:47-55`, `Sources/OpenPhotoApp/AppState.swift:487-519`
- **Problem:** `SemanticIndex.init` calls `allEmbeddings(model:)`, unpacking every stored Float16 vector to Float and concatenating into one contiguous `[Float]` of count×dim. For a typical CLIP dim of 512–768, fp32: at 100k assets that is 100k×768×4 ≈ 295 MB resident (≈590 MB at dim 768/100k if 1024-dim); at 1M assets ≈ 3 GB — on a Mac that already OOM'd to 41 GB during indexing (per project memory). The full index is rebuilt from scratch whenever `semanticIndexDirty` (AppState.swift:502-504), re-reading and re-unpacking the entire embeddings table on the search task. `query` then does a full vDSP_mmul over all rows and an O(N log N) `.sorted()` of the whole score array to take topN — no partial-select, no ANN. This is a brute-force design that is fine at a few thousand assets and unusable at the roadmap's target scale, and the full-rebuild-on-dirty pattern means every new embedding invalidates and re-materializes hundreds of MB.
- **Proposed change:** Decouple the index abstraction from the brute-force impl behind a `SemanticSearcher` protocol so the storage strategy can evolve without touching callers. Near-term: keep vectors as Float16 in RAM (halves footprint) and convert to fp32 only inside the vDSP call, and replace the full `.sorted()` with a bounded top-N selection (partial heap / `vDSP` + a k-selection) — O(N) instead of O(N log N). Medium-term: make the index incremental (append new rows instead of full rebuild on dirty) and add an on-disk ANN tier (e.g. an IVF/HNSW sidecar or sqlite-vec) so 1M vectors don't need to be resident. The protocol seam lets the brute-force path remain the small-library fallback.
- **Risk:** Medium. The Float16-resident + partial-select change is low-risk and local to SemanticIndex. Incremental updates require coordinating with AppState's dirty flag. An on-disk ANN tier is a larger dependency decision; gate it behind the protocol so it's additive. Embedding blob format is unchanged (already Float16 LE) so no format-doc change.

#### DZ08 — FaceClusterer uses greedy single-link agglomeration → transitive chaining merges distinct people

- **Severity:** High  |  **Confidence:** Confirmed  |  **Area:** data-model  |  **Subsystem:** Cull & faces (clustering/dedup)  |  **Effort:** M
- **Location:** `Sources/OpenPhotoCore/Faces/FaceClusterer.swift:43-60`, `Sources/OpenPhotoApp/AppState.swift:217`
- **Problem:** cluster() joins an item to the FIRST existing cluster where ANY member is within threshold (single-link, FaceClusterer.swift:47-49). Single-link clustering is the textbook 'chaining' failure mode: if A~B and B~C but A is far from C, A, B, C all collapse into one cluster. With faces of the same gender/lighting/pose this transitively bridges different people into one giant blob, while a tight threshold over-fragments the same person across pose changes. There is no centroid, no average/complete linkage, and no notion of cluster cohesion. Order-dependence compounds it: the FIRST matching cluster wins and clusters never split or re-merge, so result quality depends on input row order. This is a correctness defect in the headline user-facing feature (People suggestions), and it caps cluster quality regardless of how the 0.4 threshold is tuned.
- **Proposed change:** Replace greedy single-link with a centroid-based or density-based scheme. Minimal: maintain a running unit-mean centroid per cluster and assign each face to the nearest centroid within threshold (k-means-like / online centroid linkage) — kills chaining, still O(n·k). Better target: factor a small `Clustering` abstraction with a pluggable `Linkage`/`ClusterModel` strategy (centroid now, HDBSCAN/DBSCAN later) so the algorithm is swappable behind FaceClusterer.cluster's stable signature. Keep the deterministic largest-first ordering. Add adversarial chaining tests (A~B~C with A far from C must NOT co-cluster).
- **Risk:** Medium. Signature (items, threshold)->[[Int64]] is unchanged and only AppState.loadPeople and tests call it, so blast radius is contained. Risk is behavioral: thresholds tuned for single-link won't transfer to centroid linkage, so the 0.4 default must be re-tuned and snapshot tests updated. No persisted data migrates (clusters are recomputed each load).

#### DZ09 — All matchers are brute-force O(n²) with no spatial/ANN index — blocks the 10k+ library roadmap

- **Severity:** High  |  **Confidence:** Confirmed  |  **Area:** concurrency-architecture  |  **Subsystem:** Cull & faces (clustering/dedup)  |  **Effort:** L
- **Location:** `Sources/OpenPhotoCore/Faces/FaceClusterer.swift:43-53`, `Sources/OpenPhotoCore/Cull/DuplicateGrouper.swift:18-23`, `Sources/OpenPhotoCore/Cull/BurstGrouper.swift:13-22`
- **Problem:** Every similarity matcher does full pairwise comparison. FaceClusterer is O(n·k) clusters but each membership test scans ALL members (single-link), degenerating to O(n²) cosine over 512-dim Float when faces collapse into few large clusters — exactly the chaining case. DuplicateGrouper is O(m²) Hamming within each folder bucket (DuplicateGrouper.swift:18-23); one big folder (a 5k-photo dump) is 12.5M comparisons. BurstGrouper is linear (only adjacent pairs) so it is fine. There is zero ANN/LSH/sort-prune infrastructure anywhere in Sources (grep confirms). loadCullGroups/loadPeople load the ENTIRE embeddings/phash table into memory and run this single-threaded inside one Task.detached — for the documented 10k import this is the same memory-pressure/CPU-hog class of problem already flagged for indexing. This is the explicit roadmap blocker (friend's real ~10k import is the acceptance test).
- **Proposed change:** Introduce a thin nearest-neighbour layer rather than scattering loops. For dHash dedup: BK-tree or multi-band LSH on the 64-bit hash to prune candidates before Hamming (turns O(m²) into ~O(m log m)). For faces: blocking/canopy via LSH buckets on the embedding before exact cosine, so clustering only compares within candidate buckets. Encapsulate as a `NearestNeighborIndex` protocol in Core with a trivial brute-force impl (current behavior) and an LSH/BK-tree impl, selected by N. Stream from the catalog in chunks instead of materializing the whole table. Keep the pure grouper signatures; they consume candidate pairs from the index.
- **Risk:** Medium-High. New abstraction and the most algorithmically involved change, but it sits behind the existing pure-function boundary so callers (AppState) and the on-disk format are untouched. Main risk is recall regressions from approximate pruning — mitigate by keeping brute-force as the default below a size threshold and as the test oracle.

#### DZ10 — Face identity clustered on a generic scene feature-print, and named people are ignored during clustering

- **Severity:** High  |  **Confidence:** Confirmed  |  **Area:** abstraction  |  **Subsystem:** Cull & faces (clustering/dedup)  |  **Effort:** M
- **Location:** `Sources/OpenPhotoCore/Derivation/FaceStage.swift:49-71`, `Sources/OpenPhotoCore/Catalog/Catalog+Faces.swift:112-123`, `Sources/OpenPhotoApp/AppState.swift:208-228`
- **Problem:** FaceStage runs VNGenerateImageFeaturePrint on a padded face CROP (FaceStage.swift:49,56-71). VNGenerateImageFeaturePrint is a general-purpose image similarity descriptor, NOT a face-identity embedding (it is not trained to be invariant to pose/lighting/expression for the same identity, and not discriminative across identities). The clustering quality ceiling is therefore set by the wrong embedding, independent of the linkage fix. Separately, the clustering pipeline only ever consumes unassignedFacesWithEmbeddings() (personID IS NULL AND source='auto') — it NEVER compares an unassigned face against existing named people's confirmed faces (Catalog+Faces.swift:112-123, AppState.swift:216-217). So once you name 'Alice', new photos of Alice are clustered only among themselves and re-suggested as an unknown blob, instead of being proposed as 'likely Alice'. The abstraction conflates 'visual look-alike grouping' with 'identity recognition' and only implements the former.
- **Proposed change:** Two-part. (1) Treat the embedding source as a named, versioned policy: add a face-embedding model identifier alongside the vector (mirror the embeddings table's `model` column, which faces lacks) so a future real face-recognition model can supersede VNGenerateImageFeaturePrint via the same reconcile/re-derive path. (2) Add a recognition step distinct from cold-start clustering: a `assignToNearestPerson(face, people, threshold)` that compares each unassigned face to per-person centroids of confirmed faces, surfacing 'add to <Person>?' suggestions; reserve agglomerative clustering for the genuinely-unknown remainder. This splits the conflated abstraction into 'cluster unknowns' vs 'recognize against known'.
- **Risk:** Medium. The recognition step is additive and self-contained (new query + new pure function), low blast radius. Swapping the embedding model is the bigger move: it forces a re-derivation of all faces and a format-doc update (per CLAUDE.md, the faces embedding is machine-derived and rebuildable, so no user-data risk, but the catalog-schema doc must record the model column).

#### DZ11 — EmbedStage model + tokenizer rebuilt per search query (no shared, owned model lifetime)

- **Severity:** High  |  **Confidence:** Confirmed  |  **Area:** concurrency-architecture  |  **Subsystem:** Derivation / ML pipeline  |  **Effort:** S
- **Location:** `Sources/OpenPhotoApp/AppState.swift:510`, `Sources/OpenPhotoApp/AppState.swift:1456`, `Sources/OpenPhotoCore/Derivation/EmbedStage.swift:44`, `Sources/OpenPhotoCore/Derivation/EmbedStage.swift:67`, `Sources/OpenPhotoCore/Derivation/EmbedStage.swift:95`, `Sources/OpenPhotoCore/Derivation/EmbedStage.swift:104`, `Sources/OpenPhotoCore/Derivation/CLIPTokenizer.swift:29`

#### DZ37 — Three sibling transfer engines with no shared copy-verify-record-journal abstraction

#### DZ38 — Lossy error model: Bool/optional-try swallow the reason exactly where the format spec wants it recorded

- **Severity:** Medium  |  **Confidence:** Confirmed  |  **Area:** error-model  |  **Subsystem:** Sync & send engines  |  **Effort:** M
- **Location:** `Sources/OpenPhotoCore/Sync/VerifiedCopy.swift:10 (returns Bool, no reason)`, `Sources/OpenPhotoCore/Sync/SyncEngine.swift:33-41 (conflict 'needs a reason field, out of scope')`, `Sources/OpenPhotoCore/Sync/SyncEngine.swift:159-171 (failed item with no cause)`, `Sources/OpenPhotoCore/Sync/SyncPlan.swift:21 (SyncResult.failed is [PlanItem], no reason)`, `Sources/OpenPhotoCore/Send/SendEngine.swift:49-51 (??-collapses any throw to 'send failed')`
- **Problem:** The transfer layer collapses every distinct failure cause into a Boolean or a swallowed optional-try. #### DZ39 — The read-append-atomic-rewrite pattern is duplicated across five append-only JSONL files instead of one shared append-log abstraction

- **Severity:** Medium  |  **Confidence:** Confirmed  |  **Area:** duplication  |  **Subsystem:** Vault format, IO, sidecars  |  **Effort:** M
- **Location:** `Sources/OpenPhotoCore/Sync/SyncLog.swift:5`, `Sources/OpenPhotoCore/Vault/BinStore.swift:70`, `Sources/OpenPhotoCore/Import/ImportRegistry.swift:78`, `Sources/OpenPhotoCore/Send/SendRegistry.swift:81`, `Sources/OpenPhotoCore/Send/DeviceRegistry.swift:67`

#### DZ40 — AtomicFile's durability contract omits the parent-directory fsync the format spec mandates, weakening crash-consistency of the manifest rename on passive drives

- **Severity:** Medium  |  **Confidence:** Needs verification  |  **Area:** error-model  |  **Subsystem:** Vault format, IO, sidecars  |  **Effort:** S
- **Location:** `Sources/OpenPhotoCore/IO/AtomicFile.swift:6`, `Sources/OpenPhotoCore/IO/AtomicFile.swift:21`, `docs/format/vault-format-v1.md:87`
- **Problem:** AtomicFile fsyncs the temp file's data (fh.synchronize, line 14) then calls FileManager.replaceItemAt (line 21). The data is durable but the *rename* (the directory entry change that makes the new manifest visible and removes the temp) is not fsync'd to the containing directory. The format doc §4/§10 specifies 'temp file -> fsync -> rename' as the atomicity contract third-party writers rely on, and the whole drive model is 'passive drives, crash-consistent after a sync'. On a power loss right after a sync rewrites manifest.jsonl on a removable drive, the directory metadata may not have reached stable storage, so the manifest could roll back to the pre-rename version while the copied originals are already on disk — exactly the 'manifest behind reality' drift the design treats as recoverable, but avoidable. replaceItemAt's own durability guarantees are also filesystem-dependent and undocumented in this code. This is the one place where 'atomic' is asserted but not fully delivered, and it is the single choke point every state write flows through, so the gap is systemic.
- **Proposed change:** After the rename, open the destination's parent directory and fsync it (open(dir, O_RDONLY); fcntl/fsync; close), or use the lower-level rename(2)+fsync-dir sequence rather than replaceItemAt, and document the realized guarantee in a comment that cites format §10. Optionally expose a durability level (e.g. .barrier vs .relaxed) so hot-path catalog-adjacent writes can opt out while manifest/vault.json always barrier.
- **Risk:** Low. Localized to one ~15-line function; no format or call-site change. Verify the directory-fsync actually fires on the target filesystems (APFS/exFAT on removable drives) since exFAT semantics differ; that verification is why confidence is 'Needs verification'.

# Part III — Tile system optimization plan

_Synthesized in the main loop from the tile pipeline trace, the GPU-cost diagnosis, the structured redesign options, and the adversarial feasibility check (all of which completed); the workflow's final plan-writing agent was cut off by the spend limit. The supporting analyses are reproduced in Appendix C._

### Bottleneck

At small tile sizes (minimum zoom, many hundreds of cells on screen) the dominant cost is **the number of GPU-textured `CALayer`s and offscreen-rendering passes the WindowServer must composite**, not VRAM and not decode. Texture sizing is already correct — `ThumbnailStore.displayImage` downsamples on decode to the bucketed `gridThumbnailPixels`, so tiny tiles hold tiny images, and estimated VRAM (~90–130 MB at min zoom) is comfortably within unified memory. The hitch comes from two compounding sources: (1) **per-tile chrome that forces offscreen passes** — every `MediaTile` has a rounded-rect clip mask and the selection ring / checkmark / per-glyph badge shadows in `TimelineTileBadges`, each an offscreen pass, and these scale 1:1 with the tile count that min zoom maximizes; and (2) **`LazyVGrid` keeps every scrolled-past row's image layers alive** until far offscreen, so a dense grid accumulates hundreds-to-thousands of `IOSurface`-backed layers. A macOS **Space switch recomposites the whole window**, so it must re-capture that entire layer set at once — the ~1 s hitch. Confidence is high that this is pass/layer count, not VRAM (the diagnosis traced the actual masks/shadows; a plain-color-tile control composites smoothly).

### Plan (ordered: quick wins → structural change → fallback)

**Step 1 — Chrome diet + cost-bounded cache (effort S, low risk, ship now).** Two independent quick wins, no architecture change:

## Current tile pipeline

**Catalog → AppState.** `AppState.refreshQueries()` runs `library.timelineSections(...)` (the `timelineSQL` UNION of local instances + drive-only presence rows) and stores `sections: [TimelineSection]` plus `flatItems = sections.flatMap(\.items)`. A `TimelineItem` (Records.swift:86) is a small Codable metadata value — hash, kind, dims, paths, flags — **no pixel data**. So the catalog/AppState layer is cheap and unrelated to the rendering cost; it just hands an array of value structs to the grid.

**Grid (LazyVGrid).** `TimelineView.grid` is a `ScrollView { LazyVStack(pinnedViews: [.sectionHeaders]) { ForEach(sections) { Section { LazyVGrid(columns: [.adaptive(minimum: state.gridMinSize)], spacing: 3) { ForEach(section.items, id: \.instanceID) { cell($0) } } } } } }`. `FolderGridView` is the same shape with one `LazyVGrid` over a flat `items` array. The grid is column-adaptive on `gridMinSize` (slider/pinch range 48...220, default 132, persisted to `UserDefaults`). At min zoom (`gridMinSize` ≈ 48–120) the adaptive layout packs many columns; on a wide window this is easily 10–16 columns and, with the `LazyVGrid`'s overscan, **several hundred cells realized at once** (the prompt's ~400 is realistic for a tall window at the smallest cells).

**Per cell.** `cell(item)` builds a `MediaTile` wrapping a `ThumbnailImage(timelineItem:library:targetPixel:)`. `MediaTile` (MediaTile.swift) is itself a deep view: `Color.clear` → `.aspectRatio(1)` → `.overlay { thumbnail }` → `.clipShape(RoundedRectangle)` → `.overlay { badges() }` → a conditional selection-ring `.overlay` → a conditional checkbox `.overlay(alignment:.topLeading)` → `CellFrameIfNeeded` modifier → `.contentShape` → `.onTapGesture`. So **each cell is ~6 nested overlays plus the badge ZStack** (`TimelineTileBadges` is a `ZStack` of up to 3–4 SF-Symbol `Image`s with `.shadow`). Every cell also evaluates `state.isBackedUpOnCanonical(item)` on the main actor during body build.

**ThumbnailImage.** Identity is `id = instanceID`; the in-memory cache key is `"\(id)@\(targetPixel)"` (NSString). Body:
- `Theme.tile` placeholder `Color`, then if a decoded `CGImage` is available, `Image(decorative: cached, scale: 1).resizable().aspectRatio(contentMode: .fill)`, all inside `.frame(maxWidth:.infinity, maxHeight:.infinity).clipped()`.
- It reads `tileMemoryCache` **synchronously in `body`** (`tileMemoryCache.object(forKey: cacheKey)?.image ?? asyncImage`) so a recycled cell paints instantly.
- `.task(id: cacheKey)`: on a hit, set `asyncImage` and return; on a miss, clear `asyncImage` (anti-reuse guard) and run `await Task.detached(priority: .userInitiated) { await load(px) }`, then store into the NSCache and publish. `cacheKey` includes `targetPixel`, so **changing zoom changes the task id and re-runs the loader for a different pixel size.**

**ThumbnailStore (provider).** The provider closure (`init(timelineItem:)`) calls `lib.thumbnails.displayImage(for:sourceURL:kind:maxPixel:px)` (falling back to `cachedDisplayImage`). `displayImage` (ThumbnailStore.swift:45) ensures the canonical 512px content-addressed JPEG exists on disk (`<cacheDir>/<hex[0..2]>/<hex>.jpg`), then **decodes that JPEG via `CGImageSourceCreateThumbnailAtIndex` with `kCGImageSourceThumbnailMaxPixelSize = min(maxPixel, 512)`**. So the on-disk thumb is a fixed 512px JPEG; per request it is re-decoded/downsampled to the requested `maxPixel`. This produces a fresh `CGImage` at roughly `targetPixel` on the long edge.

**CGImage → GPU.** `Image(decorative: cgImage)` hands that `CGImage` to SwiftUI; the compositor uploads it as a **CALayer-backed GPU texture** sized to the decoded `CGImage` (≈ `targetPixel`²), not to the on-screen cell. There is one such texture per visible tile, plus separate layers for each badge symbol, the rounded-rect clip, and (in select mode) the ring/checkbox.

**Sizing.** `thumbPixels = gridThumbnailPixels(forCellMin: state.gridMinSize)`: `needed = Int(minSize * 4)`, bucketed up to the next multiple of 64, clamped to `[128, 512]`. The `×4` is a **hardcoded "2× retina × 2× adaptive-growth" worst case** — there is no `@Environment(\.displayScale)` / `NSScreen.backingScaleFactor` read anywhere in the app (confirmed by grep). So at `gridMinSize = 48`, `needed = 192 → bucket 192`; at 80 → 320; at 100 → 400; at 120 → 480; at ≥128 → capped 512. The bucket-to-64 quantization is what keeps the slider from re-decoding on every tick.

## Where the cost is at small sizes

**1. View / layer / texture count.** At min zoom with ~400 cells visible, the realized SwiftUI view count is large: each `MediaTile` is ~6 nested overlays + a badge `ZStack` of 2–4 `Image`s with shadows, plus the `ThumbnailImage` ZStack (placeholder + image) — call it **~12–16 views per tile, so ~5,000–6,000 SwiftUI views**, each with its own `body` evaluated on the main actor. More importantly there is **one distinct GPU texture per tile** for the photo, and that texture is **oversized for the cell**: at `gridMinSize = 48–120` the cell draws at ~48–120pt but the decoded `CGImage` is 192–512px on its long edge. Even accounting for ~2× retina backing, the `×4` factor means the texture is roughly **2× larger per axis (≈4× the pixels / VRAM)** than the cell actually needs at the small end. At 400 tiles a 512px RGBA texture is ~1 MB each → on the order of **hundreds of MB of GPU texture memory live at once** at the smallest cells where the `×4` rule still resolves high (≥128pt → 512), and ~150 MB even at 320px. This is the per-image-texture pressure the memory note already root-caused as the Space-switch hitch.

**2. Decode size mismatch — the core inefficiency.** `targetPixel` is **not** matched to on-screen points × backing scale; it is `cellMin × 4`, an intentional worst-case over-provision. For a dense grid of tiny tiles this is the dominant waste: every tile decodes a 192–512px image to fill an ~48–120pt cell. The decode itself (`CGImageSourceCreateThumbnailAtIndex` off the 512px JPEG) is cheap-ish per call but happens **once per visible tile on a cache miss**, on `.userInitiated` detached tasks with no concurrency cap — so a fresh scroll/zoom can fan out hundreds of simultaneous JPEG decodes competing for CPU and the ImageIO subsystem, then hand hundreds of oversized textures to the compositor in a burst. The on-disk store is fine; the cost is paid in (a) GPU upload of textures larger than needed, and (b) the decode fan-out.

**3. NSCache behaviour.** `tileMemoryCache` is a single app-wide `NSCache<NSString, TileImageBox>` with `countLimit = 6000` and **no `totalCostLimit` — it is bounded by count, not bytes.** With 6000 entries each potentially a 512px CGImage (~1 MB), the cache can hold ~hundreds of MB to ~6 GB of decoded images before count-eviction kicks in. The key is `"id@targetPixel"`, so **key cardinality multiplies by the number of distinct pixel buckets visited**: zooming the slider across its range steps through buckets 128/192/256/320/384/448/512 — each is a *new key for the same photo*, so zooming in then out **does not reuse** the prior decode; it re-decodes at the new bucket and inserts a new entry, inflating cache occupancy and forcing re-decodes on the way back. Within a single zoom level, scrolling back over previously-seen tiles is a hit (good). The `×4`+bucketing design deliberately trades cache-key churn (a few buckets) for not re-decoding on every slider tick, but each crossed bucket boundary is a full re-decode + re-upload of every visible tile.

**4. Work on Space-switch / scroll / zoom.**
- **Scroll:** `LazyVGrid` realizes/derealizes cells. A newly-realized recycled cell reads the NSCache synchronously in `body` (instant if hit). A miss schedules a detached decode. No synchronous decode on the main thread, but the **main-actor body rebuild of thousands of nested overlay views** and the burst of texture uploads are the scroll cost.
- **Zoom (slider/pinch):** changing `gridMinSize` (a) re-lays-out the adaptive grid (column count changes) and (b) recomputes `thumbPixels` → changes every tile's `cacheKey`/`.task(id:)`. Crossing a 64px bucket boundary **invalidates the task id for every visible tile**, triggering a full re-decode + texture re-upload wave; staying within a bucket only re-lays-out. Pinch fires many `MagnifyGesture.onChanged` updates, each mutating `gridMinSize` and thus re-evaluating the whole grid body.
- **Space-switch:** switching to a dense grid re-realizes hundreds of cells; the synchronous cache read means it paints from cache if warm, but on a cold cache (or a freshly-crossed zoom bucket) it's hundreds of simultaneous decodes plus hundreds of oversized-texture uploads in one frame — the known min-zoom hitch.

**Net at ~400 tiles, min zoom:** ~400 oversized GPU textures (192–512px each, ~4× the pixels the cell needs at the small end; ~150–400 MB VRAM), ~5–6k main-actor SwiftUI views, a single count-bounded (not byte-bounded) shared NSCache whose key includes the pixel bucket (so zoom churns it), and decode/upload bursts on every scroll into cold cells, every Space-switch, and every zoom step that crosses a 64px bucket. The two highest-leverage costs are (1) **texture size driven by the hardcoded `cellMin × 4`** rather than the actual cell-points × real backing scale, and (2) the **unbounded-by-bytes, per-pixel-bucket cache key** that defeats reuse across zoom and lets decoded-image memory grow with count rather than VRAM/RAM budget.

# GPU / Compositing Diagnosis: OpenPhoto Photo Grid at Small Tile Sizes

## Verdict on the per-image-texture diagnosis

**Confirmed, with one important refinement.** Each visible tile does become its own GPU-backed `CALayer`, and at minimum zoom you have hundreds of them — so the suspected root cause is real. But the *dominant* cost is not raw texture count or VRAM; it is the **per-tile offscreen render passes** that the tile chrome forces (rounded-rect clip + drop-shadowed overlays), multiplied by hundreds of layers, and re-evaluated when the window recomposites on a Space switch. Texture *count* is the secondary cost; offscreen passes are the headline.

## How a SwiftUI `Image`/`NSImage` in a `LazyVGrid` maps to layers and textures

SwiftUI on macOS renders through AppKit/Core Animation. Every leaf that produces pixels — each `Image(decorative:)`, each rounded-rect stroke, each SF Symbol badge, each `.shadow`, each `Theme.tile` fill — is realized as a `CALayer` (or a layer that the render server uploads as a texture). In this codebase the grid leaf is `MediaTile`, and a single tile is **not one layer** — it is a stack:

- `Color.clear` base + `Theme.tile` placeholder fill (`ThumbnailImage`'s `ZStack`)
- the decoded `Image(decorative: cached)` → its own GPU texture (the actual photo bitmap)
- `.clipShape(RoundedRectangle(cornerRadius: 3))` → a **mask** on the content
- `.overlay { badges() }` → `TimelineTileBadges`, a `ZStack` of up to 3–4 SF Symbol image layers, **each with `.shadow(radius: 2)`**, plus possibly a `Capsule().background(.black.opacity(0.45))` translucent layer
- in select mode: a `RoundedRectangle().strokeBorder` selection ring layer + a `checkmark.circle.fill` symbol layer **also `.shadow(radius: 2)`**

So even at rest a browsing tile is roughly **4–7 layers**, and 2–4 of those carry a drop shadow or a rounded-rect mask.

The decoded photo itself is a texture sized by `gridThumbnailPixels(forCellMin:)`: `cellMin × 4`, bucketed to 64, capped at 512px. At the slider/pinch floor (`gridMinSize` 48; range `48...220`, default 132) that is `48 × 4 = 192 → bucket 192px`. Good news: the texture is *correctly downsized* for small cells (this is the deliberate mitigation documented in `ThumbnailStore.displayImage`), so VRAM per photo is modest. The problem at min zoom is **multiplicity of layers and passes**, not oversized textures.

## Why hundreds of small layers are materially worse than a few large ones

Three compounding reasons, all visible in this code:

1. **Offscreen passes don't amortize.** A `.clipShape(RoundedRectangle)` over bitmap content, and especially `.shadow(...)` (which must render the source into an offscreen buffer, blur it, then composite), are per-layer offscreen render passes. One big image with one rounded mask = one pass. 600 tiles × (1 mask + ~3 shadowed badges) ≈ **a few thousand offscreen passes per recomposite**. Offscreen-pass count scales with tile count, and small tiles maximize tile count on screen. This is the single biggest lever here.

2. **Compositor bookkeeping is per-layer, not per-pixel.** The render server walks the layer tree, computes dirty regions, allocates/binds each backing store, and issues draw calls per layer. Hundreds of tiny layers means hundreds of draw calls and hundreds of backing-store bindings to cover the *same screen area* that a handful of large layers would cover with a handful of draws. GPU state changes and draw-call overhead, not fill-rate, dominate.

3. **A Space switch re-composites the entire window from its layer tree.** When macOS animates between Spaces it captures and re-renders window contents; the whole visible layer tree is re-evaluated and the offscreen passes (shadows, masks) are re-run, all inside the ~16ms frame budget of the Space transition. With a few large layers this is trivial; with ~600 tiles × multiple shadowed/masked sublayers it blows the budget — which is exactly the reported "hitch at minimum zoom." The denser the grid, the worse the transition, matching the symptom precisely. (`wantsLayer`/explicit layer handling exists only in the viewer/player, not the grids — confirmed via grep; the grids are pure SwiftUI layer-backed.)

## Textures resident and approximate VRAM at min zoom

Take min zoom (`gridMinSize` ≈ 48, texture ≈ 192px) on a large window showing ~600 visible tiles (your 400–800 estimate):

- **Photo textures:** ~192×192 px, RGBA8 ≈ 192·192·4 ≈ **147 KB** each. 600 tiles ≈ **~86 MB** resident for visible photo bitmaps. (`LazyVGrid` only materializes near-visible cells, so off-screen rows aren't textured, but `tileMemoryCache.countLimit = 6000` keeps *decoded CGImages* alive in RAM — a separate, larger memory pool, not VRAM.)
- **Chrome textures / offscreen buffers:** placeholder fill + clip mask + per-badge shadow buffers. These are small individually but there are several per tile, so **layer/backing-store count is in the low thousands** even though their VRAM is well under the photo total.
- **Total VRAM at min zoom: roughly 90–130 MB.** That is *not* alarming on Apple-silicon unified memory — which is the proof that **VRAM is not the bottleneck**. The bottleneck is the per-frame compositing/offscreen-pass work that must complete during the Space-switch animation.

(Note the downsizing only helps the photo bitmap. If the grid handed the compositor full 512px textures at min zoom, photo VRAM would jump ~7× to ~600 MB — `ThumbnailStore`/`gridThumbnailPixels` already prevent that, so don't "fix" texture size; it's the one thing already right.)

## Anti-patterns present in this code (ranked by impact)

1. **Per-badge drop shadows.** `TimelineTileBadges.glyph` uses `.shadow(radius: 2)` on every presence/favorite glyph; the select-mode checkmark uses `.shadow(radius: 2)`; `MapView` pins too. Each shadow is an offscreen blur pass per tile. At hundreds of tiles this is the dominant recomposite cost. **Highest-value fix:** drop shadows on grid badges entirely at small sizes, or replace with a cheap non-blurred contrast treatment (e.g. a `.black.opacity` capsule background, which the video/livephoto capsules already do without a shadow).
2. **Always-present rounded-rect clip mask per tile.** `MediaTile`'s outer `.clipShape(RoundedRectangle(cornerRadius: 3))` is a mask on bitmap content for every cell. At `cellRadius: 3` on a 48px tile the rounding is nearly invisible but still forces a masked composite. **Fix:** at small cell sizes, skip the rounded clip (square tiles) — removes one mask pass per tile for free visually.
3. **Multiple translucent overlay layers per tile** (capsule backgrounds at `0.45` opacity, the `0.12` rubber-band fill) add blended layers, though these are cheap relative to shadows.
4. **Badge layer count is unconditional.** `TimelineTileBadges` builds its `ZStack` of glyphs even when tiles are 48px and the glyphs are illegible/invisible. **Fix:** gate badge rendering on a min cell size (e.g. hide all glyphs below ~72px) to collapse 3–4 sublayers per tile down to just the photo + clip.

## Will `.drawingGroup()` / `Canvas` / rasterization help?

- **`.drawingGroup()` on each tile: net negative.** It would flatten one tile's substack into a single Metal-rendered texture, but it *forces* an offscreen pass per tile (the thing you're trying to reduce) and breaks `LazyVGrid` cell recycling / async image-in semantics. Don't apply per tile.
- **`.drawingGroup()` on the whole grid: wrong tool.** It would rasterize the entire (tall, scrolling, lazily-materialized) grid into one giant offscreen buffer, defeating `LazyVGrid` virtualization and ballooning memory. Avoid.
- **Layer rasterization (`shouldRasterize`): not directly reachable** from this SwiftUI code and would just cache the same expensive composite; it caches results but doesn't reduce the first paint or the Space-switch re-capture meaningfully here.
- **Fewer-larger-layers is the right direction, achieved by reducing per-tile chrome** (items 1–4 above), not by collapsing tiles into one layer. Concretely: kill badge shadows, drop the rounded mask at small sizes, and hide badges below a threshold. That cuts the offscreen-pass count by ~3–4× with zero architectural change.
- **The "real" fix the memory note already anticipates** — a single tiled/`Canvas`-style renderer that draws all visible thumbnails into one (or a few) large layers instead of one layer-stack per photo — is the correct Phase 5 architecture. It is the only thing that makes layer/pass count independent of tile count, which is what a Space switch actually stresses. (Matches the deferred "tile-grid optimization" roadmap item.)

## Confidence

- **High** that the symptom is offscreen-pass + layer-count cost during window recomposite, not VRAM exhaustion: the code shows per-tile shadows + masks + multi-layer badge stacks, texture sizing is already correctly downscaled, and computed VRAM (~90–130 MB) is comfortably within unified memory.
- **High** that removing per-badge `.shadow` and the small-size rounded mask, and gating badges by cell size, will materially reduce the hitch — these are textbook offscreen-pass sources and they scale 1:1 with the tile count that min zoom maximizes.
- **Medium** on exact texture/VRAM figures (depends on real window size, Retina backing scale of the layer backing stores, and exact visible-cell count); the order of magnitude is firm, the precise MB is an estimate.

The overlay is not "thin" — it has to reconstruct nearly the full interactive surface. Going feature by feature:

## What C preserves cleanly

- **Async thumbnail fill-in + `.task(id: cacheKey)` swap.** Preserved at the data layer. C feeds the same `provider` / `ThumbnailStore` / `tileMemoryCache` / `gridThumbnailPixels`. But note the *consumption* changes shape: today each `ThumbnailImage` owns a `@State asyncImage` and a `.task(id:)` keyed on `"\(id)@\(targetPixel)"`. A Canvas plane cannot hold per-cell `@State` or run N `.task`s — Canvas's draw closure is synchronous and pulls from a cache only. So C must hoist all N loads into a parent-owned, observable dictionary (`[id: CGImage]`) plus N detached load tasks managed by hand, and invalidate the Canvas when any entry lands. That is a real rewrite of the load/observe loop, not a reuse. The "no nil→tile→async flash on recycle" guarantee and the **reuse-guard** (`key == cacheKey` check that fixed the "same photo on 3 map pins" bug, ThumbnailImage.swift:49) must be re-implemented in that parent model or the bug returns. Doable, but it is net-new concurrency code, contradicting "reuses ThumbnailStore/tileMemoryCache/the provider unchanged" — the *store* is unchanged; the *view-side loader* is replaced.

- **Selection ring + selected styling.** Preserved, and arguably C's natural strength: the ring is a cheap stroked `RoundedRectangle` (MediaTile.swift:24-28) that belongs in the overlay, not the image plane. Fine.

- **Keeper/other badges.** The badge sets (`TimelineTileBadges`, `keeperBadge`, `confidenceBadge`) are SF Symbols + capsules — these stay as overlay views. Fine. Caveat: badges currently render *on the same rounded outer frame* as the image (the explicit fix noted in MediaTile.swift:4-7 — "corners never poke past the ring and badges are never clipped"). In C the image is in the Canvas and the badge is in a separately-positioned overlay, so badge/ring/clip alignment must be reproduced from computed rects. Pixel-exact alignment across the slider's continuous zoom is now your responsibility, not the layout engine's.

## What C makes harder

- **Rubber-band marquee.** This is the crux and the recommendation hand-waves it ("the overlay can publish cellFrame or use computed rects"). Today the marquee works via `CellFramesKey` preference aggregation, but **crucially only while `active` (select mode is on)** — `cellFrame(...)` installs the per-cell `GeometryReader` *only* when selecting (SelectionUI.swift:17-26, and the explicit comment: "When browsing, the per-cell GeometryReader ... is gone entirely"). `updateDrag` then intersects `dragRect` against `frames[id]` (SelectionModel.swift:81-92). With a Canvas plane there are **no per-cell views to attach a GeometryReader to**, so "publish cellFrame" is not available — you *must* go the "computed rects" route. That means the overlay has to compute each cell's frame analytically from the `LazyVGrid` adaptive-column geometry (column count derived from container width and `state.gridMinSize`, row offsets, `Theme.gridGap`, section headers, the Cleanup horizontal scrollers). That math is non-trivial and must stay bit-exact with whatever lays out the visible cells, across pinch-zoom, four grouping modes, and pinned section headers — or marquee selection silently selects the wrong tiles. This is the rubber-band/`CellFramesKey` plumbing the recommendation claims C "avoids"; in reality C *replaces* it with a harder, hand-rolled coordinate model. Calling this "avoided" is the recommendation's weakest claim.

- **Drag-and-drop via `PhotoMovePayload`.** Currently `.draggable(dragPayload(for:))` is attached per-tile and only in Folders' drag-to-move mode (FolderGridView.swift:107). `.draggable` needs a real view with a hit region and supplies the system drag image automatically. In C the photo is a Canvas pixel, not a view, so the per-cell `.draggable` must live on the overlay cell. That works only if the overlay actually instantiates a hit-testable view per *visible* cell (so it's not purely a redraw-on-Canvas overlay — it's N lightweight views again, just without the image texture). Also the drag preview: `.draggable` today snapshots the tile (image included); a transparent overlay view would drag an empty rectangle unless you supply a custom preview, which has to re-render the thumbnail — re-introducing a texture for the drag image. Solvable, but it is explicit extra work the recommendation doesn't budget.

- **Context menus (`.contextMenu`).** Two distinct menus: People's `FaceTileMenu` (Use as Thumbnail / Move to person… / Remove from person, PeopleView.swift:661-691) and the implicit right-click target. `.contextMenu` requires a per-cell view with a hit region — same conclusion as drag: the overlay must carry a real interactive view per visible cell. The recommendation's own option-D note even flags that `.drawingGroup()` "risks hit-testing on `.contextMenu`/`.draggable`" — the same hit-testing concern applies to *anything* that removes the per-cell view, which is exactly what C does to the image. C dodges it only by keeping per-cell overlay views, at which point the "one layer per section" win is about textures, not view count.

- **Hover.** Minor today (no `onHover` on tiles currently — the grids don't hover-highlight), so nothing to break. But if hover is ever added, C pays the Canvas-redraw-on-hover cost the recommendation itself flags as a possible "smoothness ceiling" reason to fall back to A.

- **Live update on edit/delete.** Today an edit/delete mutates the source arrays (`state.flatItems`, `items`, `cullGroups`) and SwiftUI re-diffs the `ForEach`; the changed tile's `.task(id:)` re-fires if its `cacheKey` changed. In C, the Canvas has no per-item identity diffing — you must explicitly invalidate the plane and re-pull the affected `id` from the cache. Deletion changing the grid's item count also shifts every computed cell rect, so the analytic frame model must recompute. Workable but, again, manual where SwiftUI was automatic.

## Overstated speedups / claims to flag

1. **"Reuses ThumbnailStore/tileMemoryCache/the provider/gridThumbnailPixels unchanged, and needs no SwiftUI↔AppKit concurrency bridge."** Half-true. The *core* store is unchanged, but the *view-side* async-load loop (per-cell `@State` + `.task(id:)` + reuse-guard) is replaced by a hand-managed parent dictionary and N detached tasks. That is new concurrency code with the same race-condition surface that the reuse-guard exists to close. Lower risk than an AppKit bridge, but not "free."

2. **"Avoids the rubber-band/CellFramesKey plumbing."** Misleading. C *removes* the GeometryReader/preference path but *must add* an analytic per-cell-rect computation that has to track adaptive columns, gridGap, pinch-zoom, section headers, and the Cleanup horizontal scrollers. For Cleanup especially (per-group horizontal `ScrollView`s, CleanupView.swift:130-138) and Timeline's pinned headers, computed rects are materially harder than today's measured frames. Net plumbing is comparable or worse, just relocated.

3. **"Thin SwiftUI overlay."** Not thin. To preserve `.draggable`, `.contextMenu`, tap, selection ring, badges, and per-cell drag-preview, the overlay must instantiate a hit-testable, badge-bearing view for **every visible cell** — i.e. it is `MediaTile` minus the image texture. The win is real (collapsing N image *textures* to ~1 per section is exactly what the spike says fixes the Space-switch hitch), but it is a *texture-count* win, not a *view-count* win. Any framing of C as "one layer + thin chrome" undersells the overlay.

4. **The confirmed bug is texture COUNT, and C only fixes it if the overlay truly adds zero GPU-backed layers.** SF-Symbol/capsule badges and the stroked ring are vector and cheap, so this holds — *provided* the custom drag-preview and any future hover effect don't quietly re-introduce per-cell textures. Worth stating as an explicit invariant for the C implementation, because it's easy to violate (e.g., a `.drawingGroup()` on the overlay, or a thumbnail-bearing drag preview held resident).

## Net judgment

- **D (cache + decode discipline, drop `drawingGroup`):** endorsed as written. `countLimit:6000` → `totalCostLimit` by byte cost is a clean, low-risk win, and the spike's own finding that `.drawingGroup()` doesn't reduce texture count *and* endangers `.contextMenu`/`.draggable` hit-testing is consistent with this code (those modifiers are load-bearing in Folders and People). Drop it.

- **C:** correct *direction* for the real bug (collapse image textures), but scoped optimistically. The load-loop rewrite, the analytic per-cell frame model for the marquee, and the per-visible-cell interactive overlay (for drag + context menu + drag preview) are each real work the summary treats as already-solved. Start on Timeline as proposed, but treat **rubber-band correctness under pinch-zoom + pinned headers** and **drag-preview fidelity** as the two acceptance gates, since those are where C is hardest and where regressions would be silent (wrong tiles selected, empty drag image) rather than crashes.

- **A (NSCollectionView):** the recommendation's "fallback only" placement is defensible given the Swift-6 state-bridge and 6+ shared interaction features to re-host — but note that *every* interaction A would have to re-host natively (rubber-band, multiselect, drag, context menu, prefetch) is also work C must re-host in SwiftUI. The honest comparison is "re-host in AppKit with native marquee/drag" (A) vs. "re-host in SwiftUI with hand-computed marquee + custom drag preview" (C). C is still the lighter lift and stays in-paradigm, so the sequencing stands — but the gap is smaller than "S/next vs XL/fallback" implies.

## Appendix E — Subsystem summaries

**Catalog, queries & search** — The catalog is a normalized GRDB/SQLite schema (assets keyed by content hash; instances per local copy; vault_presence per drive-only copy) with single-column indexes on takenAtMs, hash, dirPath, personID, countryCode/city. The browse path is a UNION ALL of a local branch (assets⋈instances) and a drive-only branch (assets⋈vault_presence with a NOT EXISTS over instances and a correlated MIN(rowid) dedupe). The decisive scaling weakness is that EVERY browse/search/folder/dedupe query is built as `SELECT * FROM (timelineSQL) WHERE ...` — it materializes the entire N-row union into a derived table FIRST, then filters and sorts it in memory with no pagination and no index usable through the subquery boundary. So timeline load, folder navigation, structured filter, and "all hashes" are all O(N log N) full-library passes that allocate the whole result set (full TimelineItem structs, ~20 columns each) on the heap. These are triggered by refreshQueries() on essentially every state change (delete, restore, import, sync, drive connect, tag sync, undo) and search runs on a 300ms keystroke debounce. At 10k assets this is fine; by 100k each refresh is a multi-hundred-ms main-thread-blocking allocation storm and search has visible lag; at 1M it is multi-second beachballs and hundreds of MB of transient allocation per refresh. Separately, SemanticIndex holds ALL E embeddings as one contiguous fp32 [Float] matrix resident in RAM (200MB at 100k, 2GB at 1M) and re-sorts the full score vector per query, and folder/path GLOB predicates and json_each tag predicates cannot use a B-tree index so they full-scan. Several whole-table Set builders (knownSizeDateKeys, assetHashes, knownHashes, instanceHashes) hold N strings resident. The catalog is a well-normalized GRDB SQLite schema with a clean separation between authoritative human metadata (mirrored from XMP sidecars) and rebuildable machine-derived caches, and a single Sendable `Catalog` façade fronting a `DatabaseQueue` — the rebuildable-cache contract is documented per-table and the migration discipline is exemplary. The dominant structural weakness is that the entire browse/search/derivation read path is built on a non-materialized `timelineSQL` UNION-ALL-over-a-derived-table that (a) cannot use the only useful index (`assets.takenAtMs`) for its top-level ORDER BY, (b) is re-parsed and re-planned as an inline subquery in a dozen call sites with no pagination, and (c) forces every browse to fetch and sort all rows in RAM. Compounding this, the `assets` table — the hot filter table — carries exactly one secondary index (`takenAtMs`) while `structuredFilter` predicates on `kind`, `favorite`, `rating`, `cameraModel`, `livePairHash`, and `isLivePairedVideo`, so every structured filter degrades to a full table scan plus correlated subqueries. The folder browse uses `GLOB` recursion that a B-tree cannot serve, and `SemanticIndex` loads all embeddings into a single contiguous fp32 matrix in RAM with a full sort per query. None of these are correctness bugs at today's library sizes, but together they are a hard scaling ceiling that blocks the documented roadmap toward 100k–1M-asset libraries and a server-side consumer of the canonical drive. The data model itself is sound; the query layer needs a materialized/paginated browse view, targeted composite indexes, and an ANN/disk-backed embedding store.

**Vault format, IO, sidecars** — The on-disk format is clean and third-party/server-friendly (line-delimited JSONL manifest, per-file XMP sidecars, content-addressed hashes, self-describing vault.json) — but the MUTATION model is uniformly "read the whole file, transform in memory, sort, rewrite the whole file atomically." manifest.jsonl and bin.jsonl are each one monolithic file: every single-entry change is O(M) (or O(B)) in time and resident memory, and because the app's batch operations (move K photos, delete K photos, propagate to D drives, replay queued ops) call these single-entry primitives inside per-file loops, the real cost is quadratic — O(K·M) for moves, O(K·B)+O(K²) for deletes, O(D·K·M) for drive propagation. The manifest write additionally re-sorts all M entries on every write. The sidecar design is one small file per asset, so ingest, export, and Finder-tag sync are each O(N) file opens with a full XMLDocument+XPath DOM parse per sidecar (ingest runs after EVERY scan). Hashing and AtomicFile are individually well-built (streaming, constant-memory, properly autorelease-pooled), but AtomicFile's fsync makes every wholesale rewrite a durability barrier, amplifying the quadratic loops into real wall-clock hangs. None of these are cached: there is no in-memory manifest — every reader re-reads and re-parses the file from disk. At 10k assets the move/delete loops produce visible multi-hundred-ms to multi-second hangs; at 100k–1M they become beachballs and the manifest rewrite alone holds the entire manifest resident twice (parsed array + encoded Data). The on-disk format is the strongest part of this subsystem: it is genuinely self-describing, content-addressed, and writes through one disciplined choke point (AtomicFile) that every state file flows through. The format doc is unusually rigorous and the code matches it closely, so the third-party/server consumer story is real. ContentHash is correctly streaming + autorelease-bounded, FaceRegion's Vision↔MWG conversion is symmetric and well-documented, TagMerge is a clean pure function, and the sidecar XMP shape is standards-aligned. The structural weaknesses are all about the manifest as a data structure rather than its on-disk shape: manifest.jsonl (and every sibling .jsonl) is modeled as a single whole-file value that is read, mutated, sorted, and rewritten in full on every change. There is no in-memory index and no per-entry mutation primitive, so per-file editing loops are O(M^2), the manifest has no single owner (10+ call sites open it directly), and the same read-append-rewrite anti-pattern is copy-pasted across BinStore, SyncLog, and three registries. Secondary structural gaps: AtomicFile's atomicity contract omits the parent-directory fsync the format doc itself mandates, the sidecar-per-file model has no batch I/O abstraction so ingest/export are unavoidably O(N) sequential opens, and BinStore collapses distinct deletions that share a relative path. None of these threaten the format's portability; they threaten scaling and maintainability at the 10k–100k asset sizes the roadmap targets.

**Scan & indexing pipeline** — The scan pipeline is a single-threaded O(files) directory walk feeding a series of fully-materialized O(N) arrays (found, aligned, newAssets, pairCandidates, instances) that are all resident simultaneously, plus the entire manifest loaded as an O(M) dictionary for the size+mtime fast-path. Hashing is correctly streaming/constant-memory per file (ContentHash.ofFile), and metadata extraction and embedded-XMP reads are wrapped in autorelease pools, so per-file memory is bounded — but the orchestration arrays scale linearly with library size and are held for the whole scan, so a 1M-asset library materializes several large arrays at once. Live-photo pairing is O(n) per scan via dictionaries (not quadratic). The real scaling defects are: (1) the pipeline is not parallelised — files are walked, hashed, and extracted strictly serially even though hashing/extraction are embarrassingly parallel and I/O-bound; (2) replaceInstances does a wholesale DELETE-then-reinsert of every instance in the local vault on EVERY scan, so even a single new photo rewrites all N instance rows; (3) the watcher debounce is correct (2s) but the onChange callback triggers a FULL scanAll() of every vault rather than an incremental rescan of the changed subtree, so any file event re-walks and re-stats the entire library; (4) the scan is not resumable — interruption discards all hashing work because the manifest and instances are only written at the very end. knownHashes() pulls every asset hash into an in-memory Set on each scan (O(N) time and space). At 10k these are sub-second; the cliffs appear at ~100k (multi-minute scans, hundreds of MB resident) and become severe at 1M (long uninterruptible indexing, large resident arrays, and a full-library rewrite triggered by a single file change via FSEvents). The pipeline is a single, linear, free-function `Scanner.scan` that walks a vault, applies a size+mtime manifest fast-path, hashes changed files, extracts metadata for catalog-unknown hashes, pairs Live Photos, then wholesale-replaces the vault's instance rows and atomically rewrites the manifest. The smaller pieces are genuinely well-factored: `ContentHash` streams in constant memory, `MetadataExtractor`/`EmbeddedMetadata` correctly pool ImageIO temporaries (the OOM fix), `MediaKind`/`MediaMetadata`/`ManifestEntry` are clean value types, and `LivePhotoPairer` is a pure, testable function. The structural weaknesses are at the pipeline level: (1) `scan` is one ~130-line monolith holding the entire file list, the entire old manifest, the entire known-hash set, and all new records in RAM simultaneously — an O(files) + O(manifest) memory profile that doesn't stream and isn't resumable, directly blocking the documented memory-safety roadmap; (2) the scan is strictly single-threaded and unbatched, so hashing + metadata extraction (the expensive, embarrassingly-parallel work) cannot use the `Task.detached(.utility)` parallelism the design calls for; (3) there is no orphan-asset pruning — `replaceInstances` cleans up the per-vault instance rows but `assets`/`vault_presence` rows for vanished content are never removed, a correctness gap for the catalog and any third-party/server consumer reading from it; (4) the watcher fires a single coarse whole-library rescan with no path scoping, so any FSEvent re-walks every file. None of these are line-level bugs — they're shape decisions that will be expensive to unwind once the catalog schema and format are frozen.

**Derivation / ML pipeline** — The pipeline is a resumable, retry-capped (3-attempt), per-asset/per-stage job queue persisted in SQLite (derivation_jobs, PK hash+stage). Correctness-wise the design is sound: idempotent markDerived, drive-only assets skipped not failed, graceful model absence. Scaling-wise it is fundamentally a serial O(N) loop per stage with five stages, dominated by per-asset full-resolution image decode and ML/Vision inference. The runner (drainDerivation, AppState.swift:1473) drains the ENTIRE pending set of each stage one item at a time inside Task.detached(.utility) with a Task.yield between items — so initial derivation of a 100k–1M library is hours-to-days of background work, which is acceptable as long as it stays off the main actor and bounded in memory. The real scaling hazards are: (1) four of the five file-backed stages each decode the SAME full-res image independently (4× redundant full-res decodes per asset, each holding a full bitmap + a 512-d MLMultiArray / CVPixelBuffer resident); (2) the whole pending-hash list (O(N) strings) is materialized into a Swift array up front for each stage; (3) the GeoNames table (~30k City structs) is held resident in RAM for the app lifetime; and most acutely (4) semantic SEARCH constructs a brand-new EmbedStage() on every debounced query, which re-parses the ~49k-line BPE merge file to rebuild the CLIPTokenizer and recompiles the text Core ML model from scratch every single search — none of that lazy load state is shared with the runner's instance. The ReverseGeocoder, contrary to the audit hypothesis, already uses a 1°×1° grid bucket, so per-photo lookup is ~O(1), not O(G); the geocode cost is resident memory, not query time. The derivation subsystem is built around a clean, well-judged core abstraction: `DerivationStage` (DerivationStage.swift) is a small Sendable protocol — `id`, `eligibleKind`, `isAvailable`, `needsFile`, and a single `run(hash:url:catalog:) -> Bool` — with sensible protocol-extension defaults, and stages are uniformly registered in one array (AppState.swift:1456). Resumability and retry-capping live correctly in the catalog (Catalog+Derivation.swift): `derivation_jobs(hash,stage,status,attempts)` plus a LEFT-JOIN `pendingDerivation` query that excludes done/over-cap jobs, and the runner deliberately pulls the WHOLE pending set so an unreachable newest asset can't starve older reachable ones — a thoughtful, documented design (the "4.1 starvation lesson"). Graceful degradation (missing model → `isAvailable=false` → stage skipped, jobs stay pending, nothing marked failed) is consistent and correct. The two real structural problems are (1) **per-stage state has no home**: every search query constructs a fresh `EmbedStage()` and pays full Core ML compile + full BPE-tokenizer build on the hot path, and the runner's `EmbedStage` is a separate instance from the search path's, so the model is compiled at least twice and the tokenizer is rebuilt per query; and (2) the **runner lives in the @MainActor AppState god object, drains all stages strictly serially with no batching/bounded parallelism**, coupling the pipeline policy to the UI layer and leaving inference single-threaded. Notably, the prompt's hypothesis that `ReverseGeocoder` does an O(N·G) linear scan is INCORRECT: it already builds a 1°×1° grid spatial index and scans an expanding ring (ReverseGeocoder.swift:31,47-70) — that part is well-designed; the only minor geocode cost is the in-RAM city table and a one-time loader inefficiency.

**Cull & faces (clustering/dedup)** — The clustering/dedup leaf algorithms are pure, unit-tested, and correctly run off the main actor via Task.detached, so nothing here beachballs the UI directly — the scaling risk is detached-CPU latency that turns the People and Cleanup features into long-spinner experiences at power-user/stress sizes, plus one correctness defect. The two headline costs are both super-linear and unguarded. FaceClusterer.cluster (FaceClusterer.swift:43-60) is greedy single-link agglomerative: O(n²·d) worst-case over n unassigned faces × ~512-768-dim vectors, and its single-link rule (join if ANY member is within threshold, line 47) is a design defect that transitively chains distinct people into one cluster (wrong clusters, not just slow). It also re-runs in full after every person-naming action (facesDirty→loadPeople), so the quadratic cost is paid repeatedly. DuplicateGrouper.group (DuplicateGrouper.swift:18-23) is per-folder brute-force pairwise Hamming under union-find: O(Σ n_f²), which is benign when photos spread across many folders but collapses to O(N²) for the very common case of one giant flat import folder, and it re-fires on every Cleanup mode-Picker toggle. Feeding both is phashRowsWithDirPath, which materialises the entire local∪drive timeline UNION (with correlated NOT EXISTS / MIN(rowid) subqueries) and decodes ~N rows before grouping even begins, repeated per toggle. Supporting issues: unassignedFacesWithEmbeddings scans an unindexed source predicate and decodes n×d vectors that the clusterer immediately copies again (no index on faces.source or embeddings.model), and burst sharpness is recomputed by per-item thumbnail decode + Laplacian on every load instead of being a cached derivation. None crash at 10k; the cliffs land at ~20k+ unassigned faces and at large single-folder imports (100k power-user, 1M stress), manifesting as multi-second-to-minute spinners on People and Cleanup. Fixes: replace single-link with centroid/HDBSCAN + ANN index for faces (→ O(n·log n·d)), replace per-folder pairwise Hamming with a BK-tree / multi-index-hashing (→ near-linear), restrict the duplicate query to indexed local instances, add the missing indexes, and cache sharpness + memoize grouping per catalog generation so UI toggles stop recomputing. This subsystem is a set of small, pure, well-tested algorithm primitives in OpenPhotoCore (BurstGrouper, DuplicateGrouper, FaceClusterer, KeeperSelector, PerceptualHash, FocusMeasure), each cleanly Sendable/synchronous and orchestrated entirely from AppState.loadPeople()/loadCullGroups() off the main actor. The boundary discipline is genuinely good: the algorithms have no Vision/CoreML/GRDB coupling, take plain value tuples, are deterministic, and are unit-testable in isolation — KeeperSelector and FocusMeasure in particular are clean. The structural weaknesses are concentrated in (1) algorithmic scaling: every matcher is brute-force O(n²) pairwise with no spatial/ANN index, and FaceClusterer additionally uses greedy single-link agglomeration that chains transitively-connected faces into wrong "blob" clusters; (2) a semantic data-model defect — face identity is clustered on a generic VNGenerateImageFeaturePrint scene descriptor rather than a face-identity embedding, and clustering ignores already-named people entirely (no nearest-person assignment); (3) thresholds and magic windows are hardcoded at the AppState call sites rather than being part of a config/policy object; and (4) the three "group-then-rank-keeper" pipelines are structurally duplicated inline in one giant closure. None of these threaten on-disk format/data integrity (the catalog is rebuildable and faces are append-only auto rows), so severities top out at High — but they directly block the documented roadmap of scaling to a friend's real 10k+ library and a future face-quality UX.

**Import engines** — The import subsystem is structured as a thin ImportSource protocol (enumerate → fetch → thumbnail → delete) with the ImportEngine orchestrating stage→hash→dedup→place→rescan→verify per batch. Most per-item work is correctly amortized: dedup is a single indexed catalog query per item, the batch rescan runs once, and copy+hash is inherently I/O-bound and streamed. The subsystem scales acceptably to the 10k typical case but has several super-linear or memory-resident hot spots that bite the power-user (100k) and the stress (1M / friend's-real-import acceptance test) cases. The dominant problems are: (1) ImportRegistry.append rewrites the ENTIRE imports.jsonl on every single item (O(R) write per item ⇒ O(R²) per batch, with full fsync), (2) pairLiveItems is O(P²) because of an inner firstIndex over an iPhone-heavy Live-photo set, (3) the verify step builds a whole-manifest dictionary (O(M) memory, the full manifest decoded into RAM) for every batch, (4) the engine re-enumerates the entire source (full FS walk / full PHAsset fetch) just to resolve missing Live partners, and (5) every source's enumerateItems() materializes all N items (+ a full id→file/entry map) resident in memory with no paging, and TakeoutSource does this single-threaded with a synchronous EXIF decode per file. None are O(1)-trivial; the registry-rewrite and the O(P²) pairing are the two that turn a large import into a multi-minute beachball. The import subsystem is a clean source/engine split: a single `ImportEngine.run` orchestrates a well-ordered, integrity-first pipeline (fetch → independent re-hash → destination-aware dedup → atomic place → rescan → manifest-verify → registry), and five concrete `ImportSource` adapters (camera/volume/PhotoKit/Takeout/foreign-vault) keep their device specifics local. The hard invariants are respected: the engine always re-hashes its own copy rather than trusting any source-claimed hash, places via move-after-staging, and verifies against the scanner's independent manifest hash before recording. CameraSource's ICC concurrency (single-flight enumerate/open, copy-out-then-resume continuation discipline) is genuinely hard-won and well-documented — leave it alone. The structural weaknesses are all about boundary erosion and per-item cost amplification rather than correctness: the ImportSource protocol does not actually capture what the engine and UI need from sources (so they downcast to concrete types and pass out-of-band callbacks), the durable import-registry rewrites its entire backing file on every single appended row (O(N²) per session), Takeout JSON resolution is a per-file filesystem stat storm with no directory index, dedup identity is fragmented across three unrelated fingerprint schemes, and the verify step reads the whole manifest plus triggers a whole-vault rescan once per batch. None of these threaten data integrity today, but they cap throughput on the documented 10k-item real-world import that is the acceptance test, and the leaky protocol blocks adding the next import source cleanly.

**Sync & send engines** — The sync/send engines are clean and dictionary-based at the plan layer (the diff is O(S+Dst), not nested-loop), but they scale poorly the moment work is applied repeatedly, because the on-disk model is "read the whole JSONL, mutate, rewrite the whole JSONL" and several callers wrap that in a per-file loop. The manifest (M ≈ N lines), the drive bin log (B lines), and the in-memory sends/devices registries all parse fully and rewrite fully on every mutation. Single-shot operations (one plan(), one apply(), one propagate()) are linear and fine to ~100k. The cliffs are (1) DriftReconciler bulk repair/restore/adopt, which loops over R findings and inside each one does a full Manifest.read+write plus a full bin-log read+write — O(R·M + R·B), quadratic by ~10k findings; (2) CatalogIngest.ingestDriveFile, called once per adopted file by adoptAll, which re-reads all N catalog hashes and full-deletes-and-reinserts the drive's entire presence set per file — O(A·(N+P)); (3) SendEngine/SendReverifier dedup which is an O(items·present) nested membership scan with no hash index; and (4) CatalogSnapshot.write/import which VACUUM-copies the whole catalog and per-hash file-copies all thumbnails on every sync. The three engines (SyncEngine, SendEngine, DeletionPropagator) share the VerifiedCopy/Manifest/BinStore primitives but have no shared batch-mutation abstraction, which is exactly why every "all" loop re-pays the full-rewrite cost. This is a generally well-conceived subsystem. The core integrity primitives are clean and correct: VerifiedCopy (temp→fsync→re-hash→rename, never-overwrite) is the right shared atom; SyncEngine.plan/planClone diff manifests with a destination dictionary (O(S+Dst), not nested-loop); DeletionPropagator.propagate correctly does a SINGLE atomic manifest rewrite per batch with a recoverable "files-move-first, manifest-last" ordering that self-heals on interruption; the pure/effectful split (DeletionPropagator.eligible, SendReverifier.reconcile, CanonicalManagement/BackupStatus set math) is genuinely testable. The data model (PlanItem/SyncPlan/DriftFinding/SendItem) is coherent and Sendable. The structural weaknesses are three: (1) a pervasive "read-whole-file / mutate-one-row / rewrite-whole-file" idiom in DriftReconciler and CatalogIngest that turns every "repair/adopt/restore all" loop quadratic in both the on-disk manifest and the SQLite presence table; (2) three sibling engines (SyncEngine, SendEngine, DeletionPropagator) that re-implement the same copy-verify-record-then-journal pipeline with no shared "VerifiedTransfer + journal" abstraction, plus an inconsistent value-vs-reference and result-shape design; and (3) a thin, lossy error model (Bool/optional-try swallowing) that erases the distinction between "hash mismatch", "unreadable", and "I/O error" exactly where the format spec wants reasons recorded. None of these is a data-integrity bug today, but #1 blocks the documented 10k+ real-import roadmap and #3 undercuts the third-party-readable canonical-drive goal.

**LibraryService orchestration & selection** — LibraryService is a plain Sendable class performing fully synchronous SQLite + filesystem I/O on the caller's actor; in practice almost every call site is AppState (@MainActor), so its costs land directly on the UI thread. The dominant scaling problem is the browse path: timelineItems() materializes the ENTIRE catalog (a UNION ALL of local instances + drive-only presence rows) into an array of N TimelineItem structs with an outer ORDER BY that forces an O(N log N) filesort with no LIMIT/pagination, and folderTree() additionally does a full recursive FileManager filesystem walk of the primary vault unioned with a GROUP-BY count query. Both run together inside refreshQueries() on the main actor, and refreshQueries() is invoked after essentially every mutation and on every grouping toggle — so the whole library is re-read, re-sorted, re-grouped and the disk re-walked on each one. The selection layer is value-correct (Set-backed) but the SwiftUI callers around it rebuild an O(P) orderedSelectable array and do an O(P) firstIndex linear scan on every single tap, and feed an O(P) cell-frames dictionary into updateDrag on every drag tick (~60/s). Eviction/rehydrate/drive-source helpers re-fetch the full vault_presence table per item and per Live-half, giving O(K · presence-rows) with a linear .first(where:) scan each time. None of these are algorithmically wrong, but several are super-linear or re-do whole-library work that should be cached/paginated, and they bite hard between the 100k power-user and 1M stress tiers. This subsystem is the Core façade (`LibraryService`, a plain `Sendable` class) plus the UI-agnostic `SelectionModel`/`UndoAction`/`PhotoMovePayload` value types. The selection layer is genuinely well-designed: `SelectionModel` is a pure, `Sendable`, unit-testable value type whose tap/range/drag math is correct and whose per-operation cost is bounded (one O(P) scan per drag tick or tap, not the O(K·P) the brief feared), and `UndoAction` is a small data-only descriptor replayed through existing public ops with the native `UndoManager` capping depth at 50 — so undo memory is trivially bounded and there is no bespoke undo engine to maintain. The façade itself is the weak point. Its synchronous, side-effecting browse methods (`timelineSections`, `folderTree`) are invoked directly on the `@MainActor` inside `AppState.refreshQueries()`, where `folderTree` additionally performs a full recursive filesystem walk (`directoriesUnder`) unioned with a SQL aggregate — a main-thread hazard that grows with library size and is called after every mutation. The biggest structural problem is that the move/folder-reorg use case is split across the Core/App boundary: `LibraryService.movePhotos` only touches the local primary vault, while all drive propagation, offline-op queueing, presence-row re-keying, mac→drive path mapping, and undo-record assembly live in `AppState+FolderReorg.swift` — putting format-level logic and the documented multi-drive invariant in the SwiftUI layer rather than in the Core that the format docs describe. Finally, the eviction/rehydrate path repeatedly fetches whole-drive presence row sets and linear-scans them per item, an O(items × drive_rows) pattern that wants an indexed point lookup.

**AppState architecture** — AppState is a single ~1740-line `@Observable @MainActor` class (1107+633 lines plus two extensions) that is simultaneously the view model for every screen, the orchestrator for library/query/search/people/cull/drives/derivation/send/undo, and the off-main work dispatcher (29+8 hand-rolled `Task.detached` sites). It is the sole binding (`@Bindable var state: AppState`) for 34 SwiftUI views with a very wide fan-out (73 distinct member accesses), so any mutation invalidates all views. The design has real strengths: the off-main discipline is consistent (heavy SQL/ML always inside `Task.detached(.utility/.userInitiated)`, only observable state published back on main), `LibraryService` is a clean Sendable Core façade, the undo model is admirably disciplined (data-only descriptors that re-dispatch tested ops with inverse args, stale-safe, no redo), and the derivation drain correctly pulls whole pending sets up front to avoid the documented starvation bug. The structural problems are concentrated in three areas: (1) `refreshQueries()` is a synchronous main-actor full-rebuild (full DB UNION load + O(N) grouping + a recursive FileManager filesystem walk + bin load + per-drive presence loop) invoked from ~19 in-class sites and 8 view sites — a main-thread hazard that scales with library size and blocks the documented 10k+ import acceptance test and the deferred tile-grid work; (2) the god object conflates seven cohesive subsystems with no internal seams, making the publishing granularity all-or-nothing and the file unmaintainable; (3) drives/role orchestration (~700 lines) duplicates a fixed presence/badge/drift refresh sequence across ~10 methods with no single reconcile primitive. None of these are data-integrity bugs, but they block the roadmap and concentrate maintenance risk.
